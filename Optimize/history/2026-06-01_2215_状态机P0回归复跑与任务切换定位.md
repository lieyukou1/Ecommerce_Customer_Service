- 对 `functional-test-workspace/scenarios/api-scenarios.js` 做了第一批基线放宽，减少“对象首轮进入必定是 knowledge”这类旧语义误报。

2. 状态机入口
- 在 `customer-service-backend/atguigu/engine/dialogue_engine.py` 增加“任务上下文中切换到新对象”的入口处理：
  - 若当前仍有 active task，且用户切到不同对象，则先清掉旧 runtime state；
  - 再把新对象作为新的 `object_requires_intent` 澄清起点；
  - 避免旧任务与新对象并存导致的状态串味。

# 仍未完成的真实问题

1. 任务启动后被一步走完，但状态断言仍按“进入 active_task 等待槽位”编写
- 典型场景：`track-switch_knowledge-clarify-task_work-order-followup`
- `"那先帮我催一下，家里老人怕热"` 这一轮同时包含“启动催办 + 填充原因”，当前实现会直接完成催办，不会停在 `ask_urge_reason`。
- 这更像场景设计问题，不一定是代码 bug。

2. 任务打断/切换时仍有稳定问题
- 典型场景：`task-switch_progress-to-urge-to-complaint`
- 在催办 `ask_urge_reason` 阶段，用户切到投诉，仍出现 HTTP 500。
- 失败前状态保持在：
  - `active_task = work_order_urge_submission`
  - `active_system_task = system_collect_information`
  - `paused_tasks = []`
- 说明 `start_flow` 与“当前仍在 collect 阶段的任务中断”之间的协作仍不稳。

3. 跨对象任务切换语义仍需明确
- 典型场景：`task-switch_urge-then-complaint-on-another-work-order`
- 当前新对象到来时会清掉旧任务，避免串味；
- 但旧回归 case 仍期待旧任务被压入 `paused_tasks`，这与新入口策略已不一致。
- 这里需要决定最终语义：
  - 新对象到来时直接 reset 旧 task
  - 还是保留为 paused task 并允许恢复

4. 仍有一些非核心噪音
- 若干 case 还在校验开场澄清文案里必须出现对象摘要；
- 若干“退出后起新话题”仍把 knowledge 问句当成 chitchat；
- 这些属于回归基线继续收口的问题，不是当前最核心的状态机 bug。
