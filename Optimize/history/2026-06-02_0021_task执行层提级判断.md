# 本轮目标

基于最新长对话回归和新接入的 `last_task_outcome` 观测字段，判断当前是否应该把第三阶段 P0 继续往 `task/command/processor.py`、`task/flow/executor.py` 这类执行层深挖。

# 为什么现在做

前一轮我们已经补上了一个关键观察口：

- `last_route` 只能说明这一轮被判成了什么轨道
- `active_task` 只能说明这一轮结束时有没有挂住任务
- 但它们都不能准确说明：这个 task 是没进去，还是进去后当轮就收口了

现在 `last_task_outcome` 已经能把 task 收口结果显式打出来，所以该先做一次“是否提级”的判断，避免又回到对着 badcase 补下游分支的老路。

# 结论

当前 **不建议** 立刻对 `task executor / processor` 做大范围深改。

更准确地说：

1. 现在已经有足够证据表明，回归里一大批 task 相关失败并不是“任务根本没走进去”
2. 其中相当一部分其实是：
   - 已进入 `task`
   - 并且在当轮内完成
   - 最终会话状态按设计落回 `focused_knowledge`
3. 这类失败更像是：
   - 回归预期写成了“应保留 active_task”
   - 或当前 track 语义和断言语义还没有完全对齐

所以现在直接去动执行层，风险大于收益，也不符合第三阶段“先提级判断，再决定是否下潜”的方法论。

# 证据分组

## A. 明确属于“已进入 task，但当轮完成”的场景

这些 case 里，`last_route.kind = task`，同时 `last_task_outcome.kind = completed_with_focus`，最终没有 `active_task` 残留：

1. `track-switch_knowledge-clarify-task_service-item-to-business`
   - `start:human_handoff`
   - 结果：`completed_with_focus`
   - 判断：更像办理/转人工类一次性动作，不像需要长期挂起的任务

2. `object-switch_service-item-a-to-b_and-exit`
   - `start:service_item_detail_query`
   - `start:work_order_status_query`
   - 结果：都为 `completed_with_focus`
   - 判断：更像一次性查询型 task，完成后保留 `focused_object` 继续知识追问是合理的

3. `object-switch_work-order-a-to-b_followup`
   - `start:service_progress_tracking`
   - 结果：`completed_with_focus`
   - 判断：也是查询/追踪型 task，当轮返回结果后落回 `focused_knowledge` 并不奇怪

4. `exit-and-reset_task-exit-then-start-new-object`
   - `start:work_order_status_query`
   - 结果：`completed_with_focus`
   - 判断：这条失败更像测试把“工单状态追问”当成 knowledge，但当前 runtime 把它表达成 task 型即时查询

这几组都不支持“执行层坏了”的结论。

## B. 明确属于“任务稳定进入等待补槽位”的场景

这些 case 里，`last_task_outcome.kind = waiting_for_slot`，并且 `active_task`、`active_system_task` 都保留得很完整：

1. `track-switch_chitchat-knowledge-task_service-item-to-work-order`
   - `complaint_request_submission`
   - 结果：`waiting_for_slot`

2. 多个投诉/催办切换场景
   - `complaint_request_submission`
   - `work_order_urge_submission`
   - 结果：`waiting_for_slot`
   - 有的还正确带上了 `paused_tasks`

这说明执行层并不是整体失灵。至少对“需要继续收集槽位”的任务，主链是能把任务挂住的。

## C. 当前真正值得继续盯的，是“同一 flow 表现不一致”的少数场景

最值得关注的是：

1. `track-switch_knowledge-clarify-task_work-order-followup`
   - `turn-3-switch-to-urge`
   - `reason = start:work_order_urge_submission`
   - 结果却是 `completed_with_focus`

而同一个 flow 在其他场景里多次表现为：

- `last_task_outcome.kind = waiting_for_slot`
- `active_task.flow_id = work_order_urge_submission`

这说明这里确实可能有问题，但问题更像是：

- 某条输入触发了不同的规划/参数填充语义
- 或某个 flow 的收口条件过早满足
- 或同一 flow 在不同前置上下文下被规划成了不同命令组合

这依然还不是“大改整个执行层”的证据，更像一个应该继续往：

- planner / route 决策
- flow 设计语义
- 单个 flow 的收口条件

去缩小范围的点状问题。

# 当前判断分类

把这轮 task 相关失败按是否值得提级，先分成三类：

1. **预期/语义对齐问题**
   - `work_order_status_query`
   - `service_item_detail_query`
   - `service_progress_tracking`
   - `human_handoff`
   - 特征：`completed_with_focus`
   - 建议：优先改回归断言语义，不先改执行层

2. **局部可疑问题**
   - `work_order_urge_submission` 在少数场景下出现 `completed_with_focus`
   - 特征：同一 flow 在不同 case 表现不一致
   - 建议：后续先做更细粒度定位，不直接扩大到整个 `processor/executor`

3. **已验证健康的执行路径**
   - `complaint_request_submission`
   - 多数 `work_order_urge_submission`
   - 特征：`waiting_for_slot + active_task + active_system_task` 成套保留
   - 结论：不支持“执行层普遍有问题”

# 对 P0 排期的影响

这次判断支持我们继续按既定 P0 走：

1. 继续推进显式状态机主线
2. 继续把状态迁移、任务收口、轨道语义做清楚
3. 不把这轮 badcase 命中率当作当前阶段的唯一目标

也就是说，当前更应该做的是：

- 让 runtime 的状态表达更清楚
- 让回归断言能区分“即时完成的 task”和“需要挂起的 task”

而不是现在就重新拆 `executor`。

# 本轮未做

本轮 **没有** 修改：

- `task/command/processor.py`
- `task/flow/executor.py`
- 任何具体业务 flow 的执行代码

这是有意控制范围的结果。

# 下一步建议

1. 先把回归里的 task 断言继续分型
   - 区分“应等待补槽位”
   - “应即时完成并保留 focus”
   - “应完成后回 idle”

2. 再决定是否需要对少数可疑 flow 做点状排查
   - 优先查 `work_order_urge_submission` 的异常收口场景

3. 保持第三阶段方法论边界
   - badcase 是门禁
   - 不是这一轮优化的唯一胜利标准
   - 只有在证据指向执行层语义矛盾时，才提级到 `processor / executor`
