# 方法论与 Skill 演进上下文

## 1. 这份上下文是干什么的

这份文档不是某个业务项目的上下文，也不是某次具体优化的流水账。

它服务的是另一条长期工作线：

- 如何把“人机协作优化方法论”沉淀成稳定知识体系
- 如何把这套方法论进一步转成 AI 更容易执行的 skill、模板和脚手架
- 如何在后续新项目里反复试用、修正，而不是每次重新从零总结

后续如果要单独开一个会话窗口持续完善这套东西，可以直接把这份文档当作入场材料。

## 2. 当前已经形成的核心判断

### 2.1 这套方法论的默认对象不是“独立开发项目”

当前已经明确，这套方法论的默认语境是：

- 有老师代码、教学骨架或既有主线
- 不是从 0 到 1 独立架构
- 核心工作是理解、继承、裁剪、增强、收口和优化

也就是说，它不是一份泛化到所有软件项目的总手册，而是一套更聚焦、更真实可用的“既有骨架项目协作方法论”。

### 2.2 方法论应该分层，而不是混成一篇

当前确认过的结构是：

- 一份母文档
- 三份阶段子文档

母文档只保留：

- 适用边界
- 阶段划分
- 共通原则
- 使用方式

子文档按阶段拆开：

- 01：问题定义与架构设计
- 02：主体搭建
- 03：优化与稳定化

其中：

- `03` 是最重要、最厚、最不该过度压缩的一份
- `01` 和 `02` 不假装是 greenfield 指南，而是围绕“既有骨架项目”写具体方法

### 2.3 真正有价值的，不是抽象正确，而是可执行

已经验证出来的一个重要经验是：

- 只有结构没有实操清单，文档很快会变成“看着对，实际不好用”

因此后续方法论文档应该优先保留这些东西：

- Day 1 实操清单
- 阶段进入条件
- 阶段退出条件
- 常见协作坑
- 具体判断顺序

而不是继续堆抽象原则。

### 2.4 共通原则应只在母文档定义一次

这一点已经明确：

- “唯一主目标”
- “优先级显式化”
- “Decision Log 比 Change Log 更重要”
- “目标漂移检查”
- “原子化改动”
- “最小可观测性意识”
- “session 锚点”

这些原则应统一在母文档定义。

阶段文档里可以展开成具体动作，但不应再反复重复总定义。否则后续维护会越来越痛。

### 2.5 导航页必须短，不要又长成正文

导航页的定位已经被校正为：

- 30 秒判断现在该看哪份

它不应该承担方法论正文，也不应该写大量解释性文字。

## 4. 对 AI 使用这套方法论的关键认识

### 4.1 方法论文档本身更偏向“给人看”

当前这些文档最适合做的是：

- 帮人判断边界
- 帮人拆阶段
- 帮人识别常见协作误区
- 帮人把项目经验沉淀成稳定原则

它们并不是 AI 最适合直接消费的运行时形态。

### 4.2 不要高估“AI自动遵守规范”

这一点已经明确校正过预期：

- 做完一套方法论文档，不代表 AI 会自动进入规范模式
- 做成 skill，也不代表 AI 会自动触发任何流程

更现实的定位是：

- 把原本需要手写的一大段背景说明，压缩成一句稳定可调用的话

也就是说，目标不是“AI 自动自律”，而是：

- 你更容易给出完整、可重复、低遗漏的启动指令

### 4.3 人类层和 AI 执行层应该分开

当前已经形成一个稳定判断：

- 方法论文档负责“为什么”
- AI 执行规范负责“怎么做”

它们不该继续混在同一份文档里。

更清晰的分层是：

1. 人类层
   - 母文档
   - 阶段文档
   - 导航页

2. AI 执行层
   - skill
   - phase cards
   - 模板
   - bootstrap 脚本

## 5. 当前 skill 路线的定位

### 5.1 当前 skill 的目标不是“大而全”

已经明确不走的路线是：

- 先设计一个庞大完整的自动化规范平台

当前更合理的路线是：

- 先做一个能在下一次新项目里立刻省力的 MVP

也就是：

- 显式调用
- 阶段判断
- 缺失目录与文档补齐
- 生成最小 context / plan / handoff / evaluation 骨架

### 5.2 skill 的真正价值

当前对 skill 的定位已经比较清楚：

- 不是让 AI 自动进入某种人格模式
- 不是替代人的判断
- 不是自动完成全套协作纪律

它真正的价值是：

- 把长背景说明压缩成稳定调用
- 把重复脚手架动作固定下来
- 把 phase 对应的基础规则以短卡片形式交给 AI

### 5.3 skill 当前采用的结构

当前已经做出一版本地 MVP skill 包，结构大致分成：

- `SKILL.md`
- `agents/openai.yaml`
- `references/phase-01.md`
- `references/phase-02.md`
- `references/phase-03.md`
- `assets/templates/`
- `scripts/bootstrap_project.py`

这条路线已经验证可行，后续可以继续按真实使用反馈迭代。

## 6. 当前 skill MVP 的明确边界

### 6.1 已经做到了什么

当前 MVP 已经具备：

- 显式调用入口
- “既有骨架项目”适用范围定义
- phase 01 / 02 / 03 的短卡片
- 最小模板目录
- 可增量执行的 bootstrap 脚本

脚本当前负责补齐缺失的最小骨架，例如：

- `PROJECT_CONTEXT.md`
- 计划文档
- `SESSION_HANDOFF.md`
- `Evaluation/README.md`
- `Optimize/history/`
- `Evaluation/`
- `Material/`

### 6.2 还没有做到什么

当前 skill 明确还没有做到：

- 自动触发
- 自动判断你一定正在做哪类项目
- 自动接管长期协作纪律
- 自动维护真实业务边界
- 自动决定复杂优先级

phase 判断目前仍然是：

- 结合 repo 现状做启发式判断
- 必要时由人确认或纠偏

### 6.3 这不是“自动驾驶”

这条认知后续必须一直保留：

- skill 是显式调用的协作脚手架
- 不是自动模式切换器

如果后续继续迭代，也不要把目标想象成“让 AI 自己懂一切”，而应该继续围绕：

- 更容易调用
- 更少遗漏
- 更稳定复用

## 7. 后续完善时最应该盯的方向

### 7.1 优先盯真实使用反馈

当前最重要的不是继续闭门扩文档，而是：

- 在真实新项目第一天拿它用
- 观察哪一步仍然需要大量人工补充
- 观察哪些模板太薄，哪些规则太重

也就是说，后续改进顺序应优先是：

1. 用
2. 记问题
3. 迭代

而不是：

1. 继续想象各种完整功能
2. 先做很多模板
3. 结果实际用不上

### 7.2 真正值得补强的方向

后续更值得补的方向包括：

- 哪些启动输出最值得固定
- 哪些 phase 判断最容易模糊
- 哪些模板在实际 session 中反复被手改
- 哪些脚手架文件命名需要更通用
- 哪些动作该在 skill 中做，哪些仍应保留给人判断

### 7.3 暂时不该急着补的方向

暂时不建议过早投入的方向包括：

- 复杂自动分类器
- 过多模板
- 很重的脚手架系统
- 试图把所有协作习惯都编码成硬规则

因为这会让 skill 很快变重，而且不一定经得起真实项目验证。

## 8. 后续开新窗口时建议直接带上的关键信息

如果后续要单独开一个窗口继续完善方法论 / skill，建议至少直接贴出下面这些事实：

1. 这套方法论默认服务于“既有骨架项目”
2. 当前已经是“母文档 + 3 个阶段子文档 + 导航页”的结构
3. `03` 仍然是最重要、信息最密的一份
4. 方法论文档主要给人判断，skill 主要给 AI 执行
5. skill 是显式调用，不自动触发
6. 当前已经有一版本地 MVP，核心是：
   - phase cards
   - 模板
   - bootstrap 脚本
7. 后续迭代优先依据真实新项目的使用反馈，而不是继续纸面扩展

## 9. 一个推荐的后续工作流

后续如果用单独窗口维护这套东西，可以按下面的节奏推进：

1. 在真实新项目里调用一次 skill MVP
2. 记录使用中最不顺的 3 到 5 个点
3. 先判断问题属于哪层：
   - 方法论文档过厚
   - phase card 不够清晰
   - 模板不对路
   - 脚本脚手架不够
   - 预期管理有偏差
4. 只改最影响下一次使用的一层
5. 记录这次修改带来的真实收益

这条线要尽量避免重新掉回：

- 一边讨论方法论
- 一边讨论具体业务项目
- 一边顺手扩很多模板

否则很快又会混线。

## 10. 当前最重要的总原则

这条线后续继续往前走时，最该守住的不是“做出最完整的规范系统”，而是：

**让这套方法论和 skill 在真实协作里越来越省力、越来越稳定，而不是越来越宏大。**
