# 本轮目标

继续第三阶段状态机重构主线，不再围绕单条 badcase 深挖，而是补上状态机入口层的显式路由观测能力，让“本轮最终走了哪条轨道”成为结构化状态。

# 为什么现在做

当前回归和日志有一个长期问题：

1. 很多时候只能从 `active_task / focused_object / conversation_state` 反推这一轮到底走了 `task / knowledge / clarify / chitchat`
2. 一旦任务在当轮内启动又收口，或者状态在末尾被重算，回归就会把“真实走过的路径”和“最终停留状态”混在一起

这会让我们不自觉地继续围着 badcase 打补丁，而不是推进方案 B 的显式状态机主线。

# 实际改动

1. 扩展 `DialogueState`
   - 文件：
     `customer-service-backend/atguigu/domain/state.py`
   - 新增：
     - `TurnRoute`
     - `DialogueState.last_route`
     - `record_route()`
     - `clear_last_route()`
   - 在 `begin_turn()` 时清空上一轮路由结果，保证 `last_route` 表示的是当前轮的最终入口决策

2. 在 `DialogueEngine` 主入口记录显式路由
   - 文件：
     `customer-service-backend/atguigu/engine/dialogue_engine.py`
   - 在以下分支显式记录：
     - `validation_failed -> clarify`
     - `task_turn_selected -> task`
     - `knowledge_turn_selected -> knowledge`
     - `chitchat_turn_selected -> chitchat`
     - `object_requires_intent -> clarify`
     - `object_slot_handoff -> task`
     - `object_message_during_active_task -> task`
     - `runtime_exit_requested -> chitchat`

3. 回归 runner 优先读取显式路由
   - 文件：
     `functional-test-workspace/scripts/api-runner.js`
   - `inferTrack()` 现在优先读取 `state.last_route.kind`
   - `summarizeState()` 现在把 `last_route` 也纳入快照
