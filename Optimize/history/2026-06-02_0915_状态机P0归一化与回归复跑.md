# 2026-06-02 09:15 状态机 P0 归一化与回归复跑

## 本轮目标

在不回到 badcase 补丁路线的前提下，继续推进状态机 P0：

1. 稳住本轮已完成的 engine 拆分
2. 增加一层清晰、可解释的计划归一化协议
3. 用正式回归确认这轮改造没有把主链打散

## 为什么现在做

上一轮已经把 `dialogue_engine.py` 从超大模块拆回到可控体量，但 planner 输出仍然存在“结构合法、语义不稳”的情况，尤其是：

- 退出当前上下文时，LLM 可能给 `cancel_flow`
- 代码层虽然能执行，但会把“运行时退出”混进“任务轨道”

这类问题如果继续靠下游补丁收，会再次把架构拖回去，所以本轮先补协议层。

## 实际改动

1. 保持新的 engine 拆分结构，并先做编译校验
2. 新增 `customer-service-backend/atguigu/plan/turn_plan_normalizer.py`
   - 职责：把 planner 产出的 `TurnPlan` 归一化为运行时控制协议
   - 当前规则：
     - 当只有一个 `cancel_flow`
     - 且当前确实存在运行时上下文
     - 归一化为 `directive.exit_runtime`
3. 接入 `DialogueEngine`
   - planner 输出后，先经过 `TurnPlanNormalizer`
   - 再进入 validator / state machine / handler
4. 调整 `turn_plan.jinja2`
   - 强化 `cancel_flow` 与 `directive.exit_runtime` 的边界
   - 把“先不继续 / 全部退出 / 重新开始”这类表达更明确地导向 `directive`
5. 更新协作文档
   - `IMPLEMENTATION_PLAN.md`
   - `SESSION_HANDOFF.md`
   - `Optimize/06_第三阶段优化计划.md`
