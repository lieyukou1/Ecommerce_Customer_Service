# 本轮目标

继续第三阶段 P0，但不直接下潜去改 `task/processor` 或 `task/executor` 的业务执行逻辑，而是先把 task 收口结果做成显式可观测状态。

# 为什么现在做

前一轮已经确认了一件很关键的事：

- 很多回归失败并不是“没进 task”
- 而是“已经进了 task，但 task 在当轮内等待补槽位或直接收口”

如果只看：

- `last_route`
- `conversation_state`
- `active_task`

就很容易把两类问题混在一起：

1. task 根本没走到
2. task 走到了，但状态在当轮结束前又发生了收口变化

所以这一轮优先补观测层，而不是直接改更深的执行逻辑。

1. `waiting_for_slot`
