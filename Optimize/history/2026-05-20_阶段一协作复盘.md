# 2026-05-20 阶段一协作复盘

## 1. 这次协作解决了什么

这轮协作最后真正完成的，不只是“想明白一个方向”，而是把第一阶段最容易悬空的几件事都落到了地上：

1. 明确了项目不是做老师原电商系统复读，而是做物业场景完整映射
2. 明确了前端、中台、数据库、部署这几层分别要承担什么
3. 把前端和中台实际改出来了，而不是只写方案
4. 把数据盘做成了可演示状态
5. 把虚拟机部署和验收跑通了
6. 把老师体系里容易混淆的“AI 客服后端”和“业务中台”拆清楚了

## 2. 这次最明显的痛点

### 2.1 项目目标在前几轮里反复漂移

最开始的表述在下面几种目标之间来回切换过：

- 做阶段方案
- 做实践指导书
- 复现老师前端与中台
- 做物业化迁移系统
- 只做中台展示
- 最终还要为智能管家主体做前置准备

这类漂移不是谁做错了，而是“已有老师骨架 + 新业务迁移 + 多系统并存”天然容易糊在一起。

这次的经验是：

- **既有骨架项目，一上来必须先钉死“本轮真正交付物是什么”**

## 3. 第二个卡点：系统边界很容易混

这次至少有三套东西同时存在：

1. 老师资料目录
2. 我们正在改造的物业前端与物业中台
3. 后续还没真正开工的智能管家主体

如果不先明确边界，就很容易出现这些问题：

- 把“中台展示可用”误当成“智能管家已经开工”
- 把“客服后端的架构要求”直接套到“业务中台”上
- 把“阅读区目录”和“正式项目目录”混用

这次学到的最硬的一条经验是：

- **既有系统迁移项目，必须先区分阅读源、改造源、正式项目源。**

## 4. 第三个卡点：老师文档、老师代码、老师部署残留并不天然一致

老师资料很全，但真实推进时仍然有三层需要交叉核验：

1. 老师文档怎么说
2. 老师现有代码怎么写
3. 当前环境里实际残留了什么配置

这次非常典型的残留包括：

- 前端代理仍指向老师公网地址
- 数据库默认地址仍保留老师虚拟机 IP
- 顶层 compose 不能直接拿来当唯一真相

经验是：

- **已有骨架项目不要只信文档，也不要只信代码，要把部署配置单独当成第三个事实源。**

## 5. 第四个卡点：没有验收记录，很多“看起来通了”其实并不稳

如果这次只停留在浏览器能打开，后面一定会留下隐患：

- 数据有没有真的落库
- 写接口是不是成功
- 异常路径是不是明确
- 前端到底连的是本机还是虚拟机
- 中台和 MySQL 之间到底走的是公网还是 Docker 内网

这次把验收记录单独落成文件以后，很多判断一下子就稳定了。

可复用经验：

- **既有骨架迁移类项目，验收记录不是收尾材料，而是阶段内稳定边界的核心材料。**

## 6. 第五个卡点：同步 / 异步问题非常容易被“老师后端”四个字带偏

你后面提的这个问题特别有价值。

因为老师体系里其实不是“所有后端都异步”，而是：

- AI 客服后端异步
- 电商业务中台同步

如果不把这点拆清楚，就很容易出现两种偏差：

1. 误以为现在的物业中台技术路线不对
2. 或者为了追求“看起来高级”，提前重写已经稳定的中台

这次沉淀下来的经验是：

- **判断是否对齐老师，不能只看技术名词，要先看对齐的是老师哪一层。**

## 7. 这次做对了什么

1. 最终把目标从“继续写计划”拉回到了“真实系统落地”
2. 迁移时没有把老师骨架完全打散
3. 数据、接口、前端语义、部署链路都形成了闭环
4. 发现疑点时继续向下验证，而不是靠猜
5. 在阶段末留下了能复用的验收记录

## 8. 后续可直接复用的工作习惯

以后再做“基于既有系统二开/迁移”的项目，建议默认沿用下面几个习惯：

1. 开始前先写清楚本轮交付物是“方案 / 指导书 / 代码 / 部署 / 验收”里的哪一个
2. 先锁三类目录：
   - 老师资料只读目录
   - 当前改造工作目录
   - 正式项目主体目录
3. 先锁系统边界：
   - 这轮做前端壳
   - 做业务中台
   - 还是做 AI 主体
4. 先跑一轮部署与联调，再补经验总结
5. 每个阶段都至少留一份正式验收或状态记录

## 9. 对方法论的影响

这次经验说明，第一阶段方法论确实还可以继续增强，但**不建议立刻直接覆盖正式方法论**。

更稳的方式是：

1. 先把这次经验提炼成候选更新项
2. 等第二阶段再跑出更多真实样本
3. 再决定哪些内容应该并入正式方法论

原因很简单：

- 这次经验非常有价值，但目前仍然主要集中在“已有骨架迁移 + 多系统边界澄清 + 部署验收前置”这类问题上

它们已经足够成为强候选，但还适合先做增量沉淀，而不是直接改总文档。

## 10. 一句话总结

这次最宝贵的，不只是把物业系统前端和中台做出来了，而是我们摸清了一类项目的真实推进规律：

**当项目不是从零搭，而是站在老师骨架、已有代码和已有部署链路上开发时，真正难的不是写代码本身，而是先把边界、目标、目录、环境和验收钉牢。**
