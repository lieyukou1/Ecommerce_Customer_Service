# 本轮目标

在不继续追着单条 badcase 打补丁的前提下，先对 `DialogueEngine` 做第一刀可读性重构，把当前已经补得比较重的主入口整理成更清楚的阶段化流水线。

# 为什么现在做

第三阶段当前唯一 P0 是方案 B：显式会话状态机重构。

上一阶段的问题不是“某一个 if 写错了”，而是：

1. `engine` 同时承担了入口分流、状态切换、执行调度、收尾更新
2. 相同模式的逻辑在文本路径和对象路径里重复出现
3. 后续如果继续在这个基础上追加状态机接管，阅读成本会越来越高，架构图也画不清

所以这一步的目标不是“把所有回归打绿”，而是先把主链整理成后续能继续接管的形状。

# 实际改动

改动文件：

- `D:\Desktop\SGG_Project\Ecommerce_Customer_Service\customer-service-backend\atguigu\engine\dialogue_engine.py`

本轮只做结构性整理，尽量不改业务策略：

1. 抽出统一入口分发
   - 新增 `_route_turn()`
   - `hand_dialogue()` 现在只保留：
     - session/turn 准备
     - turn 路由
     - turn commit

2. 收口文本校验失败路径
   - 新增 `_respond_validation_failure()`
   - 把原先分散在 `_handle_text_message()` 里的
     `record_route -> mark_clarifying -> clarify_responder`
     抽成一段固定动作

3. 收口任务执行路径
   - 新增 `_execute_task_commands()`
   - 把文本任务和对象任务里重复的：
     - 记录路由
     - 标记 transitioning
     - 调 `task_handler.handle()`
     - 执行 `_finalize_task_transition()`
     统一放在一处

4. 收口路由记录
   - 新增 `_record_decision()`
   - 让 `knowledge / chitchat / clarify` 的路由写法一致

5. 收口对象澄清路径
   - 新增 `_respond_object_requires_intent()`
   - 新增 `_describe_object()`
   - 新增 `_is_object_switch_during_active_task()`
   - 让对象消息处理从“大片条件分支”变成：
     - 预处理对象
     - 判断是否任务中切对象
     - 判断是否可直接补 slot
     - 判断是否在 active task 中继续交给 task handler
     - 否则统一走 clarify

6. 小心处理 `pending_turn`
   - `hand_dialogue()` 中补了空判断，避免后续如果 turn 生命周期再调整时这里变成隐式风险点
