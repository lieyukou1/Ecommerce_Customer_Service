# 2026-06-01 第三阶段第一轮退出机制优化报告

## 1. 本轮唯一主目标

补齐退出机制，让住户可以通过自然语言退出当前运行时上下文，而不再依赖手动删库。

## 2. 为什么现在做

原因很明确：

- 当前项目已经进入第三阶段：能跑，但状态收口不完整
- `focused_object`、`active_task`、`paused_tasks`、`active_system_task` 都存在“进去后不易退出”的问题
- 这已经不再是单点 badcase，而是状态机层面的稳定性问题

按照优化阶段方法论，这类问题应优先于 prompt 细修和效果美化处理。

## 3. 本轮改动

### 3.1 状态层

修改：

- [state.py](D:/Desktop/SGG_Project/Ecommerce_Customer_Service/customer-service-backend/atguigu/domain/state.py)

新增能力：

- `clear_paused_tasks()`
- `clear_focused_object()`
- `has_runtime_state()`
- `clear_runtime_state()`
- `runtime_state_labels()`

并让：

- `reset_running_state_for_new_session()`

改为复用统一清理逻辑。

### 3.2 引擎层

修改：

- [dialogue_engine.py](D:/Desktop/SGG_Project/Ecommerce_Customer_Service/customer-service-backend/atguigu/engine/dialogue_engine.py)

新增能力：

- 在文本消息进入 planner 之前，先识别显式退出类文本
- 若命中退出意图，则直接清空 runtime state 并返回确认消息

当前首批支持的退出口令方向包括：

- `取消`
- `退出`
- `重来`
- `重新开始`
- `清空状态`
- `不看这个了`

并补充了更贴近日常表达的变体，例如：

- `取消吧`
- `不用了`
- `不继续了`
- `先这样`
- `先这样吧`
- `先不办了`

同时把退出确认文案从“内部状态播报口吻”改成了更自然的住户可读回复。

### 3.3 业务内取消路径

修改：

- [processor.py](D:/Desktop/SGG_Project/Ecommerce_Customer_Service/customer-service-backend/atguigu/task/command/processor.py)

补齐能力：

- `cancel_flow` 不再只结束 `active_task`
- 现在会同步清理：
  - `paused_tasks`
  - `focused_object`

这意味着“统一退出入口”和“业务流内部取消入口”现在都具备了状态收口能力。

### 3.4 Prompt 对齐

修改：

- [turn_plan.jinja2](D:/Desktop/SGG_Project/Ecommerce_Customer_Service/customer-service-backend/atguigu/prompt/jinja2/turn_plan.jinja2)

补充规则：

- 当住户表达“退出当前上下文”时，不要强行输出 `task / knowledge / chitchat`
- 保持三个字段为 `null`，交由系统层统一处理

### 3.5 API 型知识回复去调试味

修改：

- [responder.py](D:/Desktop/SGG_Project/Ecommerce_Customer_Service/customer-service-backend/atguigu/knowledge/responder.py)

补齐能力：

- 服务项目/工单类 API 知识回复不再直接平铺 `focused_object` 字段
- 改为整理成更自然的摘要句式
- 给后续连续追问和对象切换优化留出更清晰的输出基线

## 4. 本轮没有做什么

本轮刻意没有做：

- knowledge 轨的对象切换
- service_item 连续追问改写
- cancel 语义细分
- 测试态重置 API
- 大范围 prompt 重构

## 5. 验证方式

本轮已做验证：

- 使用 `py_compile` 对以下文件做语法校验：
  - `customer-service-backend/atguigu/domain/state.py`
  - `customer-service-backend/atguigu/engine/dialogue_engine.py`
  - `customer-service-backend/atguigu/knowledge/responder.py`
  - `customer-service-backend/atguigu/task/command/processor.py`

本轮尚未完成的验证：

- 真实前端/接口联调下验证“取消/退出”是否能正确清空 runtime state
- 联调验证 `cancel_flow` 场景是否会同步清掉 `paused_tasks / focused_object`
- 联调确认退出后是否仍保留历史 turns 且不再残留焦点对象
- 观察 API 型知识回复在前端里是否仍有机械拼接感

## 6. 当前结论

这轮已经把“统一退出 runtime state”的最小主链补上了。

也就是说，项目现在第一次具备了：

- 不删库退出
- 不靠 session 超时退出
- 不把退出意图硬丢给 planner 猜测

但它仍然只是第一步，还不等于上下文切换机制已经完整。

## 7. 后续优先级

下一批应继续按下面顺序推进：

1. knowledge 轨对象切换
2. 连续追问时不再重复静态摘要
3. 取消语义细分
4. 测试态状态重置入口
5. 退出与切换的回归样例固化
