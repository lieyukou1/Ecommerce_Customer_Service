# 本轮目标

按第三阶段 P0 排期，先做“回归断言分型”的第一批落地：

1. 让回归能显式判断 task 是挂起还是当轮完成
2. 只修正已经确认属于“语义误报”的场景预期
3. 不下潜改 `task/command/processor.py`、`task/flow/executor.py`

# 为什么现在做

昨晚已经完成提级判断，结论很明确：

- 不是所有 task 失败都代表执行层坏了
- 很多失败其实是：
  - 这轮被判成了 `task`
  - 但 task 在当轮就完成
  - 最终落回 `focused_knowledge`

如果回归断言仍然只会写：

- `expected_track = task`
- `active_task_flow_id = xxx`

那它实际上把两类语义混成了一类：

1. 进入 task 并等待补槽位
2. 进入 task 并当轮完成

这会持续制造误报，也会把我们往错误的执行层方向推。

## 1. runner 新增 task outcome 显式断言字段

在 `api-runner.js` 里新增：

- `expected_task_outcome_kind`
- `expected_task_outcome_kinds`
- `expected_task_outcome_flow_id`
- `task_outcome_reason_includes`

这样场景现在可以明确表达：

- 这一轮是否应出现 `last_task_outcome`
- 它是 `waiting_for_slot` 还是 `completed_with_focus`
- 触发的是哪类 task 语义

## 2. 首批场景预期调整

本轮只调整已经确认属于“即时完成型 task”的几类场景：

1. `human_handoff`
2. `service_item_detail_query`
3. `service_progress_tracking`
4. `work_order_status_query`

对应策略：

- 不再强行要求 `active_task_flow_id`
- 允许 `expected_tracks` 包含 `task`
- 显式要求 `expected_task_outcome_kind = completed_with_focus`
- 用 `task_outcome_reason_includes` 标明触发的是哪类即时完成语义

## 3. 保留可疑 case 不洗白

本轮**没有**把下面这种 case 改成通过：

- `work_order_urge_submission` 在少数场景里异常出现 `completed_with_focus`

原因是昨晚已经判断过：

- 这类 case 还可能是真正的不一致点
- 现在不应该为了降低失败数，把它们一起放宽掉
