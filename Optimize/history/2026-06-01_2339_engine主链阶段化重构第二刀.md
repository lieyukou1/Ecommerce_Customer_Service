# 本轮目标

继续第三阶段 P0，做 `DialogueEngine` 第二刀重构。

本轮只服务于方案 B 的主线：把 `engine` 主链进一步整理成更接近“阶段化状态机入口”的形状，不插入新的横向需求，也不为了多过几个 badcase 去继续补局部判断。

# 为什么现在做

上一刀已经把重复动作收了起来，但从阅读视角看，`engine` 里仍然有两个问题：

1. 文本入口仍把“准备上下文、生成计划、校验、决策、执行”揉在一个函数里
2. 对象入口虽然短了一些，但“构建上下文”和“执行上下文”还没分层

这会影响两件事：

- 后续继续接状态机时，入口层不够清楚
- 最后要画架构图时，读代码的人看不出稳定阶段

# 实际改动

改动文件：

- `D:\Desktop\SGG_Project\Ecommerce_Customer_Service\customer-service-backend\atguigu\engine\dialogue_engine.py`

本轮继续只做结构性整理：

1. 新增文本入口上下文对象
   - `TextTurnContext`
   - `TextTurnBuildResult`

2. 把文本入口拆成两段
   - `_prepare_text_context()`
   - `_build_text_turn_context()`

现在文本主链更接近：

- 先处理运行时退出
- 再准备文本上下文
- 再构建 turn context
- 最后执行 route

3. 新增对象入口上下文对象
   - `ObjectTurnContext`

4. 把对象入口拆成“构建上下文”和“执行上下文”
   - `_build_object_turn_context()`
   - `_execute_object_turn()`

5. 把对象路径里的三类动作再明确分开
   - `_handle_object_switch_during_active_task()`
   - `_handoff_object_slots()`
   - `_continue_active_task_with_object()`

这样对象入口现在可以按顺序读成：

- 构建对象上下文
- 判断是否任务中切对象
- 判断是否补 slot
- 判断是否继续 active task
- 否则走对象澄清

6. 修正了本轮重构中自检发现的一个问题
   - 一开始文本路径会在校验失败时重复调用澄清回复
   - 已在本轮内收口，改成 `TextTurnBuildResult` 只返回失败原因，由外层统一回复

# 本轮观察

这轮有一个很重要的信号：

当前很多失败并不是“没识别到 route”，而是：

- `last_route` 已经是 `task`
- 但最终状态又回到 `focused_knowledge`
- 或者任务在本轮内被快速收口，`active_task` 没有稳定沉淀

这说明我们接下来的重点应该继续放在：

- 显式状态迁移的落点
- 任务执行后的状态收口语义

而不是回去补更多关键词命中或局部条件分支。

# 未完成项

# 下一步优先级

1. 继续方案 B 的 P0 主线
   - 重点不再是继续拆小函数本身
   - 而是把“状态迁移”这一层从执行细节中再往外提一层

2. 优先盯住架构级信号
   - `route=task` 但 `active_task` 未稳定沉淀
   - `task` 在当轮快速收口后，最终状态如何显式表达

3. badcase 继续只作为门禁
   - 用来判断状态机接管有没有覆盖到核心结构问题
   - 不作为本轮是否成功的唯一指标
