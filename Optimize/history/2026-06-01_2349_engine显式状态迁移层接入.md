# 本轮目标

继续第三阶段 P0，往 `engine` 主链里接入一层真正显式的“状态迁移计划”，不再让状态落点只散落在各个执行分支和收口判断里。

# 为什么现在做

前两刀已经完成了两件事：

1. 把 `DialogueEngine` 入口读起来更顺
2. 把文本入口 / 对象入口整理成更清楚的阶段链

但还有一个核心问题没有解决：

- 当前代码虽然已经能看出 route 是什么
- 但“这条 route 最后把会话状态落到哪里”仍主要是直接在各处调用 `mark_* / transition_to()`

这会导致：

1. 最终状态落点的规则分散
2. 后面画状态机图时，很难说清“谁在负责决定迁移”
3. 任务场景下 `last_route=task` 但最终落回 `focused_knowledge` 时，代码上不够显式

所以这一轮的目标不是继续拆函数，而是把“迁移决定”本身提炼出来。

# 实际改动

改动文件：

- `D:\Desktop\SGG_Project\Ecommerce_Customer_Service\customer-service-backend\atguigu\engine\dialogue_engine.py`

本轮主要新增并接入了一层显式状态迁移模型：

1. 新增 `StateTransitionPlan`
   - 字段：
     - `target_state`
     - `event`
     - `reason`

2. 新增统一应用入口
   - `_apply_route_decision()`
   - `_build_route_transition_plan()`
   - `_apply_state_transition()`

现在 route 决策不再只是：

- 记一条 `last_route`
- 然后直接手写 `mark_clarifying / mark_chitchat / mark_focused_knowledge / mark_transitioning`

而是变成：

- 先记录 route
- 再生成一个迁移 plan
- 最后统一 apply

3. 文本路径接入新的迁移层
   - `validation_failed`
   - `knowledge`
   - `chitchat`
   - `task`
   这些 route 现在都通过统一迁移层落状态

4. 对象澄清路径接入新的迁移层
   - `object_requires_intent` 不再直接手写 `mark_clarifying`

5. 任务收口路径接入新的迁移计划
   - `_finalize_task_transition()` 不再直接写一串 `mark_*`
   - 新增 `_build_task_finalize_transition_plan()`
   - 先生成“任务执行后的最终迁移计划”，再统一 apply

这一步很关键，因为它把：

- `task_waiting_for_slot`
- `task_active`
- `system_flow_active`
- `focused_context`
- `idle`

这些收口结果，第一次明确成了“显式可读的状态迁移计划”

# 本轮最重要的收获

这一轮最重要的不是 pass 数变化，而是代码结构上已经出现了清晰边界：

1. route 是什么
2. route 应该先进入哪个会话状态
3. task 执行后最终停在哪个状态

这三件事开始被拆开了。

也就是说，`engine` 现在已经不只是“可读”，而是开始具备真正的状态机入口骨架。

# 还剩什么

当前剩下的主要不是再拆入口，而是两类更核心的问题：

1. `task` 路径里的业务执行与状态沉淀还不完全一致
   - 典型现象：`last_route=task`，但 `active_task` 没稳定留下
   - 这说明接下来要继续盯 task 执行后状态承载语义

2. 文本 / 对象两条入口虽然都接了迁移层，但还没有完全共用一套更高层的“会话仲裁协议”

# 下一步优先级

1. 继续 P0 主线
   - 重点转向 task 执行后的状态沉淀语义
   - 看清楚哪些任务是“开始即完成”，哪些任务应该稳定进入 `active_task`

2. 不回退去补局部 badcase 规则
   - badcase 继续只作为门禁和信号源

3. 为最终架构图做准备
   - 现在已经可以把 `engine` 主链大致画成：
     - turn entry
     - context build
     - route decision
     - transition planning
     - track execution
     - finalize transition
