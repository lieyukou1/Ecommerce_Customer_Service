# 状态字段进入退出对照表

这份表不是产品说明，而是后续修改退出机制时的工程参照。

## 1. 核心状态字段

| 字段 | 含义 | 主要进入条件 | 当前退出条件 | 当前问题 | 后续建议 |
| --- | --- | --- | --- | --- | --- |
| `focused_object` | 当前聚焦对象，通常是 `work_order` 或 `service_item` | 前端发送对象消息；引擎写入焦点对象 | 新 session 重置、统一退出、`cancel_flow` 清理 | 极易黏住，知识轨会反复复用旧对象 | 下一步继续补对象切换、取消知识上下文 |
| `active_task` | 当前正在执行的业务 flow | planner 输出 `start_flow` / `resume_flow` | `end_active_task()`、`cancel_flow`、新 session、统一退出 | 业务内取消和全局退出都已可清理，但仍需联调观察是否被后续流程重新写回 | 继续做分层取消语义 |
| `paused_tasks` | 被中断后压栈的业务任务 | 新 flow 启动或恢复时中断旧任务 | 恢复后弹栈、新 session、统一退出、`cancel_flow` 清理 | 住户感知不到它的存在，也缺显式清空入口 | 增加“清空任务栈”语义和可观测性 |
| `active_system_task` | 开场白、中断说明、恢复说明、取消说明等系统 flow | `CommandProcessor` 激活系统流程 | `end_active_system_task()`、新 session、统一退出 | 一般能结束，但和业务退出语义没有完全打通 | 保持系统流程短命，避免跨轮残留 |
| `pending_turn` | 当前轮的暂存 turn | 每轮 `begin_turn()` 时生成 | `commit_turn()` 后置空 | 正常情况下不会残留 | 保持现状 |
| `sessions` | 历史会话列表 | 新 session 创建 | 不主动删，只关闭当前 session | 正常 | 保持现状 |
| `current_session_id` | 当前活跃 session | 启动 session 时写入 | 关闭 session、新 session 重建 | 正常 | 保持现状 |

## 2. 状态字段与主要读写模块

| 字段 | 写入模块 | 读取模块 | 风险说明 |
| --- | --- | --- | --- |
| `focused_object` | `engine/dialogue_engine.py` | `planner.py`、`turn_validator.py`、`knowledge/*`、`clarify/*`、`task/flow/executor.py` | 读它的模块很多，所以一旦黏住会全链路受影响 |
| `active_task` | `task/command/processor.py` | `dialogue_engine.py`、`task/handler.py`、`task/flow/executor.py` | 业务流入口和恢复逻辑都依赖它 |
| `paused_tasks` | `state.py`、`task/command/processor.py` | `planner.py`、`task/command/processor.py` | 主要是恢复机制会读 |
| `active_system_task` | `task/command/processor.py`、`task/flow/executor.py` | `state.py`、`task/action/builtin/response.py` | 多用于系统提示语 |
| `pending_turn` | `dialogue_engine.py` | `planner.py`、`clarify/*`、`knowledge/*`、`chitchat/*` | 是当前轮输入的主要来源 |

## 3. 当前退出机制的分层口径

### 3.1 本轮已实现

本轮先实现：

- **统一运行时退出**
- **业务内取消路径补齐状态收口**

触发方式：

- 住户显式输入“取消”“退出”“重新开始”“清空状态”等文本
- 业务 flow 内部明确触发 `cancel_flow`

清理范围：

- `focused_object`
- `active_task`
- `paused_tasks`
- `active_system_task`

保留内容：

- 历史 `sessions`
- 历史 `turns`

### 3.2 本轮尚未实现

下面这些还没有完全补齐：

- 只退出 `focused_object`，但保留任务流
- 只取消业务 flow，但保留知识上下文
- 从 `service_item A` 自动切换到 `service_item B`
- “那空调清洗呢”这种跨对象追问的自然切换
- 测试态 API 重置入口
- API 型知识回复虽然已去掉字段平铺，但连续追问仍未具备真正对象切换能力

## 4. 后续改动建议顺序

| 顺序 | 事项 | 目标 |
| --- | --- | --- |
| 1 | 统一退出机制 | 先解决“进去了出不来” |
| 2 | 取消语义细分 | 区分取消 flow / 清焦点 / 全局重置 |
| 3 | 对象切换机制 | 解决旧对象黏住问题 |
| 4 | knowledge 轨连续追问优化 | 解决静态摘要重复输出 |
| 5 | 回归样例固化 | 防止退出与切换问题回潮 |
