# 优化与稳定化阶段方法论

## 1. 这份文档解决什么问题

这份文档不是某一个项目的复盘记录，也不是某一次 bug 修复的流水账。

它要解决的是一个更具体的问题：

- 当项目已经建立在老师代码、教学骨架或既有主线之上，并进入“能跑，但不稳”的阶段时，怎样和人或 AI 一起高效推进优化
- 怎样减少返工
- 怎样判断现在该补规则、改 prompt、改代码，还是先停下来补评测
- 怎样让协作过程留下可复用的方法，而不只是留下代码改动

这份文档默认服务的是**既有骨架项目**，但示例仍然优先使用 AI 应用、数据系统、RAG、NL2SQL、Agent 编排这类场景。

母文档已经给出了跨阶段共通原则的简版定义；这份文档不再重复做总定义，而是把这些原则展开成优化阶段真正可执行的动作、判断和收口方式。

原因很简单：

- 这类项目最容易出现 badcase 驱动迭代
- 最容易出现“提示词、规则、代码、数据模型”交织的问题
- 也是人机协作最容易失控的区域

## 2. 一页纸速查表

如果是在新 session 开始时快速复习，或者需要让 AI 先抓住核心，不必先通读全文，先看这一页。

### 2.1 遇到 badcase 时

1. 不要立刻改 prompt
2. 先按层分问题：
   - 数据层
   - 检索层
   - 编排层
   - 提示词层
   - 守卫层
   - 展示与体验层
3. 再决定该补规则、改 prompt、改代码还是补数据
4. 修完后决定是否进入回归集

### 2.2 新想法出现时

1. 先放进 backlog
2. 检查当前最高优先级未完成项
3. 如果不是当前最高优先级，不直接开做

### 2.3 每轮优化开始前

1. 写明唯一主目标
2. 写明本轮不做什么
3. 明确验收标准
4. 让 AI 或协作者先复述一次目标和边界

### 2.4 每轮优化结束后

1. 检查是否目标漂移
2. 跑 smoke
3. 跑相关回归
4. 更新 plan
5. 写 history
6. 如有必要，更新下一轮 session 的入场摘要

### 2.5 什么时候该暂停

如果同类问题连续 2 到 3 轮仍然只能靠特判维持，就暂停继续补洞，重新判断是不是该升层处理。

### 2.6 什么时候算这一轮优化可以收口

至少同时满足下面几条，才算这一轮不是“改了一半”：

1. 当前主目标已经验收
2. 相关 smoke 和必要回归已经跑过
3. 新增 badcase 已决定是否入库
4. plan、history、上下文锚点至少更新到能交接
5. 当前没有更高优先级未完成项被你用低优先级想法绕过去

## 3. 先分清三类工作，不要混着做

项目协作里最容易失控的，不是难题本身，而是把不同类型的工作混在一轮里做。

常见三类工作：

1. 功能拓展
2. 质量优化
3. 结构重构

这三类可以长期并存，但不应该在同一轮里没有边界地并推。

原因：

- 功能拓展会引入新变量
- 质量优化需要稳定观察对象
- 结构重构会改变定位路径

如果把它们混在一起，最后很难判断：

- 成功率上升是因为功能补齐了，还是 prompt 改好了
- 系统变差是因为数据变了，还是状态流改坏了
- badcase 重现不了，是因为修好了，还是因为路径已经变了

更稳的顺序通常是：

1. 先让主链路跑通
2. 再补最小可用可观测性
3. 再处理稳定性和 badcase
4. 最后做结构整理和抽象提升

例子：

- 如果 SQL 生成链路还没有稳定 validate 和 execute 的闭环，就不应该先大改 prompt 示例和数仓扩展
- 如果文档问答的主体识别还没有稳定，就不应该先拆很多模块做结构美化

## 4. 每一轮优化必须有唯一主目标

一轮优化最怕的是目标太多。

不好的描述：

- 这轮把效果、结构、演示体验都提升一下
- 这轮顺便把几个长期问题一块做了

更好的描述：

- 这轮唯一目标是让 SQL 修正后重新校验
- 这轮唯一目标是建立 badcase 诊断工具
- 这轮唯一目标是清理非领域 prompt 示例

为什么要这样做：

- 优化必须可归因
- 没有唯一主目标，就没有清晰验收
- 协作时，人与 AI 都容易被新问题牵走

建议每轮开始前写一句：

`本轮唯一主目标是什么，什么明确不做。`

比如：

- 本轮只做多轮 follow-up，不扩 seed
- 本轮只做评测入口，不改主链路
- 本轮只做 prompt 示例清理，不改数据模型

## 5. 唯一主目标不是写下来就够了，还要防止目标漂移

真实协作里，目标最常见的失控方式不是一开始就定错，而是中途慢慢漂移：

- AI 看到顺手能改的地方，顺便多改了几处
- 人在执行过程中觉得“这个也很重要，不如一起做”
- 修一个问题时，顺手把相关模块也改了

所以“唯一主目标”必须配一个收口动作。

### 5.1 每轮结束后的目标漂移检查

每轮结束时至少问三个问题：

1. 实际改动是否超出了开始前声明的边界
2. 超出的改动是否影响了其他链路
3. 这些超出部分是否需要补回归验证或补 history 说明

### 5.2 什么时候必须补额外验证

如果本轮出现以下情况之一，就不能只用原计划的验证：

- 改动触到了其他模块
- 改动影响了状态流、脚本入口或主图拓扑
- 改动改变了原来的拒答/澄清边界
- 改动涉及公用 helper、normalizer、shared support

### 5.3 一个简单的实践动作

每轮优化开始时，写：

- 本轮要改什么
- 本轮不改什么

每轮结束时，再写：

- 实际改了什么
- 哪些属于顺手扩散
- 顺手扩散部分是否已补验证

这个动作能显著减少“以为只改了一点，结果影响了半条链”的情况。

## 6. 先建立边界，再追求能力

很多系统前期不稳定，不是因为“能力不够”，而是因为“边界不清”。

必须先回答的问题包括：

- 什么问题支持，什么问题不支持
- 什么输入拒答，什么输入澄清
- 哪些表和字段允许使用
- 哪些输出格式必须拦截
- 哪些值没落库时要停下来澄清

边界清楚后，系统往往会立刻稳定很多。

这比一开始追求“什么都能答”重要。

原因：

- 稳定拒答，比偶尔答对更有产品价值
- 没有边界的系统，badcase 会无限发散
- 很多看似复杂的问题，本质上只是没有提前拦截

适合用边界规则解决的问题：

- 非当前领域问题拒答
- 多语句 SQL 拦截
- 未落库实体澄清
- 只允许只读查询

不适合用边界规则硬扛的问题：

- 同类查询反复理解错
- 已知可回答问题持续召回不到
- 核心结果依赖大量临时特判

这类问题通常要往更深层处理。

## 7. 看到 badcase 后，先分层，再决定怎么修

badcase 驱动优化是正常工程方式，但不能一上来就改 prompt。

先判断它属于哪一层。

常见分层如下。

### 7.1 数据层

典型问题：

- 数仓建模不贴业务
- seed 太薄
- 元数据缺失
- 索引里的值不完整
- 样本数据和演示场景不匹配

典型修法：

- 补表
- 补字段
- 补 seed
- 重建元知识
- 修索引内容

### 7.2 检索层

典型问题：

- 别名识别不到
- 召回不准
- 重复文档挤占结果
- 某类值能搜到但排得太后

典型修法：

- 补 alias 规则
- 改召回关键词扩展
- 优化去重策略
- 调整 rerank 输入质量

### 7.3 编排层

典型问题：

- 修正后没有重新验证
- 错误节点直接中断，没有稳定返回
- 多轮状态没有保存
- 某类问题应走澄清却被放进主链路

典型修法：

- 改 graph 路由
- 改状态字段
- 改 service 层包装
- 改 checkpoint 或重试策略

### 7.4 提示词层

典型问题：

- 输出格式不稳定
- 同义词映射容易漂
- 比较类问题总会扩出无关指标
- 示例残留旧域，模型带偏

典型修法：

- 加约束
- 改示例
- 补负面指令
- 精简输出要求

### 7.5 守卫层

典型问题：

- 本应拒答的问题进入主链路
- 本应澄清的问题被模型强行回答
- 危险 SQL 没有被拦截

典型修法：

- 加 domain guard
- 加 grounding guard
- 加 security guard

### 7.6 展示与体验层

典型问题：

- 结果其实对，但报错不稳定
- SSE 中断
- 用户只看到堆栈
- 返回结果格式很难看

典型修法：

- 统一错误返回
- 增加阶段流式状态
- 优化前端展示和消息结构

## 8. 什么叫“同类问题”，什么时候该升层处理

“同类问题连续 2 到 3 轮还只能靠特判维持，就该升层处理”这条原则很有用，但前提是你真的知道什么叫“同类”。

不要只按表面现象分类。

更好的判断顺序是：

1. 根因相同
2. 修法相同
3. 表现相似

优先级必须按上面的顺序判断，而不是反过来。

### 8.1 同类判定的启发

#### 第一优先：根因是否相同

例如：

- 都是因为状态没有保存
- 都是因为资产类型被误落到错误字段
- 都是因为旧领域示例把模型带偏

即使表现不同，也应先视为同类。

#### 第二优先：修法是否相同

例如：

- 都需要加 guard
- 都需要补 time semantic 规则
- 都需要改 SQL 回环逻辑

如果修法完全一样，也要警惕是不是已经进入该升层的时候了。

#### 第三优先：表现是否相似

例如：

- 都是“查不到”
- 都是“结果为空”
- 都是“对比类问题出错”

这只能作为最弱参考，因为表面相似的问题，根因很可能完全不同。

### 8.2 什么情况下该升层

以下情况通常说明该升层了：

- 你已经连续补了 2 到 3 个特判
- 每次修法都越来越像例外列表
- 同类问题修完这个，又冒出下一个变体
- 修法开始污染主链路可读性

这时就不要继续贴补丁，而要停下来问：

- 这是不是数据模型问题
- 这是不是编排缺机制
- 这是不是 prompt 结构本身有偏

## 9. 什么时候该补规则，什么时候该改 prompt，什么时候该改代码

这是协作优化里最核心的判断之一。

### 9.1 适合补规则的情况

- 业务边界明确
- 对错判断清晰
- 输入模式稳定
- 不需要模型自由发挥

例子：

- 非 REIT 主题拒答
- SQL 只允许 `SELECT` 和 `WITH`
- “产业园/高速公路”映射到底层资产类型
- 未录入项目名要澄清

这类问题用规则不是亡羊补牢，而是在明确产品边界。

### 9.2 适合改 prompt 的情况

- 模型方向大体对
- 但输出习惯不稳定
- 问题主要体现在语言层或格式层

例子：

- SQL 不要带 markdown fence
- 不要把同比问题改写成多个无关指标
- 不要输出旧领域示例语义

### 9.3 适合改代码或架构的情况

- 同类问题反复出现
- 不是一句规则能说清楚
- 涉及状态流、错误流、重试、持久化、路由

例子：

- `correct_sql` 后必须回到 `validate_sql`
- execute 错误必须稳定流式返回
- 多轮 follow-up 需要 checkpointer
- 脚本和主图的调用方式要兼容

### 9.4 适合补数据的情况

- 问题本来合理，但系统没有相应语义支撑
- 系统靠猜能偶尔答对，但不稳定
- 需要支持的查询根本没有足够字段或实体

例子：

- 项目维度太薄
- 分红事件事实表不存在
- region 或 project seed 不足

一个健康原则：

`规则和 prompt 可以修边界与表达，代码负责修流程与机制，数据负责修语义承载能力。`

## 10. 什么时候需要建立评测工具

评测工具不是越早越好，也不是越晚越好。

### 10.1 太早建立的风险

如果主链路都还没跑通，过早建设复杂评测平台通常收益很低。

原因：

- 成功和失败都还没定义清楚
- 测什么都不稳定
- 工具会比系统本身还复杂

这时最适合的是最轻量的手动 smoke 或脚本化单例验证，而不是全套平台。

### 10.2 太晚建立的风险

如果已经开始连续改 badcase，却还没有统一评测入口，就会出现：

- 修好 A，弄坏 B
- 总觉得在进步，但没有整体证据
- 大量时间花在重复手工试问题
- 同一个问题隔几轮又重新修

### 10.3 最合适的时机

当系统同时满足下面三个条件时，就应该建立最小评测：

1. 主链路已经能端到端跑通
2. 成功、失败、拒答、澄清已有基本定义
3. 已经开始出现重复 badcase 或重复验证动作

### 10.4 良好可用的评测工具应该怎么建

不要一开始追求“大一统平台”，先做最小可用评测。

建议分三层。

#### 第一层：Smoke Test

目标：

- 防止主链路回归

内容：

- 几条核心正向题
- 几条必须拒答题
- 几条必须澄清题

特点：

- 题少
- 跑得快
- 每次改动都能跑

#### 第二层：Regression Set

目标：

- 防止老问题复发

内容：

- 来自真实 badcase
- 修一个，收一个

特点：

- 代表真实问题
- 能验证“修复是否真的留住了”

#### 第三层：Exploration Set

目标：

- 发现新问题，不做硬门槛

内容：

- 长尾问法
- 新场景
- 未来想支持但还未完全稳定的问题

特点：

- 可以不每次全跑
- 更多用于发现方向，而不是卡 release

## 11. 为什么需要人机共同维护的回归测试集

一个动态题库会让协作进度更对齐。

这是以后特别值得长期坚持的一条原则：

`每次修掉一个值得记住的问题，都要判断它是否应该进入回归集。`

回归集最好不是纯人工维护，也不是纯自动收集，而是人机共管。

### 11.1 机器负责什么

- 从日志中抽候选 badcase
- 汇总失败阶段
- 提取 SQL、错误信息、trace
- 生成初步诊断报告

### 11.2 人负责什么

- 判断这个问题是否真的重要
- 判断它是偶发噪声还是系统性问题
- 判断它属于哪一层
- 决定是否加入回归集，以及加入哪一类

### 11.3 为什么不能全靠机器

因为不是所有异常都值得固化：

- 有些是外部服务抖动
- 有些是低价值长尾
- 有些不值得为了一个极端问题污染主链路

### 11.4 为什么不能全靠人工

因为纯人工容易漏：

- 漏日志
- 漏重复问题
- 漏掉相似 badcase 的聚类关系

更健康的工作流是：

1. 工具从日志中收集候选问题
2. 生成 badcase 批次报告
3. 人判断哪些进入回归集
4. 归档并标记类别

## 12. 回归题库应该怎么分层维护

不要把所有问题扔进一个大列表。

建议至少分成以下几类。

### 12.1 P0 核心链路集

作用：

- 每次改动必跑

特点：

- 只保留最关键的 5 到 10 题
- 代表当前系统最核心的价值链

### 12.2 Badcase 回归集

作用：

- 防止旧病复发

特点：

- 必须来自真实失败案例
- 修一个，入一个

### 12.3 边界行为集

作用：

- 保证系统不乱答

特点：

- 拒答
- 澄清
- 空结果
- 错误流

这类题经常被忽略，但很决定系统是否稳。

### 12.4 演示题集

作用：

- 保障面试或 demo

特点：

- 不一定最难
- 但必须高成功率、可解释

### 12.5 探索题集

作用：

- 发现未来问题

特点：

- 暂时不稳定
- 但值得持续观察

分层维护的好处是：

- 不同集合的门槛不同
- 不会把研究样本误当上线门槛
- 协作双方更容易知道当前关注点

## 13. 回归题不是只进不出，要有退役机制

一个回归集如果只进不出，最终会变成维护负担。

### 13.1 为什么需要退役机制

随着时间推移：

- 系统边界会变化
- 模块会重构
- 某些旧问题会自然失去意义
- 某些题已经不再验证当前逻辑

如果一题已经不能代表今天的系统风险，它就不该继续占据高频回归资源。

### 13.2 什么条件下可以考虑退役

满足以下任一情况时，可以考虑退役：

- 相关模块已被整体重构，旧 case 测的不是当前逻辑
- 原有产品边界已经不存在
- 旧 case 与新的更通用 case 高度重复
- 该 case 长期只是噪声，不再产生决策价值

### 13.3 退役不是删除历史

退役不等于彻底删掉。

更好的做法是：

- 从高频回归集中移除
- 标记为 archived / retired
- 在 history 或题库备注里写清退役原因

这样以后如果有人追问，仍然能追溯。

## 14. 日志、审计、诊断工具什么时候补最合适

日志不是善后工具，而是调优前置工具。

### 14.1 太早补重型监控平台的风险

- 主链路还没成形
- 会被基础设施工作拖慢
- 记录很多但没有用

### 14.2 太晚补日志的风险

- badcase 定位完全靠猜
- 修复前后无法对比
- 同一个问题会重复分析

### 14.3 比较好的节奏

推荐这样分阶段：

#### 阶段 A：主链路刚跑通

补最小日志：

- request id
- query
- final status
- failed stage

#### 阶段 B：开始连续调优

补审计：

- generated output
- final output
- correction history
- duration
- result count

#### 阶段 C：开始积累 badcase

补诊断工具：

- badcase batch collector
- status detector
- trace analyzer

一个简单判断：

`当你开始频繁说“这条为什么又坏了”时，就说明诊断工具该上了。`

## 15. 不要只记录“改了什么”，还要记录“为什么现在改，为什么现在不改”

高质量协作文档不只是 change log，更应该是 decision log。

每个重要决定都最好回答：

1. 为什么现在做
2. 为什么不是别的方案
3. 为什么有些东西现在不做
4. 未来在什么条件下再做

这样做的价值非常大：

- 避免几天后重复讨论同一个问题
- 让后来参与的人能理解取舍
- 让“暂停”也成为明确结论，而不是遗忘

例子：

- seed 扩展现在不做，因为会引入更多变量，先完成 P3
- 权限控制不做，因为当前样本系统不具备真实验证条件
- 数据脱敏不做，因为当前数仓没有真实敏感字段

这类“暂不做”的记录，和“已完成”的记录一样重要。

## 16. 优先级管理必须显式化

协作中最大的浪费之一，是每次出现新想法就立刻开做。

更健康的流程是：

1. 新建议出现
2. 先写入 plan backlog
3. 检查当前最高优先级未完成项
4. 再决定是否立即执行

这条对人和 AI 都适用。

原因：

- 人容易被新想法吸引
- AI 更容易“看到能改就顺手改”
- 真正让项目失控的，往往不是想法太少，而是优先级失守

一个好习惯是：

`每次接受新的优化建议前，先问：它是不是当前最高优先级未完成项？`

如果不是，就先排期，不直接动手。

## 17. 长 session 与新 session 都要有上下文锚点

人机协作里一个很现实的问题是：

- 长 session 会逐渐遗忘早期决策
- 新 session 会丢失当前状态

所以不能假设 AI 永远“知道全部历史”。

### 17.1 长 session 的问题

- 早期约束被稀释
- 计划优先级被遗忘
- AI 容易顺手扩散改动

### 17.2 新 session 的问题

- 要重新讲项目状态
- 要重新讲哪些做过、哪些不做
- 要重新讲当前主目标

### 17.3 必须保留的上下文锚点

无论长 session 还是新 session，以下内容都应该容易找到：

1. 当前项目边界
2. 当前计划与优先级
3. 当前已完成项
4. 当前明确不做项
5. 当前主链路如何验证
6. 当前最重要的回归入口

### 17.4 推荐的上下文载体

至少维护好：

- `PROJECT_CONTEXT.md`
- `FINANCE_OPTIMIZATION_PLAN.md` 或等价计划文档
- `Optimize/history/` 下的阶段记录

### 17.5 新 session 的入场摘要应该怎么写

每轮结束后，最好能留下一个“下个 session 直接接手”的短摘要。

推荐包含：

1. 当前项目是什么
2. 当前最高优先级未完成项是什么
3. 最近刚完成了什么
4. 当前最重要的验证命令是什么
5. 有哪些明确不要做的事

### 17.6 一个很实用的动作

在开始新一轮执行前，让 AI 先用简短文字复述：

- 本轮唯一主目标
- 本轮不做什么
- 当前系统边界

如果复述不对，不要马上动手，先把上下文重新对齐。

## 18. 改动要尽量原子化，避免一轮里同时改多个层级

协作优化里很怕一种情况：

- 表面上在修一个 bug
- 实际上同时改了 prompt、graph、shared helper、脚本入口

这样做不是绝对不行，但会显著降低可归因性。

### 18.1 什么叫原子化改动

一轮改动尽量聚焦于一个明确目的，并且修改面尽量收敛。

例如：

- 只改一个 guard 的判断逻辑
- 只改一个 prompt 示例集
- 只改一个状态流回环
- 只改一个脚本入口兼容问题

### 18.2 为什么原子化重要

- 更容易验证
- 更容易回滚
- 更容易解释
- 更容易判断这次修复到底有没有效

### 18.3 什么时候允许跨层联动

以下情况可以接受跨层：

- 一个新机制确实需要主链路和验证工具同步更新
- 一个修复天然包含主逻辑和回归脚本兼容性调整
- 一层改动会直接破坏另一层验证入口

但即使这样，也要在 history 里明确写清：

- 哪部分是主改动
- 哪部分是伴随兼容性改动

## 19. 什么时候该结构优化，什么时候先别动

结构优化很重要，但时机更重要。

### 19.1 太早重构的问题

- 行为还没稳定
- 你还不知道真正的职责边界
- 一重构，调试线索全断

### 19.2 一直不重构的问题

- 模块越来越长
- 状态字段越来越多
- 每改一次就更难读
- 未来每一轮优化成本都会升高

### 19.3 适合做结构减法的时机

当满足以下条件时，通常可以开始低风险整理：

1. 主链路基本稳定
2. 核心 badcase 已经有定位方向
3. 某模块已经连续被改了多次
4. 你已经能清楚说出它“什么该留主流程，什么能下沉 support”

这时适合做：

- support 拆分
- helper 下沉
- state 字段收口
- 测试支撑层共用

不适合做：

- 在没有行为基线时大拆主图
- 为了“优雅”强行合并不同语义对象

## 20. 每次优化结束后都要做最小复盘

真正能被你带走的方法，不是某次代码，而是复盘后的抽象。

建议每轮结束都回答下面五个问题：

1. 本轮主要问题是什么
2. 根因在哪一层
3. 为什么选这个解法
4. 怎么验证它有效
5. 还有什么没解决

这五个问题会把一次“修 bug”变成一次“能力积累”。

## 21. 项目之外最值得带走的通用能力

如果把所有原则再抽象一层，协作优化真正训练的是下面这些能力：

### 21.1 问题分层能力

不是看到错就改，而是先判断它属于哪一层。

### 21.2 优先级守恒能力

不是想到什么都做，而是守住当前最重要未完成项。

### 21.3 边界意识

不是一味追求覆盖，而是先明确系统该回答什么、不该回答什么。

### 21.4 验证意识

不是“我觉得修好了”，而是“我用什么证据证明它修好了”。

### 21.5 取舍记录能力

不是只记录改动，而是把为什么做、为什么不做留下来。

### 21.6 从单次修复中抽取长期方法的能力

这其实是协作优化最值钱的部分。

## 22. 可直接复用的工作流程模板

以后做新的协作优化时，可以直接套下面这个简化流程。

### 22.1 开始前

1. 明确本轮唯一主目标
2. 明确本轮不做什么
3. 检查当前最高优先级未完成项
4. 让协作者或 AI 先复述本轮目标和边界

### 22.2 动手前

1. 判断问题属于哪一层
2. 先看现有日志、评测、history
3. 决定是补规则、改 prompt、改代码，还是补数据

### 22.3 实施中

1. 小步改动
2. 每次改动后立即做对应层验证
3. 避免同一轮无边界扩散
4. 若出现顺手扩散，及时记录并判断是否补验证

### 22.4 收尾时

1. 跑 smoke
2. 跑相关回归
3. 检查是否目标漂移
4. 更新 plan 状态
5. 写 history
6. 如有必要，更新上下文文档和下个 session 摘要

### 22.5 新问题出现时

1. 先归档进 badcase 或 backlog
2. 再判断是否现在处理
3. 不要因为“它看起来也重要”就打断当前主线

## 23. 最后的总原则

项目协作优化，不是不断修错，而是不断把下面这些东西结构化：

- 问题
- 边界
- 优先级
- 验证
- 取舍
- 上下文

真正成熟的优化节奏不是：

- 看到一个错
- 补一个洞
- 再看到一个错
- 再补一个洞

而是：

1. 识别问题层级
2. 选择合适修法
3. 用评测和日志验证
4. 把结果沉淀成规则、工具、回归集和文档

当这个循环建立起来后，项目才会越来越顺，团队和 AI 的协作也才会越来越像“系统推进”，而不是“临场救火”。
