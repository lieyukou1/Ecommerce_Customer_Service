# 主体搭建阶段方法论

## 1. 这一阶段到底解决什么

在既有骨架项目里，主体搭建阶段不是 greenfield 开发。

它更像是在做这些事：

- 沿既有主线推进
- 把当前轮需要的能力补进主链
- 保持老师结构仍然可辨认
- 在不大改框架的前提下，让系统更完整、更稳

这份文档只讲这一阶段特有的方法。

像“唯一主目标”“优先级显式化”“原子化改动”“session 锚点”这类跨阶段原则，统一看母文档：

- [AI协作项目全周期方法论](./AI协作项目全周期方法论.md)

## 2. 进入 02 后的第一轮实操顺序

如果 01 阶段已经结束，开始真正补能力，第一轮建议按这个顺序推进：

1. 先确认这轮只动哪条主链、哪几个文件
2. 冻结暂时不动的图路由、schema/meta、返回协议
3. 先补主链缺口，不先做结构美化
4. 主链一跑通，立刻补最小日志和状态标记
5. 涉及高风险节点增强时，优先用影子节点或可回退挂载方式
6. 给新增模块定义成功流转、失败流转和用户可见结果
7. 在这一轮结束前，补上最小验证入口

如果这七步里有一步还没想清楚，就不要急着继续往下堆改动。

## 3. 先保老师结构可辨认，再做能力增强

既有骨架项目里，一个很实际的判断顺序通常是：

1. 先看能不能沿原主线补
2. 再看是否需要扩节点或 support
3. 最后才考虑明显偏离原结构

不建议轻易做的事：

- 轻易换框架
- 轻易重写大图
- 轻易重构全部节点
- 轻易把教学结构改到老师自己都认不出来

因为这类项目的重要价值之一，就是你能沿骨架推进，而不是另起一套。

## 4. 主体搭建优先补纵向主链缺口

主体搭建时，不要先做横向美化。

更稳的顺序通常是：

1. 先补主链缺口
2. 再补最小错误流
3. 再补最小日志
4. 最后再考虑 support 拆分或结构整理

例如：

- 先让 SQL 生成后的 validate 回环成立，再考虑 prompt 组织得更优雅
- 先让拒答和澄清闭环成立，再考虑把 guard 抽成更漂亮的工具层

## 5. 如何控制改动边界

主体搭建阶段很容易边做边散，所以边界必须明确。

### 5.1 不轻易换框架

除非当前框架已经直接阻断目标，否则优先在现有框架内补能力。

### 5.2 不轻易重写大图

如果当前图还能表达主链，就优先改边和节点，不先整体推倒。

### 5.3 不轻易重构全部节点

先改当前轮真正相关的节点，不要因为“顺眼”就全局收拾。

### 5.4 不把搭建期做成结构美化期

搭建阶段允许某些地方不够优雅，只要：

- 行为正确
- 可验证
- 不明显污染主链可读性

## 6. 如何设置阶段冻结点

主体搭建阶段很需要冻结点，不然所有层一起动，最后很难判断问题来源。

建议至少考虑冻结以下内容：

### 6.1 当前图路由

在一轮主目标完成前，不要同时大改多个分支路由。

### 6.2 当前 schema / meta 配置

如果这一轮不是在改数仓或元知识，就尽量不要一边改图、一边改表结构定义。

### 6.3 当前返回协议

如果这一轮不是在改前端交互，就尽量别频繁改 SSE 事件形态。

### 6.4 当前日志结构

如果已经开始记录节点、状态和错误，就尽量把核心字段结构先固定下来。

冻结点的意义不是“以后不动”，而是：

- 让一轮增强有稳定参照物

## 7. 日志即契约：尽早建立最小可观测性

主体搭建阶段不需要重型监控，但一定要尽早补最小可观测性，而且最好把日志字段当成契约来维护。

至少要能知道：

- request id 是谁
- 当前 query 是什么
- 当前走到哪个节点或阶段
- 失败阶段是什么
- 最终 SQL 是什么
- 最终 status 是什么

如果后面计划要做 badcase 分析工具、状态检测工具或审计回放，建议尽早固定一个最小日志结构。

例如，哪怕很轻，也最好稳定包含：

- `request_id`
- `query`
- `node_history` 或等价阶段轨迹
- `final_sql`
- `status`
- `error` 或 `exception_stage`

这样 03 阶段的分析工具才能直接接上，而不是每次为了配日志格式去返工。

## 8. 涉及核心节点增强时，优先考虑影子节点

既有骨架项目里，最痛的情况之一是：

- 原节点还能跑
- 你为了增强它，直接在原函数里大改
- 最后既搞挂了主链，也很难回滚

更稳的做法是优先考虑影子节点机制。

做法可以很简单：

- 复制一个 `node_xxx_v2` 或 `shadow_xxx`
- 在主图中局部挂载验证
- 验证稳定后再决定是否替换原节点

适合用影子节点的场景：

- 核心路由节点
- validate / correct 这类影响主链行为的节点
- 引入多轮状态、重写、澄清等容易改变上下文流向的节点

如果只是很小的局部 bug 修复，就不需要机械地上影子节点。

## 9. 如何判断一个增强模块是否该加

在既有骨架项目里，加一个模块之前至少回答下面几个问题：

1. 它接在哪
2. 它拦什么问题
3. 成功时怎么流转
4. 失败时怎么流转
5. 用户会看到什么
6. 日志或审计里记什么
7. 怎么验证它真的有用

如果这些问题答不清，就先不要急着加。

## 10. 典型增强模块该怎么想

### 10.1 Guard 类增强

先问：

- 它拦的是错领域、错实体，还是危险输出
- 它该放在主链前、中、后哪一段

### 10.2 Validate / Correct 类增强

先问：

- 它是单次执行前校验，还是失败修正回环
- 修正后是否必须再验证

### 10.3 审计与日志类增强

先问：

- 记录哪些字段才足够定位问题
- 哪些字段只是“看起来很多”，其实没有决策价值

### 10.4 多轮与状态类增强

先问：

- 是真的需要持久化，还是 service 层改写就够
- 保存什么摘要就够，不要一上来全量记忆

## 11. 这一阶段最常见的协作坑

### 11.1 一边搭一边顺手重构

结果是主链增强和结构整理混在一起。

### 11.2 还没补最小日志就开始扩功能

结果是主链一出问题根本没法定位。

### 11.3 一个模块加上去，却没定义验证方法

结果是“感觉更强了”，但没有证据。

### 11.4 直接在核心节点上大改

结果是增强和回滚都很痛。

## 12. 什么时候可以从 02 进入 03

只有在下面这些东西已经成立后，才建议进入优化与稳定化：

1. 当前主链已经端到端跑通
2. 当前轮新增能力已经接入主链
3. 最小可观测性已经成立
4. 日志结构至少在当前轮内是稳定的
5. 新增增强模块已经有最小验证入口

如果这些还没成立，就不要急着跳进 03，因为 badcase、题库和诊断工具会失去稳定观察对象。

## 13. 这阶段结束时至少要留下什么

建议至少留下：

1. 当前主链已经补了哪些能力
2. 当前冻结点是什么
3. 当前最小日志和状态流是否成立
4. 新增增强模块的验证入口
5. 当前仍然不打算动的结构部分

## 14. 一句话总结

在既有骨架项目里，主体搭建阶段最重要的不是“我搭了多少新模块”，而是：

**我能不能沿着老师主线，把真正需要的工程能力补进去，同时不把系统改散，还给后续优化阶段留下稳定的观察接口。**
