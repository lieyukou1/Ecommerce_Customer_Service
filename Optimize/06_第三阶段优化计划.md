## 阶段定位

第三阶段当前主线不是继续“打更多 case”，而是把状态流转、上下文切换、退出机制和回归门禁收成一套能持续演进的架构。

## 当前唯一 P0

- 方案 B：显式会话状态机重构
- 推进方式：渐进接管

## 当前执行原则

1. 架构清晰、可解释、模块短小，高于 badcase 命中率
2. 不把关键词硬命中当主判定路径
3. 主判定优先顺序固定为：
   - LLM 结构化输出
   - 显式上下文决策协议
   - 状态机统一接管
4. 回归测试是阶段门禁，不是当前实现方向本身
5. 单文件原则上不超过 `450` 行，单函数原则上不超过 `80` 行

## 当前正式结论

1. P0 的结构收敛主线已经实质推进完成一大步
2. 单点回归不再作为当前架构收尾的阻塞项
3. 当前目标改为尽快完成最后一轮边界收官，之后转入面试材料准备
4. `processor.py / executor.py` 继续不纳入当前 P0 主线，除非出现明确提级证据

## 架构收尾四项

1. 清掉旧层残影
   - 删除 `engine/runtime`、`engine/turns` 的历史残留和 `__pycache__`
   - 让代码树和“6 个一级概念”一致
2. 再收一轮 `state_decision`
   - 让高层只理解“状态决策入口”
   - 不再显式背负过多中间协议对象
3. 再压一轮 `track_execution`
   - 上层只面向更少的执行入口
   - 文本/对象/任务局部上下文不继续外溢
4. 降低 `focus/` 的架构存在感
   - 高层只认 `FocusedObjectResolver`
   - `text_switch`、`slot_handoff` 继续下沉为内部策略

## 下一步顺序

1. 先完成架构收尾四项
2. 收尾完成后出最终架构图与收口报告
3. 然后转入面试材料准备

## 当前不做

- 不追着单个失败 turn 写 `if`
- 不为了短期命中率继续把 `engine` 堆胖
- 不默认下潜到 `task/command/processor.py` 或 `task/flow/executor.py`
- 不把单个 badcase 当成这一轮成功与否的唯一标准
