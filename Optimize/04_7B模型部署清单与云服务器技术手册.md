# 7B 模型部署清单与云服务器技术手册

## 1. 目标与结论

这份手册面向当前物业智能管家项目，目标是把现有外部大模型调用切到一台你自己租的 GPU 云服务器上，先完成 **7B 级模型推理部署**，暂时 **不做微调**。

结合当前项目代码，最顺滑的落地路线是：

- 模型：`Qwen/Qwen2.5-7B-Instruct`
- 服务框架：`vLLM`
- 接口协议：`OpenAI-compatible API`
- 部署方式：`Docker + NVIDIA Container Toolkit`
- 接入方式：只改 `customer-service-backend/.env`，后端代码原则上不用改

这样选的原因很直接：

- 当前项目已经是 OpenAI 兼容接法
- `vLLM` 原生提供 OpenAI 兼容接口
- `Qwen2.5-7B-Instruct` 是严格 7B 档位，适合作为第一版自部署模型
- 先把推理打通，再决定要不要做 LoRA / QLoRA，会稳很多

---

## 2. 当前项目接入点

### 2.1 LLM 初始化方式

当前项目在 `customer-service-backend/atguigu/infrastructure/llm.py` 中这样初始化模型：

```python
llm: BaseChatModel = init_chat_model(
    model=settings.llm_model,
    model_provider="openai",
    api_key=settings.llm_api_key,
    base_url=settings.llm_base_url,
    temperature=0
)
```

这意味着只要你的云端模型服务提供：

- `/v1/models`
- `/v1/chat/completions`

这样的 OpenAI 兼容接口，本项目就能直接接入。

### 2.2 当前环境变量

当前 `customer-service-backend/.env` 中与模型有关的是：

```env
LLM_MODEL=deepseek-v4-flash
LLM_BASE_URL=https://api.deepseek.com
LLM_API_KEY=***
```

后续切 7B 之后，改成：

```env
LLM_MODEL=Qwen/Qwen2.5-7B-Instruct
LLM_BASE_URL=http://<你的云服务器IP>:8000/v1
LLM_API_KEY=EMPTY
```

### 2.3 当前哪些模块会用到 LLM

这几个模块都会直接吃到新模型能力：

- `customer-service-backend/atguigu/plan/planner.py`
- `customer-service-backend/atguigu/clarify/responder.py`
- `customer-service-backend/atguigu/knowledge/responder.py`
- `customer-service-backend/atguigu/chitchat/responder.py`

所以 7B 模型上线后，你最先感受到变化的地方会是：

- 意图规划质量
- 澄清命中质量
- 知识问答措辞
- 闲聊自然度

## 3. 资源要求

### 3.1 推荐分档

| 档位 | 推荐用途 | GPU | CPU | 内存 | 系统盘 | 数据盘 | 结论 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 最低可跑 | 联调、验证链路 | 24GB 显存级 | 8 vCPU | 32GB | 50GB | 100GB SSD | 能跑，但上下文和并发要保守 |
| 开发推荐 | 日常调试、多人轻测 | 40GB 显存级 | 16 vCPU | 64GB | 100GB | 200GB SSD | 更稳，OOM 风险小很多 |
| 稳定演示 | 较长上下文、较稳延迟 | 80GB 显存级 | 16-32 vCPU | 64-128GB | 100GB | 300GB SSD | 体验最好，但成本明显上升 |

### 3.2 为什么这样分

- `Qwen2.5-7B-Instruct` 模型卡给出的参数规模是 **7.61B**，支持 **131,072** tokens 上下文，但文档也明确说明默认生成长度为 **8192**。  
- `vLLM` 会额外占用显存做 KV Cache 和运行时管理，所以“模型能装下”不等于“服务能稳定跑起来”。
- 对 24GB 卡来说，**单卡推理可行**，但建议你把 `--max-model-len` 控在 **8192** 左右，把 `--gpu-memory-utilization` 控在 **0.85~0.9**。
- 如果你想少折腾 OOM、少调参数，40GB 档会轻松很多。

上面这部分里，“24GB 可以作为单卡下限”是我结合模型规模、Qwen 文档中的上下文能力、以及 vLLM 的显存预留方式做的工程推断，不是官方逐字写死的最低门槛。

### 3.3 云服务器建议

优先选择：

- Ubuntu `22.04 LTS`
- 已支持 NVIDIA GPU 的实例
- 最好直接选带 GPU 驱动镜像的公共镜像，少踩一层驱动坑

优先级建议：

1. `L4 24GB` / `A10 24GB`：适合第一版
2. `A100 40GB`：更稳
3. `A100 80GB` / `H100`：除非你后面要上更大模型，不然当前项目有点奢侈

## 4. 推荐部署栈

### 4.1 最推荐的首版部署栈

- OS：Ubuntu 22.04 LTS
- 容器：Docker Engine
- GPU 容器支持：NVIDIA Container Toolkit
- 推理服务：vLLM OpenAI-compatible server
- 模型：Qwen2.5-7B-Instruct
- 端口：`8000`
- 反向代理：首版可不加；若要公网长期使用，再补 `Nginx/Caddy + HTTPS`

### 4.2 为什么不是先上微调

现在你的主要目标不是“把模型调到最强”，而是先建立一条稳定的：

`项目后端 -> 提示词 -> 自部署模型 -> 结构化输出 -> 状态流转`

闭环。

如果这个闭环还没稳定，就直接上微调，后面你会分不清问题到底出在：

- 提示词
- 推理模型
- 状态管理
- 流程编排
- 训练数据
- 微调参数

这会很痛苦。

## 5. 部署前检查清单

正式动手前，先确认这些：

- 已租好 GPU 云服务器
- 可 SSH 登录
- 已知服务器公网 IP 或内网 IP
- 已放行 `22` 端口
- 若本地后端直连模型服务，已放行 `8000` 端口，或准备只走内网
- 磁盘空间至少 100GB
- 明确你是“只自己测试”还是“要公网暴露模型接口”

如果只是你自己调试，建议：

- 模型端口只对白名单 IP 放行
- 不要一上来裸奔全网开放 `8000`

## 6. 云服务器技术手册

## 6.1 Step 0：选实例

第一版推荐你选下面两类之一：

- 性价比版：`24GB` 显存级 GPU
- 稳定版：`40GB` 显存级 GPU

如果你只是为了把当前物业项目跑通，我更建议：

- 先租 `24GB` 档
- 先把 `Qwen2.5-7B-Instruct + vLLM` 跑通
- 真遇到显存、延迟、并发瓶颈，再升配

## 6.2 Step 1：登录并更新系统

```bash
ssh ubuntu@<server-ip>
sudo apt update
sudo apt upgrade -y
sudo apt install -y curl git ca-certificates gnupg lsb-release
```

## 6.3 Step 2：确认 GPU 驱动

先看机器有没有正确识别 GPU：

```bash
nvidia-smi
```

你应该能看到：

- GPU 型号
- 驱动版本
- CUDA Version

如果这一步就失败，不要继续往下装 vLLM，先把驱动问题解决。

最省心的方案是：**直接换成云厂商带 NVIDIA 驱动的官方镜像**。  
自己从零配驱动当然也行，但第一版没有必要给自己上难度。

## 6.4 Step 3：安装 Docker

按 Docker 官方文档走：

```bash
sudo apt-get update
sudo apt-get install -y ca-certificates curl
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo \"$VERSION_CODENAME\") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
```

验证：

```bash
sudo docker --version
```

## 6.5 Step 4：安装 NVIDIA Container Toolkit

按 NVIDIA 官方文档安装容器工具链：

```bash
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg

curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list

sudo apt-get update
sudo apt-get install -y nvidia-container-toolkit
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker
```

验证 GPU 是否能被容器读到：

```bash
sudo docker run --rm --gpus all nvidia/cuda:12.4.1-base-ubuntu22.04 nvidia-smi
```

如果这一步能看到 GPU，说明 Docker + NVIDIA runtime 已经通了。

## 6.6 Step 5：准备模型缓存目录

```bash
sudo mkdir -p /data/hf-cache
sudo chown -R $USER:$USER /data/hf-cache
```

如果后面你要频繁换模型，这个缓存目录会很有用。

## 6.7 Step 6：启动 vLLM 服务

### 方案 A：24GB 显存级，保守启动参数

```bash
sudo docker run -d \
  --name qwen7b-vllm \
  --restart unless-stopped \
  --gpus all \
  --ipc=host \
  -p 8000:8000 \
  -v /data/hf-cache:/root/.cache/huggingface \
  vllm/vllm-openai:latest \
  --model Qwen/Qwen2.5-7B-Instruct \
  --served-model-name Qwen/Qwen2.5-7B-Instruct \
  --host 0.0.0.0 \
  --port 8000 \
  --max-model-len 8192 \
  --gpu-memory-utilization 0.85
```

### 方案 B：40GB 及以上，稍微放宽

```bash
sudo docker run -d \
  --name qwen7b-vllm \
  --restart unless-stopped \
  --gpus all \
  --ipc=host \
  -p 8000:8000 \
  -v /data/hf-cache:/root/.cache/huggingface \
  vllm/vllm-openai:latest \
  --model Qwen/Qwen2.5-7B-Instruct \
  --served-model-name Qwen/Qwen2.5-7B-Instruct \
  --host 0.0.0.0 \
  --port 8000 \
  --max-model-len 16384 \
  --gpu-memory-utilization 0.9
```

### 启动后看日志

```bash
sudo docker logs -f qwen7b-vllm
```

看到类似 “Uvicorn running” 或服务已监听 `8000`，就说明起来了。

## 6.8 Step 7：验证模型服务

先看模型列表：

```bash
curl http://127.0.0.1:8000/v1/models
```

再发一个最小对话请求：

```bash
curl http://127.0.0.1:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer EMPTY" \
  -d '{
    "model": "Qwen/Qwen2.5-7B-Instruct",
    "messages": [
      {"role": "system", "content": "你是一个简洁的中文助手。"},
      {"role": "user", "content": "你好，简单介绍一下你自己。"}
    ],
    "temperature": 0
  }'
```

这一步成功之后，再接你本地项目。

## 6.9 Step 8：让当前项目接入你的 7B 服务

修改 `customer-service-backend/.env`：

```env
LLM_MODEL=Qwen/Qwen2.5-7B-Instruct
LLM_BASE_URL=http://<server-ip>:8000/v1
LLM_API_KEY=EMPTY
```

注意：

- 这里的 `LLM_API_KEY` 可以先用占位值 `EMPTY`
- 当前你的后端代码会照常传这个字段，但 vLLM 本身不靠它做鉴权
- 真要做鉴权，建议放在 Nginx、Caddy、Cloud Firewall 或内网隔离层

## 6.10 Step 9：重启本地后端并冒烟测试

在 `customer-service-backend` 目录下重新启动后端，然后测最小链路：

1. 一条普通文本消息
2. 一条会触发澄清的消息
3. 一条对象消息

重点看：

- 是否能正常出结果
- 返回速度是否可接受
- 是否出现 JSON 结构不稳定
- 是否因为模型变了导致 planner 更容易跑偏

---

## 7. 建议的项目接入顺序

不要一上来就让前端、后端、数据库、对象消息、任务流一起压上去。建议这样接：

### 阶段 1：只测模型服务

- `curl /v1/models`
- `curl /v1/chat/completions`

### 阶段 2：只测后端 planner / responder

- 本地直接打 `/api/chat`
- 先看文本消息

### 阶段 3：测状态链

- 连续两轮对话
- 看 `dialogue_states.state_json` 是否合理变化

### 阶段 4：最后再测前端联调

- 页面文本消息
- 工单对象消息
- 服务项目对象消息

这样出问题时，你会更容易定位到底是：

- 云端模型服务问题
- 本地后端问题
- 前端代理问题
- 状态持久化问题

## 8. 验收清单

至少要过下面这些项，才能算“7B 自部署接入成功”：

- `nvidia-smi` 正常
- Docker 可用
- NVIDIA 容器运行时可用
- `vLLM` 容器正常启动
- `/v1/models` 可访问
- `/v1/chat/completions` 可返回正常结果
- 本地后端改 `.env` 后可正常调用
- `/api/chat` 不报 5xx
- 文本消息至少能完成：
  - 闲聊
  - 澄清
  - 一个简单任务流入口

## 9. 常见风险与处理

### 9.1 OOM 或容器起不来

处理顺序：

1. 把 `--max-model-len` 从 `16384` 降到 `8192`
2. 把 `--gpu-memory-utilization` 从 `0.9` 降到 `0.85`
3. 确认没有别的进程在吃显存
4. 不行就升到 40GB 档

### 9.2 首 token 很慢

这是正常现象之一，尤其是：

- 首次加载模型
- 首次跑某类 prompt
- 上下文较长

先看是不是能稳定返回，再谈体验优化。

### 9.3 输出比在线大模型更笨

这也正常。7B 自部署的意义主要是：

- 成本更可控
- 可掌控
- 可积累你自己的数据和调优闭环

不是说它一上来就一定比商用大模型聪明。

### 9.4 planner 更容易跑偏

这个是你项目里最可能实际遇到的问题。原因通常不是“部署错了”，而是：

- 提示词约束不够硬
- few-shot 样例不稳
- 输出格式约束不够强
- 小模型对复杂路由没那么稳

所以第一版接入后，不要急着怀疑整套部署，先做 prompt 和结果校验。

## 10. 回退方案

切模型前，先备份一份 `.env`：

```bash
cp .env .env.deepseek.backup
```

如果自部署 7B 效果不理想，直接把：

- `LLM_MODEL`
- `LLM_BASE_URL`
- `LLM_API_KEY`

恢复成原来在线模型配置即可。

这一步很重要，因为它能保证你在试验新模型时始终有回头路。

---

## 11. 下一步建议

建议按这个节奏推进：

1. 先部署 `Qwen2.5-7B-Instruct + vLLM`
2. 打通本地 `/api/chat`
3. 观察 planner / clarify / knowledge / chitchat 的实际效果
4. 把失败 case 积累成数据
5. 再决定是否做 LoRA / QLoRA

如果后面真的要做微调，这个项目最值得积累的数据不是“泛聊天数据”，而是这些和你业务强相关的样本：

- `turn_plan` 正负例
- 澄清命中与误判样本
- 对象消息补槽样本
- 物业知识问答样本
- 任务流进入 / 退出 / 中断样本
