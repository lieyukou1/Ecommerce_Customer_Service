# AI 协作项目全周期方法论

## 1. 这套方法论解决什么问题

这套文档默认服务于一种很具体的项目语境：

- 已经有老师代码、教学骨架或既有主线
- 不是从 0 到 1 独立架构
- 主要工作是理解、继承、裁剪、增强、收口和优化

所以它不假装你正在独立设计一套全新的系统。

它真正要解决的是：

- 面对既有骨架时，先看懂什么，后改什么
- 怎样避免一上来就把项目做成另起炉灶
- 怎样让人和 AI 的协作有节奏、有边界、可验证
- 怎样把一次次微调沉淀成稳定的方法，而不是只留下零散改动

## 3. 阶段划分与退出条件

### 3.1 阶段 01：问题定义与架构设计

适用场景：

- 刚接手一个既有骨架项目
- 还没想清楚该继承什么、该增强什么
- 还没写清本轮目标和不做清单

本阶段结束，通常至少要满足：

- 已写出本轮主目标
- 已写出继承清单和不做清单
- 已定义最小增强闭环
- 已明确本轮验收方式

对应文档：

- [01_问题定义与架构设计阶段方法论](./01_问题定义与架构设计阶段方法论.md)

### 3.2 阶段 02：主体搭建

适用场景：

- 目标已经清楚
- 正在沿既有主链补节点、补状态、补日志、补守卫、补接口
- 需要控制改动边界，避免边搭边散

本阶段结束，通常至少要满足：

- 当前主链已经端到端跑通
- 最小可观测性成立
- 新增增强模块有明确验证入口
- 当前冻结点和暂不改动部分是清楚的

对应文档：

- [02_主体搭建阶段方法论](./02_主体搭建阶段方法论.md)

### 3.3 阶段 03：优化与稳定化

适用场景：

- 系统已经能跑，但 badcase、回归、优先级、审计、日志、题库、收口开始成为主要问题
- 项目不再缺“第一条链”，而是缺“稳定和可控”

本阶段收口，通常至少要满足：

- 当前高优先级 badcase 已被归因和分层
- 已有最小 smoke 与相关回归入口
- 历史决策、变更和遗留问题可追溯
- 当前没有更高优先级的未完成项被低优先级想法打断

对应文档：

- [03_优化与稳定化阶段方法论](./03_优化与稳定化阶段方法论.md)

## 4. 共通原则

这些原则跨三个阶段都成立，因此只在母文档保留定义。

### 4.1 唯一主目标

每一轮协作先写清：

- 这轮只解决什么
- 这轮明确不解决什么

### 4.2 优先级显式化

新想法出现时，不直接开做。先判断：

- 它是不是当前最高优先级未完成项
- 它属于主链必需、稳定性增强，还是展示/未来项

### 4.3 Decision Log 比 Change Log 更重要

除了记录改了什么，还要记录：

- 为什么现在改
- 为什么现在不改别的
- 未来在什么条件下再做

### 4.4 目标漂移检查

每轮结束都回头看：

1. 实际改动是否超出边界
2. 超出的部分是否影响其他链路
3. 是否需要补验证、补文档或补回归

### 4.5 原子化改动

一轮改动尽量只解决一个主问题。

尤其在既有骨架项目里，不要把：

- 主链增强
- 结构整理
- prompt 重写
- 数据调整

混成一轮一起做。

### 4.6 最小可观测性意识

即使在前期，也要尽早保证至少能看到：

- request / query
- 当前阶段或节点
- 失败位置
- 最终结果状态

### 4.7 Session 交接与上下文锚点

长 session 和新 session 都会丢信息。

因此要稳定维护这些锚点：

- 当前主目标
- 当前边界
- 当前计划状态
- 最近一次验证结果
- 最近一次未解决遗留

推荐载体：

- `PROJECT_CONTEXT.md`
- `IMPLEMENTATION_PLAN.md` 或等价计划文档
- `Optimize/history/`

## 5. 一条很实用的使用规则

如果你不确定当前在哪个阶段，就先问自己：

1. 我现在是在判断“该做什么”吗
2. 还是已经在补主链能力
3. 还是已经进入 badcase、回归和收口

大多数既有骨架项目，真正最常反复使用的，仍然是：

- [03_优化与稳定化阶段方法论](./03_优化与稳定化阶段方法论.md)

因为前两个阶段决定方向，第三个阶段决定质量。

## 7. 一句话总结

这套方法论的核心，不是教你假装自己在独立开发一套系统，而是教你：

**在既有骨架项目里，怎样看懂原结构、守住主线、分阶段推进，并把一次次协作沉淀成可复用的方法。**
