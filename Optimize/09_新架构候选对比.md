## 候选目标

新架构候选统一围绕四个目标：

1. 更灵活的状态流转
2. 更精准的意图识别
3. 更少的规则与词汇命中代码
4. 更清晰的分离与解耦

## 方案 A：上下文仲裁器 + 保留现有三轨

- 价值：适合快速止血，把分散的上下文判断收成一层。
- 优势：改动相对可控，迁移成本低。
- 不足：仍然背着旧 `TurnPlan` 和旧三轨包袱。
- 结论：适合过渡，不适合作为长期主架构。

## 方案 B：显式会话状态机重构

- 核心：把当前隐式靠字段推进的对话主链，改为显式 `Conversation State Machine`。
- 顶层状态建议至少包含：
  `Idle`、`FocusedKnowledge`、`ActiveTask`、`Clarifying`、`Chitchat`、`Transitioning`
- 统一执行顺序：
  `Intent Understanding -> Context Decision -> State Transition -> Track Execution`
- 优势：
  - 状态迁移清晰、可测试、可回放
  - `focused_object / active_task / paused_tasks` 退回为上下文附属信息
  - 最适合与长对话回归集配套
- 不足：
  - 改造量最大
  - 需要重划内部边界
- 结论：这是当前正式选中的唯一 P0 主方案。

## 方案 C：LLM 主导的动作编排器

- 核心：让模型输出更细粒度动作，而不只输出 `task / knowledge / chitchat`。
- 优势：规则代码更少，长期扩展上限更高。
- 不足：依赖模型稳定性，对 prompt 和回归基线要求很高。
- 结论：保留为远期方向，不作为当前主重构方案。

## 最终选型

- 当前正式选择：方案 B
- 吸收的辅助思想：保留方案 A 的统一上下文仲裁入口
- 暂不直接采用：方案 C
