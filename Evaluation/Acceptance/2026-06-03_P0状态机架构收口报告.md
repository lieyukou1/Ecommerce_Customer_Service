## 结论

可以把第三阶段 P0 的后端架构重构视为**基本收口完成**。

这次最重要的成果不是把所有 badcase 打平，而是把对话引擎整理成了可解释、可观测、可继续演进的结构。

## 本轮范围

本轮只看后端内部架构收口，重点在：
- `engine`
- `plan`
- `DialogueState` 相关运行态

本轮不包含：
- 前端交互改版
- 中台接口协议改造
- 执行层大重写
- 多意图体系重做

## 已完成的架构收口

### 1. 主链固定

当前主链已经稳定为：

`DialogueEngine -> Planner -> Protocol Gate -> State Decision -> Track Execution -> DialogueState`

### 2. 旧中间层退出

旧的：
- `engine/runtime`
- `engine/turns`

已经退出正式结构。

### 3. 入口职责清晰

`DialogueEngine` 现在只承担：
- session 生命周期
- turn 开始/提交
- 文本/对象分流

### 4. 协议闸门明确

`TurnProtocolGate` 已成为模型输出进入执行主链前的统一闸门。

### 5. 状态决策显式化

`StateDecisionEngine + DialogueStateMachine` 已把这些变化收成显式过程：
- route decision
- state transition
- task outcome transition

### 6. 运行态观测面稳定

当前排查最关键的状态字段已经比较稳定：
- `focused_object`
- `active_task`
- `paused_tasks`
- `active_system_task`
- `conversation_state`
- `last_route`
- `last_transition`
- `last_task_outcome`

### 7. 执行层 wiring 回收

`track_execution/` 已经承担起自己的组装工作，`DialogueEngine` 不再亲手拼执行层细件。

## 是否需要改前端和中台

**以这轮架构重构本身来说，不需要。**

更准确地说：

1. 前端不需要为这轮重构配合改代码
   - 本轮没有改变前端调用入口
   - 没有引入新的前端必填协议
   - 没有要求前端改消息结构

2. 中台不需要为这轮重构配合改代码
   - 本轮没有重做中台接口协议
   - 没有新增必须由中台提供的新字段
   - 没有要求改现有接口路径

3. 如果你手测时发现行为问题
   - 那属于后续联调或功能问题
   - 不能倒推出“这轮架构必须修改前端/中台”

所以当前正式口径就是：  
**这轮后端架构收口，不依赖前端和中台改动。**

## 仍然存在的边界

这些是“已知还没做”的，而不是“这轮失败了”：

1. 多意图并行还没有正式接入
2. 某些模糊追问仍可能出现语义摇摆
3. `processor.py / executor.py` 还不是这轮重构对象
4. browser smoke 与真实联调仍需要人工补确认

## 建议的人工手测重点

你自己测的时候，优先测这几类：

1. 对象进入 -> 追问 -> 退出 -> 再起新话题
2. 工单 A -> 工单 B 切换
3. knowledge -> task 的切换
4. 任务中退出后是否残留旧上下文
5. service item 和 work order 之间的切换承接

## 面试讲解口径

这轮最值得讲的不是“我修了很多 case”，而是：

1. 我把原来隐式靠下游改状态的链路，收成了显式状态决策
2. 我把模型输出和系统执行协议之间补上了 `Protocol Gate`
3. 我把执行层 wiring 从顶层入口里拿掉了
4. 我把运行态观测字段固定下来，后续排查更可解释
