## 3.1 容器网络与安全说明

本系统当前采用的是 Docker Compose 默认桥接网络。

核心结论：

- **中台容器与 MySQL 容器之间的通信，不走公网，不走外部域名**
- **中台访问 MySQL 走的是 Docker 内部网络和服务名解析**

当前容器内部的访问方式等价于：

```text
mysql:3306
```

也就是说：

- `backend` 容器不会去连公网数据库
- `backend` 容器不会去解析外部业务域名
- 它是通过 Docker Compose 给服务分配的内部服务名 `mysql`，在同一个 Docker 网络中完成访问

这和老师强调的思路是一致的：服务之间通过 Docker 内部网络互通，而不是把数据库调用暴露到公网链路上。

### 当前实现的安全边界

当前部署的 `DATABASE_URL` 为：

```text
mysql+pymysql://atguigu:Atguigu.123@mysql:3306/property_service_demo?charset=utf8mb4
```

这里的主机名 `mysql` 不是公网地址，也不是宿主机域名，而是 Docker Compose 服务名。

因此，**中台和 MySQL 之间的数据交互链路属于容器内网通信**。

### 当前仍然存在的暴露面

虽然容器间通信走的是内网，但本轮部署里 `docker-compose.yml` 仍然保留了：

- 中台端口映射：`18081:18081`
- MySQL 端口映射：`3306:3306`

这意味着：

- 中台本来就是要对外提供接口，所以暴露 `18081` 是预期设计
- MySQL 虽然实际业务调用不依赖公网，但当前仍映射到了宿主机端口

不过本轮实际部署时：

- 已明确放行 `18081/tcp`
- **没有额外放行 `3306/tcp`**

所以目前数据库虽然被 Docker 映射到了宿主机端口，但在现有防火墙策略下，并未作为对外业务入口使用。

### 更严格的收口建议

如果后续要进一步对齐“数据库仅允许容器内网访问”的安全目标，建议下一步做以下优化之一：

1. 直接删除 `docker-compose.yml` 中 MySQL 的 `3306:3306`
2. 或只绑定到宿主机回环地址，而不是对所有网卡暴露
3. 保持中台通过 `mysql` 服务名访问数据库不变

这样可以把数据库访问面从“宿主机可见但受防火墙限制”，进一步收紧到“仅 Docker 内部网络可见”。

## 3.2 与老师原项目的同步 / 异步架构对齐说明

这个点需要单独写清楚，避免把老师的两个后端混成一套理解。

- 老师的 **AI 客服后端** 是全异步链路，文档明确写了 `SQLAlchemy + aiomysql`，并强调 `async/await` 一捅到底，适用于 LLM 和跨服务 HTTP 这类高延迟 I/O。
- 老师的 **电商业务中台** 则是同步链路，文档明确写的是 `FastAPI + PyMySQL + SQLAlchemy`，理由是这类 CRUD 接口单次耗时短，同步实现更直接、更易维护。
- 当前这套 **物业管理系统中台** 对标的是老师的 **电商业务中台**，不是老师的 AI 客服后端。

因此，当前实现采用同步 SQLAlchemy 是**和老师的中台设计保持一致**的，不属于偏离。

本地代码也已经印证这一点：

- [database.py](D:/Desktop/SGG_Data/23_尚硅谷大模型项目之智能客服/前端、中台项目/ecommerce-service-backend/ecommerce-service-backend/app/database.py) 使用 `create_engine` 和同步 `SessionLocal`
- [api.py](D:/Desktop/SGG_Data/23_尚硅谷大模型项目之智能客服/前端、中台项目/ecommerce-service-backend/ecommerce-service-backend/app/api.py) 当前接口均为同步 `def` 风格

老师资料中的对照依据：

- [01_架构全景与范式重构 .md](</D:/Desktop/SGG_Data/23_尚硅谷大模型项目之智能客服/课件预览包/01_架构全景与范式重构 .md>) 第 83-95 行：明确写明电商后端使用 `PyMySQL`，并解释为什么“电商后端不需要 async”
- [01-电商小二-项目概述与整体架构.md](D:/Desktop/SGG_Data/23_尚硅谷大模型项目之智能客服/day01_电商小二项目仓库说明/2_resource/01-电商小二-项目概述与整体架构.md) 第 220-240 行：前半段写客服后端 `SQLAlchemy + aiomysql`，后半段明确写模拟电商后端使用同步 `FastAPI + PyMySQL + SQLAlchemy`

结论：

- 如果目标是**严格对齐老师的中台**，当前同步版是正确方向，不需要因为“老师客服后端是异步”就把物业中台一起改成异步。
- 如果后续我们要做的是**智能管家 / 客服后端**，那一层就应该继续按老师路线走全异步，包括数据库访问、HTTP 调用、LLM 调用全部保持异步链路。
