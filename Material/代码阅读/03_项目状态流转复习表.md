## 这份表怎么用

这份表不是要你盯住 `DialogueState` 的所有字段，而是只盯 4 类最有价值的状态：

1. 会话状态
   - `current_session_id`
   - `pending_turn`
   - `sessions`
2. 对象状态
   - `focused_object`
3. 任务状态
   - `active_task`
   - `paused_tasks`
   - `active_system_task`
4. 高层记录状态
   - `conversation_state`
   - `last_route`
   - `last_task_outcome`

你复习时只要反复问这 4 件事：

1. 这一步有没有改会话？
2. 这一步有没有改对象？
3. 这一步有没有改任务栈？
4. 这一步有没有改高层路由/结果记录？

如果答案都是“没有”，那这一层大概率只是：

- 读状态
- 做判断
- 构造中间结果

这种步骤不用死盯。

## 一、文本消息主链：状态变化时间线

| 步骤 | 函数                                                         | 重点看哪些状态                                               | 发生了什么变化                                               | 为什么重要                                 |
| ---- | :----------------------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------ |
| 1    | `DialogueService.process_message()`                          | 无                                                           | 不直接改状态                                                 | 它只是 `load -> engine -> save` 的壳       |
| 2    | `DialogueEngine._prepare_session()`                          | `current_session_id` `sessions` `active_task` `paused_tasks` `active_system_task` `focused_object` | 如果还没有 session，就新建；如果 session 超时，可能新开 session，并清空旧 runtime 状态 | 这是“会话切段”和“超时重置”的入口           |
| 3    | `DialogueState.begin_turn()`                                 | `pending_turn` `last_route` `last_task_outcome`              | 创建新的 `pending_turn`；同时清空上一轮的 `last_route` 和 `last_task_outcome` | 说明“新的一轮开始了”                       |
| 4    | `FocusedObjectResolver.try_switch_focused_object_from_text()` | `focused_object`                                             | 如果文本里明确提到了另一个唯一对象，就切换 `focused_object`  | 这一步决定后面知识/任务围绕谁展开          |
| 5    | `TurnPlanner.predict()`                                      | 读：`active_task` `paused_tasks` `focused_object` `conversation_state` `sessions` | 通常不直接改状态                                             | 这一步是“读状态做理解”，不是执行           |
| 6    | `TurnProtocolGate.process()`                                 | 读：`TurnPlan` + 当前状态                                    | 通常不直接改状态                                             | 这一步是协议收口，不推进业务               |
| 7A   | `TextTurnHandler._respond_validation_failure()`              | `conversation_state` `last_route`                            | 通过 `apply_clarify()` 把本轮标记为澄清态，更新 `last_route=clarify`，`conversation_state=CLARIFYING` | 当 plan 不能执行时，系统正式进入澄清轨     |
| 7B   | `StateDecisionEngine.build_text_context()`                   | 读：`active_task` `focused_object` `conversation_state`      | 不直接改状态                                                 | 它把 plan 解释成“这一轮最后到底该去哪条轨” |
| 8A   | `StateDecisionEngine.execute_runtime_directive()`            | `active_task` `paused_tasks` `active_system_task` `focused_object` `conversation_state` `last_route` | 如果是退出指令，会清理 runtime 状态，并把高层状态拉回空闲或重算 | 这是“退出上下文”的正式收口点               |
| 8B   | `StateDecisionEngine.apply_route_decision()`（knowledge/chitchat） | `last_route` `conversation_state`                            | 记录本轮走的是 `knowledge` 或 `chitchat`，同时更新高层会话态 | 这是文本非 task 分支最关键的状态落点       |
| 8C   | `TaskCommandExecutor.execute()`                              | `last_route` `conversation_state` `last_task_outcome`        | 进入 task 前先登记 route；task 执行后再登记 outcome          | 这是 task 轨和状态机接上的地方             |
| 9    | `DialogueState.commit_turn()`                                | `pending_turn` `sessions`                                    | 把 `pending_turn` 提交进当前 session 的 `turns`，然后清空 `pending_turn` | 这一步决定“本轮是否进入历史”               |

## 二、如果文本消息进入 task：任务执行链状态变化

| 步骤 | 函数 | 重点看哪些状态 | 发生了什么变化 | 为什么重要 |
| --- | --- | --- | --- | --- |
| 1 | `StateDecisionEngine.begin_task_execution()` | `last_route` `conversation_state` | 先把这一轮记成 task 路由，通常进入任务态 | 说明“这轮正式按 task 处理了” |
| 2 | `TaskHandler.handle()` | 无 | 自己基本不直接改；只是调用 `CommandProcessor` 和 `FlowExecutor` | 它是 task 总入口，不是细节执行点 |
| 3 | `CommandProcessor._handle_start_flow()` | `active_task` `paused_tasks` `active_system_task` | 可能出现 3 种情况：启动新任务、恢复旧任务、打断当前任务再切新任务；同时可能激活系统提示流 | 这是任务生命周期变化最大的地方 |
| 4 | `CommandProcessor._handle_set_slots()` | `active_task.slots` | 把命令里的槽位写进当前任务 | 这是 planner/object 承接结果真正落状态的地方 |
| 5 | `CommandProcessor._handle_resume_flow()` | `active_task` `paused_tasks` `active_system_task` | 从 `paused_tasks` 里恢复某个任务为当前任务 | 这是“继续刚才任务”的核心 |
| 6 | `CommandProcessor._handle_cancel_flow()` | `active_task` `paused_tasks` `focused_object` `active_system_task` | 结束当前任务，清空被打断任务，并清理对象上下文；再起一个取消系统任务 | 这是 task 退出的重要收口点 |
| 7 | `FlowExecutor._advance_until_action()` | `active_task.step_id` `active_system_task.step_id` | 不断推进当前 step，直到遇到 action | 说明流程图正在往前跑 |
| 8 | `FlowExecutor._run_start_step()` | `step_id` | 把 step 从 start 推到下一个 step | 只是流程推进，不是业务变化 |
| 9 | `FlowExecutor._run_collect_slots_step()` | `active_task.slots` `active_system_task` | 如果槽位已有值，就继续；如果没有，就启动 `system_collect_information` 系统任务追问用户 | 这是“补槽”逻辑最核心的状态变化 |
| 10 | `FlowExecutor._try_fill_slot_from_focused_object()` | `active_task.slots` `focused_object` | 如果当前对象能提供该槽位，就自动补进去 | 这是对象上下文和 task 流程接上的点 |
| 11 | `FlowExecutor._run_action_step()` | `step_id` | 先推进 step，再构造 `ActionCall` | action 触发前的最后一步 |
| 12 | `ActionRunner.run()` | 无 | 自己不直接改状态 | 它只是分发 action |
| 13 | 具体 `custom action` | 不直接改主状态，返回 `slot_updates` | 比如查工单状态、查进度、提交催办、投诉，主要产出 `slot_updates` 和 `messages` | action 本身尽量不直接碰 `DialogueState` |
| 14 | `FlowExecutor.run_task()` 接收 `ActionResult` | `active_task.slots` | 把 `slot_updates` 统一写回 `active_task.slots` | 这是 action 查询结果真正落状态的位置 |
| 15 | `FlowExecutor._run_end_step()` | `active_task` 或 `active_system_task` | 当前 step 是 end 时，结束系统任务或业务任务 | 这是流程收口点 |
| 16 | `StateDecisionEngine.finalize_task_execution()` | `last_task_outcome` `conversation_state` | 根据 task 执行后状态，记录 outcome，并更新高层会话态 | 这是 task 轨执行完成后的“状态总结” |

## 三、对象消息主链：状态变化时间线

| 步骤 | 函数 | 重点看哪些状态 | 发生了什么变化 | 为什么重要 |
| --- | --- | --- | --- | --- |
| 1 | `DialogueEngine._route_turn()` | 无 | 识别当前是对象消息，转进对象轨 | 这里只是分流 |
| 2 | `ObjectTurnHandler.handle()` | `focused_object` `active_task` | 是对象轨总入口 | 这里开始决定对象是“只是切上下文”还是“要承接任务” |
| 3 | 对象消息写入当前对象 | `focused_object` | 当前对象会被设为新的 `focused_object` | 这是对象轨最直接的状态变化 |
| 4 | `FocusedObjectResolver.resolve_object_commands()` | `active_task` `focused_object` `active_task.slots` | 如果当前任务正缺对象 ID 类槽位，就返回 `SetSlotsCommand` | 对象消息可能直接变成 task 输入 |
| 5 | `ObjectSlotHandoff._flow_has_unfilled_collect_slot()` | `active_task.step_id` `active_task.slots` | 不直接改状态 | 它只是判断“这个对象能不能承接进当前 task” |
| 6 | `ObjectTurnHandler._handle_object_switch_during_active_task()` | `active_task` `paused_tasks` `active_system_task` `focused_object` `conversation_state` | 如果对象切换发生在活跃任务中，可能触发 `reset_runtime_state()` | 防止旧对象任务上下文污染新对象 |
| 7 | `_handoff_object_slots()` -> `TaskCommandExecutor.execute()` | `active_task.slots` `last_route` `last_task_outcome` | 如果对象成功补槽，就会真正进入 task 执行链 | 对象轨和 task 轨在这里接上 |
| 8 | `_respond_object_requires_intent()` | `last_route` `conversation_state` | 如果对象有了但意图不明，进入 clarify | 这就是“点了对象但还没说要干嘛”的处理方式 |

## 四、最值得背的状态变化主线

如果你不想背整张表，只背这 6 步也够用：

1. `begin_turn()`
   - 创建 `pending_turn`
   - 清空上一轮的 `last_route` 和 `last_task_outcome`

2. `planner + protocol gate`
   - 主要读状态
   - 基本不改主运行态

3. `state decision`
   - 决定这一轮最终去哪条轨
   - 更新 `conversation_state` 和 `last_route`

4. 如果进入 task
   - `CommandProcessor` 改任务栈：`active_task / paused_tasks / active_system_task`
   - `FlowExecutor` 改运行进度：`step_id / slots`

5. task 执行结束
   - 更新 `last_task_outcome`
   - 重新收口 `conversation_state`

6. `commit_turn()`
   - 把 `pending_turn` 写进 `sessions`
   - 清空 `pending_turn`

## 五、按模块复习时，到底盯什么

### `DialogueEngine`
只盯：
- `current_session_id`
- `pending_turn`
- `sessions`

因为它主要解决“这一轮有没有被正式接住”。

### `FocusedObjectResolver / ObjectTurnHandler`
只盯：
- `focused_object`
- 必要时看 `active_task.slots`

因为它主要解决“当前围绕谁”和“对象能不能承接补槽”。

### `Planner / Protocol Gate`
基本不盯写状态。

只记住：
- 它们主要读状态
- 真正推进状态是在后面

### `StateDecisionEngine / StateMachine`
只盯：
- `conversation_state`
- `last_route`
- `last_task_outcome`

因为这一层的价值就是“高层定性”和“状态记账”。

### `CommandProcessor`
只盯：
- `active_task`
- `paused_tasks`
- `active_system_task`
- `focused_object`

因为它做的是任务生命周期切换。

### `FlowExecutor`
只盯：
- `active_task.step_id`
- `active_task.slots`
- `active_system_task`

因为它做的是“流程往前推一步”。

### `Action`
通常不盯完整 state，只看：
- 返回了什么 `slot_updates`
- 返回了什么 `messages`

因为 action 大多不直接改状态，而是通过 `FlowExecutor` 统一回写。

## 六、排查问题时的最小状态检查法

以后你顺着函数链看，不要每一步都问“所有字段怎么变”，只问这 4 句：

1. 这一步有没有改 `focused_object`？
2. 这一步有没有改任务栈：`active_task / paused_tasks / active_system_task`？
3. 这一步有没有改高层记录：`conversation_state / last_route / last_task_outcome`？
4. 这一步有没有把 `pending_turn` 写进历史？

只要这 4 句能答上来，你对这个项目的状态流转理解就已经抓住主干了。

---

## 一句话总结

这个项目的状态流转，不要按“所有字段”去记，而要按下面这条主线去记：

**会话有没有接住 -> 当前围绕谁 -> 任务有没有启动/切换 -> 流程跑到哪 -> 这一轮最后被系统定性成什么。**

只要这条线清楚，面试和复习都会轻很多。
