## Q1：`chat_router` 里为什么会有这么多函数？

### 结论

不是真的“接口太多”，而是把 API 层的职责拆开了。

### 解释

- 真正的路由入口其实就几类：发消息、查历史、会话相关这类 endpoint
- 其余很多函数属于“请求对象解析”“响应对象组装”“内部领域对象转 API schema”
- 这些函数放在 `chat_router` 附近，看起来会显得多，但它们不是业务逻辑膨胀

### 你可以怎么理解

把它看成两层：

1. 路由函数：处理 HTTP 进出
2. 转换函数：把接口层数据和领域层数据对接起来

核心点不是“函数数量”，而是“业务判断有没有混进 router”。当前判断是：有些辅助函数偏多，但主业务没有全塞进 router，这点是正常的。

## Q2：`schemas.py` 里为什么定义了这么多类，而且有些还很像？

### 结论

这是接口边界建模带来的正常重复，不全是坏事。

### 解释

- `schemas` 主要服务的是 API 契约，不是内部运行态
- 很像的类，往往分别代表：
  - 请求体
  - 响应体
  - 嵌套子对象
  - 历史记录展示对象
- 它们字段可能接近，但语义不一样

### 为什么之前没这么多类也能跑

能跑，不等于适合继续扩。

以前类少，通常意味着：

- 领域对象直接暴露给接口层
- 入参出参混用
- 某个字段今天给前端，明天又被内部逻辑顺手复用

这样初期推进快，但后期会越来越乱。现在这些类的意义，是把“接口长什么样”和“内部状态怎么跑”分开。

### 当前也确实有的重复

不是说完全没有收敛空间。少数 schema 的确可能后续再合并，但原则上要先保边界，再谈去重。

## Q3：以前没有这么重的类和函数，不也能跑吗？现在这些东西是因为什么做出来的？

### 结论

这次加重的根因不是为了“更花哨”，而是为了把项目从“能跑”推进到“可解释、可维护、可扩展”。

### 主要原因

1. 对话从单轮变成多轮后，状态开始变复杂
2. LLM 输出不能直接信，需要中间协议层兜住
3. 任务、知识、闲聊、澄清这几条轨道开始互相打架
4. 如果不立边界，所有补丁都会堆回入口函数

### 所以才出现了这些中间层

- `TurnPlan`：把模型输出结构化
- `Protocol Gate`：把模型输出拦一遍
- `State Decision`：显式决定这一轮怎么走
- `Track Execution`：真正分发到业务执行层
- `DialogueState`：统一承载运行时状态

它们看着比以前重，但核心收益是：后面解释问题时终于有“层”可以讲，而不是只能顺着日志追补丁。

## Q4：`track_execution` 是个啥？为什么专门搞一层？

### 结论

`track_execution` 就是“轨道执行层”。

### 它解决的问题

前面几层已经完成了：

1. 理解用户这轮想干嘛
2. 校验这个理解能不能落地
3. 决定当前应该走哪条轨道

那最后总要有人真正去执行：

- 走 `task`
- 走 `knowledge`
- 走 `clarify`
- 走 `chitchat`

`track_execution` 就是这层“真正落地”的入口。

### 为什么不直接塞回 `DialogueEngine`

因为 `DialogueEngine` 应该是总编排，不该自己又做路由判断、又做执行细节、又做状态回写。把执行层单独拎出来之后，入口主链会清楚很多。

## Q5：`DialogueEngine` 里初始化那么多东西，到底都是什么？

### 结论

大部分不是“逻辑太多”，而是“依赖装配太多”。

### 可以分三组看

#### 1. 外部能力

- `turn_planner`
- `knowledge_handler`
- `chit_chat_handler`
- `clarify_responder`
- `task_handler`

这些是具体干活的人。

#### 2. 内部控制器

- `turn_protocol_gate`
- `state_decision`
- `focus_resolver`

这些是控制主链秩序的人。

#### 3. 执行运行时

- `text_turn_handler`
- `object_turn_handler`
- `task_executor`

这些是把“文本消息”和“对象消息”真正跑下去的人。

### 为什么看起来会很大一坨

因为 `DialogueEngine` 现在承担了“依赖总装配”的角色。也就是说，它像一个总控台，自己未必复杂，但它必须把下游模块接起来。

## Q7：我顺着调用看到 `text_turn_handler.handle()` 了，这函数里每一步都是什么意思？

### 一句话版

它本质上就是一条固定流水线：

`对象切换 -> 计划生成 -> 协议闸门 -> 状态决策 -> 轨道执行`

### 分步解释

#### 第 1 步：尝试切换当前关注对象

`focus_resolver.try_switch_focused_object_from_text(dialogue_state)`

- 先看这句文本有没有把当前 `focused_object` 切走
- 比如从一个服务项目追问切到另一个服务项目，或者从当前工单切到别的工单

#### 第 2 步：生成本轮计划

`turn_planner.predict(dialogue_state, flows, intents)`

- 调用 planner/LLM
- 把用户输入转成结构化 `TurnPlan`

#### 第 3 步：过协议闸门

`turn_protocol_gate.process(...)`

- 先做 `normalize`
- 再做 `validate`
- 如果不合法，就不继续执行，而是进入澄清分支

#### 第 4 步：没过检就直接澄清

`_respond_validation_failure(...)`

- 先在状态里记一笔 clarify
- 再调用 `clarify_responder` 回答用户

#### 第 5 步：构建文本轮次上下文

`state_decision.build_text_context(dialogue_state, gate_result.turn_plan)`

会得到一个 `TextTurnContext`，里面主要有三样：

- `turn_plan`
- `semantic_kind`
- `decision`

也就是：

- 模型打算怎么处理
- 语义上属于哪一类
- 状态机最终认定怎么执行

#### 第 6 步：真正执行

`_execute_context(...)`

这里只分四类：

1. `directive`：执行运行时指令，比如退出上下文
2. `task`：走任务执行器
3. `knowledge`：走知识问答
4. 其他：按闲聊处理

### 为什么这个函数会让人难读

因为它不是细节函数，而是“编排函数”。

它自己很短，但每一行都在跨层：

- 先修上下文
- 再产计划
- 再做协议收口
- 再做状态决策
- 再把执行分发出去

所以读它时不要纠缠“这一行内部怎么实现”，而要先抓“这一行属于哪一层”。
