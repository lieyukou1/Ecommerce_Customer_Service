# 前端与中台字段功能对照速查手册

## 定位

这是第二阶段字段与对象翻译速查手册。

## 什么时候看

- 需要翻译老师电商字段到物业字段时
- 需要确认前端对象字段、中台字段、数据库字段对应关系时

## 解决什么问题

- 字段名、对象类型、接口字段、数据库表之间如何对齐
- 第二阶段最值得优先复用的字段链是什么
- 哪些老师语义最容易误带回当前项目

## 不用于什么

- 不代替主链接入设计
- 不代替会话交接记录
- 不代替长期协作规则

## 1. 这份手册是干什么的

这份手册服务的不是“理解项目背景”，而是后续每天实际敲代码时的快速对照。

## 2. 总体语义映射

| 老师电商语义 | 当前物业语义 | 说明 |
| --- | --- | --- |
| `user` | `resident` | 左上角输入从 `sender_id=u1001` 迁移为 `resident_id=r1001` |
| `order` | `work_order` | 订单对象整体迁移为工单对象 |
| `product` | `service_item` | 商品对象整体迁移为服务项目对象 |
| `logistics_record` | `service_progress_record` | 物流主记录迁移为工单处理进度主记录 |
| `logistics_trace` | `service_progress_trace` | 物流轨迹迁移为处理轨迹时间线 |
| `refund_request` | `complaint_request` | 退款申请迁移为投诉/异议请求 |
| `shipping_urge_request` | `work_order_urge_request` | 催发货迁移为催办工单 |
| `receiver_*` | `contact_* / service_address` | 收货信息迁移为联系人与服务地址 |
| `stock_status` | `service_status` | 库存状态迁移为服务可办理/可预约状态 |

## 8. 每天最容易写错的地方

### 8.1 不要把 `sender_id` 和 `resident_id` 混用

- 左侧聊天接口 body 里当前仍然使用 `sender_id`
- 右侧业务对象列表接口路径里使用的是 `resident_id`

也就是说：

- `/api/chat/history?sender_id=r1001`
- `/commerce/residents/r1001/work-orders`

这两个地方参数值相同，但字段名还没统一。

### 8.2 不要再给催办接口传 `note`

老师写法：

```json
{ "submitted_by": "system", "note": "用户希望尽快发货" }
```

当前物业写法：

```json
{ "submitted_by": "resident", "reason": "希望物业尽快处理当前工单" }
```

### 8.3 不要再把 `stock_status` 当成服务状态字段

当前服务项目用的是：

- `service_status`

不是：

- `stock_status`

### 8.4 工单列表的 `summary` 不是数据库原列名

前端现在直接吃：

- `work_orders[].summary`

但数据库里没有 `summary` 这一列。

它是后端在列表构建时，用下面这个来源组装的：

- `work_orders.status_desc`

如果后面你改列表接口，别去 SQL 里硬找 `summary` 列。

### 8.5 工单列表现在没有真实 `cover_url`

老师订单卡片图片直接来自商品封面。

当前工单列表：

- `cover_url` 字段存在
- 但当前后端固定返回 `None`

所以如果你看到前端工单卡没图，不是接口坏了，是当前设计就没有接图片来源。

## 9. 如果你今天在看老师代码，应该怎么翻译

### 9.1 看老师前端时

如果你看到：

- `orders`
- `products`
- `sendOrder()`
- `sendProduct()`
- `activeTab = 'orders'`

那在当前项目里就应该自动翻译成：

- `workOrders`
- `serviceItems`
- `sendWorkOrder()`
- `sendServiceItem()`
- `activeTab = 'work_orders'`

### 9.2 看老师中台接口时

如果你看到：

- `/users/{user_id}/orders`
- `/orders/{order_id}/logistics`
- `/orders/{order_id}/shipping-reminders`

就自动翻译成：

- `/residents/{resident_id}/work-orders`
- `/work-orders/{work_order_id}/progress`
- `/work-orders/{work_order_id}/urge-requests`

### 9.3 看老师数据库时

如果你看到：

- `users.level`
- `orders.receiver_address`
- `logistics_records.status`
- `refund_requests.operator`

就优先去当前项目里对应找：

- `residents.resident_level`
- `work_orders.service_address`
- `service_progress_records.current_stage`
- `complaint_requests.submitted_by`

## 10. 第二阶段最值得优先复用的字段链

等后面智能管家主体开工时，最建议优先复用的是这条字段链：

1. 前端右侧工单卡点击
2. 发送 `object.type = work_order`
3. 携带 `id / title / status / amount / appointment_time / summary`
4. 智能管家识别后调用：
   - `/work-orders/{work_order_id}/status`
   - `/work-orders/{work_order_id}/progress`
5. 再把结果回填到左侧对话

这是当前系统里最稳定、最完整、最适合先接 AI 主体的一条链。

## 11. 一句话总结

后面每天开发时，可以把老师电商系统理解成一套“原语词典”，再用这份手册把它快速翻译成当前物业系统的真实字段、真实接口和真实表结构。

这样你就不用每次重新猜：老师今天讲的这个字段，在我们项目里到底该落到哪。
