## 2. 第二阶段系统分层要重新钉死

第二阶段开始后，默认应把当前项目理解成三层，而不是一层：

### 2.1 前端壳层

目录：

- `customer-service-frontend`

职责：

- 继续保留老师单页壳
- 左侧负责对话输入输出
- 右侧负责结构化对象卡片
- 不承担业务判断

### 2.2 物业业务中台

目录：

- `ecommerce-service-backend`

职责：

- 提供住户、工单、服务项目、进度、投诉、催办等业务 API
- 作为被智能管家调用的业务底座
- 不承担 LLM 推理主职责

### 2.3 智能管家主体

目录：

- `customer-service-backend`

当前现状：

- 已是老师那套异步技术栈方向
- `pyproject.toml` 里依赖包含 `aiomysql`、`httpx`、`fastapi`、`langchain`
- `.env` 已包含 `COMMERCE_API_BASE_URL`、`DATABASE_URL`、`APP_PORT=18082`
- 代码包根目录当前叫 `atuguigu`

第二阶段主战场就是这一层。

## 3. 当前最重要的链路重构目标

第一阶段时：

- `/api/chat/history`
- `/api/chat`

这两个接口是由物业中台里的占位实现直接返回固定文案。

第二阶段开始后，目标应该变成：

- `/commerce/*` 继续由 `ecommerce-service-backend` 提供
- `/api/*` 改由 `customer-service-backend` 提供真实智能管家能力

也就是说：

- **右侧对象数据继续走中台**
- **左侧对话主链切到智能管家后端**

## 4. 第二阶段启动前必须先改的代理认识

当前前端代理配置文件：

- `VITE_COMMERCE_TARGET`
- `VITE_CHAT_TARGET`

这样新会话开工时，不会再把业务中台和智能管家主体混到一起。

## 6. 哪些字段直接拿去做意图识别

不是所有字段都要先进状态，也不是所有字段都该交给 LLM 推断。

### 6.1 工单对象里适合直接给意图识别的字段

| 字段 | 用途 |
| --- | --- |
| `object.type` | 直接判断当前对象类别是不是 `work_order` |
| `object.id` | 直接作为后续工单查询主键 |
| `object.title` | 给模型理解当前业务语义 |
| `attributes.status` | 有助于判断当前是在查状态、催办、投诉还是解释流程 |
| `attributes.summary` | 作为对象上下文摘要，减少重复追问 |

### 6.2 服务项目对象里适合直接给意图识别的字段

| 字段 | 用途 |
| --- | --- |
| `object.type` | 判断当前是不是 `service_item` 场景 |
| `object.id` | 直接作为项目详情查询主键 |
| `object.title` | 辅助识别项目咨询意图 |
| `attributes.service_status` | 判断是“可预约 / 可办理 / 需审核”等解释类问题 |
| `attributes.description` | 直接补充模型上下文 |

### 6.3 文本输入里适合做主意图识别的关键词方向

工单类：

- 状态
- 进度
- 上门时间
- 催办
- 投诉
- 关闭原因

服务项目类：

- 怎么办理
- 怎么收费
- 需要什么资料
- 什么时候能办
- 适用范围

规则知识类：

- 物业费
- 装修报备
- 停车规则
- 宠物管理
- 噪音 / 公区规则

## 7. 哪些字段应该进状态槽位，而不是每轮重猜

### 7.1 会话级基础槽位

这些字段建议一进入对话就尽量落进 state：

| 槽位名建议 | 来源 | 说明 |
| --- | --- | --- |
| `sender_id` | `ChatRequest.sender_id` | 当前住户编号 |
| `active_object_type` | `object.type` | 当前激活对象类型 |
| `active_object_id` | `object.id` | 当前激活对象主键 |
| `active_object_title` | `object.title` | 当前激活对象标题 |
| `last_user_text` | `text` | 最近一轮原始用户文本 |

### 7.2 工单对象槽位

| 槽位名建议 | 来源 |
| --- | --- |
| `work_order_id` | `object.id` |
| `work_order_status_hint` | `attributes.status` |
| `work_order_summary_hint` | `attributes.summary` |
| `appointment_time_hint` | `attributes.appointment_time` |
| `work_order_amount_hint` | `attributes.amount` |

### 7.3 服务项目对象槽位

| 槽位名建议 | 来源 |
| --- | --- |
| `service_item_id` | `object.id` |
| `service_item_status_hint` | `attributes.service_status` |
| `service_item_description_hint` | `attributes.description` |
| `service_item_price_hint` | `attributes.price` |

### 7.4 为什么要这么做

因为这些字段前端已经显式给了，没必要每轮再让 LLM 去猜：

- 当前对象是谁
- 住户在问哪条工单
- 这是工单场景还是服务项目场景

第二阶段最应该继承的，就是这种“结构化对象输入减少 LLM 推断负担”的骨架优势。

## 8. 第二阶段第一批最适合接入的真实意图

建议不要一上来全接。

第一批最适合接的意图只有四类：

### 8.1 `work_order_status_query`

适用输入：

- 选中工单后问“现在什么状态”
- 纯文本里提到“工单状态”“当前处理到哪了”

调用接口：

- `GET /work-orders/{work_order_id}/status`

返回重点字段：

- `status`
- `status_desc`
- `priority`

### 8.2 `service_progress_tracking`

适用输入：

- 选中工单后问“处理进度”“什么时候上门”“时间线”

调用接口：

- `GET /work-orders/{work_order_id}/progress`

返回重点字段：

- `current_stage`
- `stage_desc`
- `service_team`
- `assignee_name`
- `traces[]`

### 8.3 `service_item_detail_query`

适用输入：

- 选中服务项目后问“怎么收费”“怎么办理”“服务范围是什么”

调用接口：

- `GET /service-items/{service_item_id}`

返回重点字段：

- `description`
- `price`
- `service_status`
- `attributes`

### 8.4 `resident_rule_qa`

适用输入：

- 物业费规则
- 装修报备
- 停车规则
- 宠物管理

调用来源：

- 知识轨道

这一类不是先查业务中台，而是接第二阶段的知识问答主链。

## 9. 第二阶段第一条闭环建议怎么接

建议优先接这一条：

`右侧选中工单 -> 左侧发对象消息 -> 智能管家识别 status/progress 意图 -> 调物业中台 -> 返回结构化答复`

### 为什么这条最适合先做

1. 前端对象输入已经准备好了
2. 中台状态和进度接口已经稳定
3. 不需要先做复杂写操作
4. 非常适合暴露 state、意图识别、接口编排这些核心问题

### 最小成功标准

1. 用户点击右侧某条工单
2. 左侧发送 `work_order` 对象
3. 用户补一句“这单现在处理到哪了”
4. 智能管家从 state 中拿到 `work_order_id`
5. 智能管家调用 `/work-orders/{id}/progress`
6. 左侧返回自然语言结果，不再是占位文案

## 11. 第二阶段写操作怎么收口

写操作不要一开始就放开太多。

### 11.1 催办

请求体目前固定结构：

```json
{
  "submitted_by": "resident",
  "reason": "希望物业尽快处理当前工单"
}
```

建议第二阶段先采用：

- 如果用户只表达“帮我催一下”，先由系统生成默认 `reason`
- 如果用户给了更具体原因，再覆盖默认值

### 11.2 投诉

请求体目前固定结构：

```json
{
  "submitted_by": "resident",
  "reason": "联调验收投诉测试"
}
```

建议第二阶段不要一开始就自动提交，先至少做一次澄清：

- 投诉原因是什么
- 是否确认提交

原因很简单：

- 投诉属于更敏感的写操作，误触发代价比查状态高得多

## 12. 第二阶段返回给前端时建议保留的输出形态

前端现在能稳定渲染两类消息：

- `text`
- `object`

所以第二阶段智能管家最稳的回复策略是：

### 12.1 查询类回复

优先返回：

- 一段清晰文本说明
- 必要时附带建议按钮 / suggestions

### 12.2 对象回显类回复

在需要强调“当前正在处理哪条工单 / 哪个服务项目”时，可以回发对象消息，继续沿老师的对象驱动路线走。

### 12.3 暂时不建议一上来改前端协议

第二阶段第一轮最好先兼容当前：

- `ChatResponse.messages[]`

不要一开工就改成一套全新的 SSE 协议或事件流，除非当前主链已经稳定。
