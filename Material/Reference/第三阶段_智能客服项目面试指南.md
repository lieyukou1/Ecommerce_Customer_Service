## 1. 项目概述

### 1.1 项目背景

这是一个面向物业社区场景的智能客服系统。  
用户可以通过文本输入，也可以通过前端点击工单、服务项目等对象卡片，继续和客服系统对话。

系统主要覆盖三类能力：

1. 业务任务处理  
   例如工单催办、投诉发起、服务办理等
2. 信息查询与知识追问  
   例如服务项目收费、办理方式、工单状态、处理进度等
3. 闲聊与兜底澄清  
   例如寒暄、模糊表达、意图不完整场景

### 1.2 项目目标

这个项目要解决的，不只是“让大模型回答问题”，而是：

- 让系统支持多轮对话
- 让状态能跨轮保存和恢复
- 让业务流程能被约束执行
- 让对象点击、任务切换、退出重开这些真实交互能跑通

一句话概括：

**LLM 负责理解，后端状态机和流程编排负责可控执行。**

### 1.3 当前版本最值得讲的成果

这次第三阶段重构之后，项目最大的变化不是多过了几个 badcase，而是：

1. 对话主链被收成了显式结构
2. 状态流转不再主要靠下游函数偷偷改字段
3. 顶层入口、协议闸门、状态决策、执行层、运行态之间的边界清楚了

## 2. 整体架构设计

### 2.1 当前主链

当前后端主链可以讲成 6 个一级概念：

1. `DialogueEngine`
2. `Planner`
3. `Protocol Gate`
4. `State Decision`
5. `Track Execution`
6. `DialogueState`

对应架构图见：

- [03-P0最终收口架构图与讲解.md](/D:/Desktop/SGG_Project/Ecommerce_Customer_Service/Docs/第三阶段架构图/03-P0最终收口架构图与讲解.md)

### 2.2 一次请求怎么走

可以按这条链讲：

`/api/chat`
-> `chat_router`
-> `DialogueService`
-> `DialogueEngine`
-> `TurnPlanner`
-> `TurnProtocolGate`
-> `StateDecisionEngine`
-> `TrackExecution`
-> `DialogueState` 落状态
-> 返回前端

### 2.3 为什么这样拆

因为这个项目同时有三种复杂度：

1. 自然语言理解复杂度
2. 多轮状态复杂度
3. 业务流程复杂度

如果把这三件事混在一个模块里，最后就会变成：

- 意图判断和状态修改耦合
- 执行层顺手改运行态
- 问题一多就只能补 `if`

所以这次重构的核心，就是把它们重新拆开：

- 理解交给 `Planner`
- 协议收口交给 `Protocol Gate`
- 状态流转交给 `State Decision`
- 业务执行交给 `Track Execution`

## 3. 核心模块怎么讲

### 3.1 API / Service 层

相关位置：

- [chat_router.py](/D:/Desktop/SGG_Project/Ecommerce_Customer_Service/customer-service-backend/atguigu/api/router/chat_router.py)
- `dialogue_service.py`

职责很简单：

1. 接收前端请求
2. 把请求模型转成领域对象
3. 调用 `DialogueService`
4. 返回结构化响应

这里你要强调一个点：

**API 层不写业务判断，只做模型转换和服务调用。**

这会让面试官觉得你有分层意识。

### 3.2 DialogueState

这是整个项目最核心的运行态聚合根。

你最值得背的字段就是这几个：

- `active_task`
- `paused_tasks`
- `active_system_task`
- `focused_object`
- `conversation_state`
- `last_route`
- `last_transition`
- `last_task_outcome`

这些字段的价值分别是：

1. `active_task`  
   当前业务任务快照
2. `paused_tasks`  
   被打断但可恢复的任务栈
3. `active_system_task`  
   系统收集信息或确认流程
4. `focused_object`  
   当前对话聚焦的工单或服务项目
5. `conversation_state`  
   顶层会话状态
6. `last_route`  
   最近一次路由决策痕迹
7. `last_transition`  
   最近一次状态迁移痕迹
8. `last_task_outcome`  
   最近一次任务执行结果痕迹

面试时可以这样说：

> 我们不是做无状态问答，而是把用户对话抽象成一份 `DialogueState`。  
> 每次请求先加载状态，处理完后再保存状态，这样才能支撑多轮任务推进、退出、恢复、对象追问和历史对话承接。

### 3.3 DialogueEngine

相关位置：

- [dialogue_engine.py](/D:/Desktop/SGG_Project/Ecommerce_Customer_Service/customer-service-backend/atguigu/engine/dialogue_engine.py)

现在这个模块已经收得比较干净了，它只做：

1. session 生命周期
2. begin / commit turn
3. 文本 / 对象消息分流

这点非常值得讲，因为它正好能体现你这次重构的成果：

**把顶层入口从“大杂烩”收回成纯编排器。**

### 3.4 Planner

相关位置：

- `plan/planner.py`
- `prompt/jinja2/turn_plan.jinja2`

职责：

1. 读取用户当前输入
2. 带上历史和运行态
3. 让模型输出结构化 `TurnPlan`

这里的关键点不是“用了 LLM”，而是：

**LLM 不直接改状态，不直接调业务接口，只负责输出结构化计划。**

### 3.5 Protocol Gate

相关位置：

- [protocol_gate.py](/D:/Desktop/SGG_Project/Ecommerce_Customer_Service/customer-service-backend/atguigu/plan/protocol_gate.py)

这是本次重构后非常值得讲的新概念。

它的作用是：

1. 承接 `TurnPlanNormalizer`
2. 承接 `TurnPlanValidator`
3. 把模型输出变成系统能接受的协议入口

也就是：

**模型会说什么，不等于系统就执行什么。**

这个点很加分，因为它天然对应“LLM 可控性”。

### 3.6 State Decision

相关位置：

- `engine/state_decision/engine.py`
- `engine/state_decision/state_machine.py`

这是第三阶段最核心的重构成果。

它把原来隐式的字段修改，收成了显式过程：

1. route decision
2. state transition
3. task outcome transition

可以概括成一句：

**先决定这一轮属于什么，再决定状态怎么迁移，最后才进入执行。**

这和之前“很多地方顺手改状态”的写法比，最大的优势是：

- 可观测
- 可解释
- 可测

### 3.7 Track Execution

相关位置：

- `engine/track_execution/`

现在这层已经比之前清楚很多了，主要负责三类执行：

1. 文本 turn 执行
2. 对象 turn 执行
3. task command 执行

而且这次又多做了一步收口：

**执行层的 wiring 已经从 `DialogueEngine` 中拿出来，收回到了 `track_execution/` 自己内部。**

这点你可以作为“结构收敛”的一个小亮点讲。

### 3.8 Focus 机制

相关位置：

- `engine/focus/`

这个模块是当前物业项目里非常贴近真实交互的点。

它解决的是：

- 用户点击工单后，后面追问该围绕哪个对象
- 用户换了个对象后，系统怎么切上下文
- 对象消息能不能直接承接到 slot

你不用讲太细，只要抓住一句：

**`focused_object` 是把“前端点击对象”接入多轮对话的关键桥梁。**

### 3.9 YAML Flow

这个项目仍然保留了 YAML 配置式业务流程。

它适合这种场景的原因是：

- 业务流程相对确定
- slot 收集比较明确
- 更适合教学和讲解

你可以说：

> 对于这类确定性较强的客服流程，我没有把全部逻辑都交给 LLM，而是把流程本身配置化，把 LLM 放在理解层，把执行留给后端。

## 4. 这次重构到底改了什么

这个问题你一定会被问到。

建议你按“三个前后对比”讲：

### 4.1 之前的问题

之前的主要问题不是不能跑，而是：

1. 状态流转太依赖下游函数隐式修改
2. 一个 badcase 一补，`engine` 就越来越胖
3. 顶层入口知道太多执行层细节
4. 很多中间概念虽然有用，但全暴露在高层，阅读负担很重

### 4.2 这次怎么改

这次我没有继续围绕 badcase 补行为分支，而是把重心转成架构收口：

1. 固定 6 个一级概念
2. 增加 `Protocol Gate`
3. 增加显式 `State Decision`
4. 清退旧的 `runtime/turns` 层
5. 回收执行层 wiring
6. 稳定运行态观测字段

### 4.3 改完之后的收益

改完后最明显的收益不是“某一题多答对了”，而是：

1. 主链能讲清楚了
2. 状态问题更容易排查
3. 后续再优化时，不需要老从顶层入口补丁
4. 项目更像真实工程，而不是 demo 脚本拼装

## 5. 为什么不用纯 Agent / 为什么还保留流程约束

这个问题很容易被追问。

你可以这样答：

> 因为这个项目是客服场景，不是开放式探索型 Agent。  
> 它里面有大量确定性任务，比如催办、投诉、状态查询、服务办理，这些流程如果完全交给模型自由发挥，风险很高。  
> 所以我的思路是：LLM 做理解和计划，后端做协议约束、状态管理和流程执行。

再补一句会更稳：

> 这也是为什么我引入 `Protocol Gate` 和显式状态机，而不是继续让模型直接带着业务往下跑。

## 6. 高频面试问答

### 6.1 你这个项目的核心亮点是什么？

答法：

1. 多轮状态管理
2. LLM 理解 + 后端约束执行
3. 对象消息承接
4. 显式状态机重构

一句版：

> 核心亮点不是“接了大模型”，而是把大模型、状态机和业务流程编排结合起来，让系统既自然，又可控。

### 6.2 你为什么要重构状态机？

答法：

> 因为项目后期最明显的问题不是单点功能缺失，而是状态流转越来越依赖下游函数隐式改字段。  
> 这样一来，badcase 一多，就只能不断补 `if`。  
> 所以我把状态流转提到更高层，收成显式 `State Decision + DialogueStateMachine`，先决定，再迁移，再执行。

### 6.3 你们怎么控制 LLM 的不确定性？

答法：

> 主要靠三层：
> 1. 结构化 `TurnPlan`
> 2. `Protocol Gate` 做 normalize 和 validate
> 3. 业务执行由确定性后端代码完成

### 6.4 为什么这个项目适合 YAML Flow？

答法：

> 因为很多物业客服任务是标准化流程，步骤明确、slot 明确、结果明确。  
> YAML 在这里更像“流程配置层”，比完全写死在代码里更容易维护，也比全交给 LLM 更可控。

### 6.5 你这次重构有没有动前端和中台？

答法：

> 这轮后端架构收口本身没有依赖前端和中台改代码。  
> 我保留了原有调用入口和协议边界，主要调整的是后端内部的职责分层和状态流转方式。

### 6.6 你这个项目还有什么不足？

这个问题别回避。

可以答：

1. 多意图并行还没正式接入
2. 某些模糊追问仍然会摇摆
3. 执行层还可以进一步工程化
4. 更真实的线上场景还需要更强监控和 trace

这种回答会显得你有判断，不是盲目吹。

## 7. STAR 版本素材

### 7.1 状态机重构

**S**  
项目后期虽然功能越来越全，但问题开始集中暴露在状态流转上。很多状态修改散落在下游函数里，新增 badcase 时很容易继续补 `if`。

**T**  
我要把项目从“能跑”推进到“结构可讲、状态可解释、后续可继续优化”。

**A**  
我把主链收成 6 个一级概念，引入 `Protocol Gate` 和显式 `State Decision`，把状态迁移统一交给 `DialogueStateMachine`，并把执行层 wiring 从顶层入口里回收。

**R**  
最终系统主链清晰了，状态观测面稳定了，项目不再只是 patch accumulation，而是具备了可持续演进的后端结构。

### 7.2 退出机制与上下文清理

**S**  
早期项目容易出现“进得去出不来”的状态残留问题，比如 `focused_object`、`active_task` 留在上下文里。

**T**  
我要让用户能退出当前上下文，重新开始新话题，而且不串味。

**A**  
我统一了 runtime exit 语义，把退出行为通过状态决策层和运行态重置通道来处理，而不是让各个分支自己想办法清状态。

**R**  
退出后再进入新对象、新话题时，状态更稳定，也更容易被回归测试和手测验证。

## 8. 一分钟版本

> 这是一个物业智能客服项目，核心不是简单接大模型，而是让大模型和后端状态机、业务流程编排配合起来。  
> 用户既可以文本提问，也可以点击工单、服务项目对象继续追问。  
> 后端主链我收成了 `DialogueEngine -> Planner -> Protocol Gate -> State Decision -> Track Execution -> DialogueState`。  
> 其中 LLM 负责输出结构化计划，`Protocol Gate` 负责校验和收口协议，`State Decision` 负责显式状态流转，执行层只负责真正落业务。  
> 这次重构的核心成果，不是多答对几个 case，而是把整个对话系统整理成了可解释、可观测、可继续演进的结构。

## 9. 三分钟版本

> 这个项目面向物业客服场景，目标是支持多轮对话下的工单处理、服务项目咨询、业务办理和闲聊兜底。  
> 它最大的难点不是“让模型能回答”，而是让系统在多轮对话里既自然又可控。  
>  
> 所以我没有让 LLM 直接执行业务，而是让它只负责输出结构化 `TurnPlan`。  
> 后端再通过 `Protocol Gate` 做 normalize 和 validate，防止模型输出直接污染业务执行。  
>  
> 在这之上，我又做了显式状态机重构，把主链整理成 6 个一级概念：`DialogueEngine`、`Planner`、`Protocol Gate`、`State Decision`、`Track Execution` 和 `DialogueState`。  
> `DialogueState` 保存 `active_task`、`paused_tasks`、`focused_object`、`conversation_state` 等关键字段，`StateDecisionEngine + DialogueStateMachine` 负责统一状态迁移。  
>  
> 这次重构最大的收益，是项目终于从“靠很多分支补状态”的形态，变成了“先决策、再迁移、再执行”的结构。  
> 这样后续无论是排查问题，还是继续做优化，都会更稳定、更可解释。
