## 一句话项目

这是一个物业智能客服项目，核心是把 **LLM 理解能力** 和 **后端显式状态机 + 流程执行能力** 结合起来。

## 一句话亮点

不是“接了大模型”，而是把主链收成了：

`DialogueEngine -> Planner -> Protocol Gate -> State Decision -> Track Execution -> DialogueState`

## 你最该记的 6 个概念

1. `DialogueEngine`：顶层编排入口
2. `Planner`：模型输出 `TurnPlan`
3. `Protocol Gate`：模型输出收口
4. `State Decision`：先决策再迁移
5. `Track Execution`：真正执行业务
6. `DialogueState`：多轮运行态聚合根

## 你最该记的 8 个字段

1. `active_task`
2. `paused_tasks`
3. `active_system_task`
4. `focused_object`
5. `conversation_state`
6. `last_route`
7. `last_transition`
8. `last_task_outcome`

## 为什么要重构

因为项目后期最大问题不是少一个功能，而是：

- 状态修改散落
- badcase 一多就补 `if`
- 顶层入口知道太多细节

所以我要把它改成：

**先计划 -> 再校验 -> 再状态决策 -> 再执行**

## 为什么不用纯 Agent

因为这是客服场景，不是开放探索场景。  
很多流程是确定性的，必须可控。

所以：

- LLM 负责理解
- 后端负责约束和执行

## 你能讲的三点技术亮点

1. `TurnPlan + Protocol Gate` 控制 LLM 不确定性
2. `StateDecisionEngine + DialogueStateMachine` 做显式状态流转
3. `focused_object` 把前端对象点击真正接入多轮对话

## 你能承认的三点不足

1. 多意图并行还没正式接入
2. 模糊追问仍会有摇摆
3. 执行层还能继续工程化

## 面试官问“前端和中台改了没”

答：

这轮后端架构收口本身没有依赖前端和中台改代码，主要改的是后端内部职责分层和状态流转方式。

## 一分钟答法

> 这是一个物业智能客服项目，重点不是单纯调用大模型，而是让系统在多轮对话里既自然又可控。  
> 我把后端主链收成了 `DialogueEngine -> Planner -> Protocol Gate -> State Decision -> Track Execution -> DialogueState`。  
> LLM 只负责输出结构化计划，协议闸门负责收口，状态决策负责显式迁移，执行层负责落业务。  
> 这次重构最大的成果不是多过几个 case，而是让项目从补丁式增长变成了可解释、可观测、可继续演进的结构。
