## 7. 面试高频问题与回答

### [7.1]--[你这个项目整体是怎么工作的？]：
    [答案]：    整体链路可以概括成 `API -> Service -> DialogueEngine -> 具体轨道 -> 状态持久化`。前端把用户消息发到 `/api/chat` 后，后端先在 `chat_router` 里把请求转成领域层 `UserMessage`，然后 `DialogueService` 根据 `resident_id` 从数据库加载该住户的 `DialogueState`，再把消息交给 `DialogueEngine` 处理。`DialogueEngine` 会先准备 session、创建 `pending_turn`，再根据消息类型分流：文本消息走 `TextTurnHandler`，对象消息走 `ObjectTurnHandler`。如果进入 task，就继续走 `CommandProcessor -> FlowExecutor -> ActionRunner`，靠 YAML flow 推进步骤。处理完成后，bot 消息会写回当前 turn，整个 `DialogueState` 再持久化回数据库。

### [7.2]--[为什么要有 DialogueState？]：
    [答案]：    因为客服系统是多轮对话，不能只看当前一句话。`DialogueState` 本质上就是这个系统的“对话运行时内存”。它里面统一保存了 `active_task`、`paused_tasks`、`active_system_task`、`focused_object`、`sessions`、`current_session_id` 等字段。这样系统才能知道用户当前办到哪一步、已经补了哪些槽位、有没有被打断的任务、当前围绕的是哪条工单或哪个服务项目，以及这段对话的历史上下文是什么。

### [7.3]--[LLM 在项目里具体做什么？]：
    [答案]：    在这个项目里，LLM 主要做两件事：**理解** 和 **表达**，不直接执行业务。第一类作用是在 `TurnPlanner` 里，根据用户消息、历史、当前任务、focused object、可用 flow 和知识意图，输出结构化 `TurnPlan`，帮助系统判断这一轮更像 `task / knowledge / chitchat / directive` 哪一类。第二类作用是在 `clarify`、`chitchat` 和部分 `knowledge` 回复里，把系统已有的判断和上下文组织成更自然的语言。真正的状态修改、流程推进、接口调用和数据回写，都是后端代码和 YAML flow 在做，不交给 LLM 自由发挥。

### [7.4]--[如何防止 LLM 乱输出？]：
    [答案]：    这个项目不是“相信 LLM”，而是“先让 LLM 提议，再让代码裁决”。我们用了四层控制：1. `TurnPlan` 结构化输出，要求模型输出固定 JSON；2. `TurnPlanNormalizer` 先做协议级归一化，比如把某些退出表达改写成统一的 runtime directive；3. `TurnPlanValidator` 对 planner 输出做严格校验，比如只允许单轨、task 命令必须合法、不能启动不存在的 flow、knowledge 轨需要满足对象依赖；4. 校验失败不执行，而是进入 `ClarifyResponder`，让用户补充信息。关键不是把 prompt 写得多复杂，而是代码层有明确的协议门。

### [7.5]--[YAML 流程是怎么跑起来的？]：
    [答案]：    YAML 里定义的是 flow、step 和 next link，本质上是一份可配置的业务流程图。系统启动时会先把 YAML 读进来，转成内存里的 `Flow / Step / Link` 对象。文本轨进入 task 后，`CommandProcessor` 先根据命令改状态，比如启动 flow、补槽、恢复任务；然后 `FlowExecutor` 根据当前 `active_task.step_id` 推进：`start` 直接推进，`collect` 检查槽位并在必要时进入系统追问，`action` 构造 `ActionCall` 给 `ActionRunner`，`end` 结束当前任务或系统任务。`ActionResult` 里的 `slot_updates` 再统一写回状态，遇到 `action_listen` 时停住，等待下一轮用户输入。

### [7.6]--[为什么不用 LangGraph，而是自己写 YAML 流程？]：
    [答案]：    因为当前项目的 task 场景，本质上更像确定性的业务 SOP，而不是开放式 Agent 图。比如工单状态查询、进度查询、催办、投诉，这些流程都很适合用“开始 -> 补槽 -> 调 action -> 结束”这种可配置流程表达。自己写 YAML 的好处有三点：第一，业务可读性强，产品、测试、面试官都更容易理解；第二，安全边界清晰，LLM 只能选择已有 flow，不会自由拼流程；第三，工程可解释性更好，状态怎么变、流程怎么跑都能讲清楚。LangGraph 当然也能做，但对这类线性业务流程，框架成本通常高于收益。

### [7.7]--[LangGraph 什么时候更合适？]：
    [答案]：    当系统不是固定流程，而是更复杂的 Agent 编排时，LangGraph 更合适。比如多工具并行调用、多节点之间循环重试、检索和推理混合编排、一个问题要经过搜索、规划、执行、再判断下一步，这种场景天然更像图，而不是像当前项目这样更像状态机和业务流程引擎。

### [7.8]--[task、knowledge、chitchat 怎么区分？]：
    [答案]：    不是按句式分，而是按“系统接下来要不要真的推进业务状态”来分。`task` 是用户在办事，系统要启动或推进业务流程，要补槽、调 action、改状态；`knowledge` 是用户在问信息，系统主要回答，不推进任务生命周期；`chitchat` 是闲聊、寒暄、轻问候。实现上先由 `Planner` 输出 `TurnPlan`，再经过 `Normalizer + Validator + State Decision` 收口，最终只允许落到一条主轨。

### [7.9]--[如何支持任务中断和恢复？]：
    [答案]：    核心靠的是 `active_task + paused_tasks + active_system_task` 这组机制。当前任务 A 执行中，如果用户又发起任务 B，`CommandProcessor` 会先把 A 中断并压入 `paused_tasks`，再启动或恢复 B，同时触发一个系统任务，告诉用户旧任务被打断、新任务接手。之后如果用户说“继续刚才的”，系统就可以通过 `ResumeFlowCommand` 从 `paused_tasks` 栈顶恢复任务，或者按指定 `flow_id` 恢复。

### [7.10]--[对象消息是怎么处理的？]：
    [答案]：    对象消息指的是前端点击工单卡片、服务项目卡片后，直接传来的结构化对象，而不是纯文本。后端收到后会先把它转成 `FocusedObject` 并写入 `state.focused_object`。然后系统不是简单地看“有没有任务”，而是按四步判断：第一，先尝试把对象解析成可执行命令，最常见的就是把对象 ID 承接成 `SetSlotsCommand` 自动补槽，能解析出命令就优先直接交给 task 执行；第二，如果当前正处于 active task，而且这是任务进行中的对象切换，系统会先 `reset_runtime_state()`，再进入 `clarify`，避免旧任务上下文和新对象混用；第三，如果当前还在 active task 上下文里，但这次对象消息没有直接解析出命令，就沿着当前 task 继续执行；第四，只有在“没有可执行命令，也没有可安全延续的 active task”时，系统才会进入 `clarify`，追问用户想基于这个对象查状态、看进度，还是继续办理业务。

### [7.11]--[ActionResult 为什么分 messages 和 slot_updates？]：
    [答案]：    这是为了把“查到了什么数据”和“怎么对用户说”解耦。`messages` 表示 action 想直接返回给用户的话；`slot_updates` 表示 action 查到的数据，要不要写回当前任务上下文。这样 action 更像纯函数，不直接乱改状态；状态回写统一由 `FlowExecutor` 做，副作用集中；某些 action 只负责查数据，后续再由响应模板根据 slots 渲染最终回复。

### [7.12]--[历史对话为什么返回 session_id？]：
    [答案]：    因为这个项目的历史对话不是一条无限长消息流，而是按 session 分段管理的。`session_id` 的作用主要有三点：第一，表示这条消息属于哪一次业务会话；第二，后端有会话超时切换逻辑，超时后会新开 session，所以需要稳定 ID 区分；第三，后续如果要做按 session 回放、归档、筛选或前端分段展示，这个字段很重要。

### [7.13]--[数字人里的 sessionId 和历史对话里的 session_id 是一回事吗？]：
    [答案]：    不是。历史对话里的 `session_id` 是后端业务会话 ID，用来表示某条消息属于哪个客服 session；数字人接口里的 `sessionId` 是数字人渲染或 RTC 会话 ID，用来初始化数字人 SDK。前者服务于业务对话状态，后者服务于音视频通道，不能混用。

### [7.14]--[状态整体 JSON 存储有什么优缺点？]：
    [答案]：    这是一个典型的工程取舍。优点是实现简单，`DialogueState` 可以直接 `to_dict()/from_dict()`，恢复上下文方便，任务、对象、历史、slots 一次性都能恢复，也很适合当前这种“单住户单状态”的教学型多轮对话项目。缺点是细粒度统计和查询不方便，每次更新通常是整份 JSON 级别，状态越长单条记录越大。如果以后要做复杂分析、监控或画像，通常会保留 JSON 在线状态，同时拆出明细表做分析。

### [7.15]--[对话状态膨胀怎么办？]：
    [答案]：    这类问题一般分三层处理。第一是**运行时清理**，任务结束或退出时及时清掉 `active_task`、`active_system_task`、`paused_tasks`、`focused_object` 这些热状态；第二是**会话分段**，当前项目已经有 session 机制，超时后会新开 session，不会把所有历史永远堆在一个连续运行态里；第三是**冷热拆分**，如果后面状态继续膨胀，可以把“当前运行态”和“历史归档”拆开，在线只保留热状态，把 turn 单独归档。

### [7.16]--[这个项目有哪些可以优化的地方？]：
    [答案]：    我会从五个方向答：第一，继续提升 `planner + protocol gate + state decision` 的稳定性，减少边界场景下的误路由；第二，把更多隐式流转收成显式 transition，让状态迁移更好解释、更好排查；第三，把在线运行态和历史明细逐步拆层，提升可分析性；第四，增加更清晰的 trace，比如一轮里的 `plan -> gate -> route -> executor -> outcome` 可视链路；第五，如果后面接本地 7B 或做领域微调，重点应放在更稳的意图识别、更自然的澄清和更像客服的知识回复，而不是把业务执行交给模型。

## 7.x 补充高频问题（结合当前项目实现）

### [7.17]--[为什么要单独做 conversation_state、last_route、last_task_outcome 这三个字段？]：
    [答案]：    这三个字段解决的是三个不同层次的观测问题。`conversation_state` 表示当前高层会话态，比如现在整体是 `ACTIVE_TASK`、`FOCUSED_KNOWLEDGE` 还是 `CLARIFYING`；`last_route` 表示上一轮最终被分发到了哪条轨，方便下一轮判断用户是在延续还是切话题；`last_task_outcome` 表示最近一次 task 执行完的结果，比如继续中、结束、取消、转系统追问。这三项的价值在于让状态机判断不必每次都从底层字段硬推理，而是有更稳定的高层观察面。

### [7.18]--[为什么要区分 active_task 和 active_system_task？]：
    [答案]：    因为业务任务和系统辅助流程不是一回事。`active_task` 是真正的业务流程，比如工单进度查询、催办、投诉；`active_system_task` 是系统为了推动主流程而插入的辅助流程，比如新任务开场白、任务中断提示、槽位收集追问。分开以后，主业务流不会和系统提示流混在一起，`FlowExecutor` 也能优先跑系统任务，跑完再回到业务任务，状态更清楚，用户体验也更自然。

### [7.19]--[为什么对象切换时有时要 reset_runtime_state？]：
    [答案]：    因为对象切换如果发生在任务进行中，很容易导致上下文串味。比如当前任务正围绕工单 A 收集投诉原因，这时用户又切到工单 B，如果不清理 runtime state，系统可能会把 A 的 task、slots、系统追问错误地延续到 B 上。所以当系统检测到“活跃任务中的对象切换”时，会做 runtime reset，把旧任务上下文清掉，避免脏状态跨对象传播。

### [7.20]--[为什么要把 Planner、Protocol Gate、State Decision 拆成三层？]：
    [答案]：    这是为了把“模型猜测”“协议裁决”“状态落地”三件事分开。`Planner` 负责理解用户这一轮想干什么；`Protocol Gate` 负责把模型输出收成系统真正能执行的协议；`State Decision` 负责在协议通过后，把这一轮真正落成状态迁移和轨道执行决策。如果把这三件事全糊在一个函数里，短期代码看起来少，长期就很难排查：到底是模型猜错了、协议没拦住，还是状态迁移写错了。

### [7.21]--[为什么 clarify 要单独做成一条轨，而不是简单返回报错？]：
    [答案]：    因为在多轮客服系统里，大量“失败”其实不是异常，而是信息还不够。比如用户选中了工单，但没说是查状态还是投诉；用户想办业务，但没说清具体 flow；用户缺对象、缺槽位、缺方向。这些都不应该直接报错，而应该进入正式的 `clarify` 轨，由系统结合当前任务、对象和原因去追问用户，把对话重新拉回可执行协议。

### [7.22]--[focused_object 在当前项目里的作用到底是什么？]：
    [答案]：    `focused_object` 可以理解成当前对话里最明确的业务锚点，通常是工单对象或服务项目对象。它主要有三个作用：第一，让知识问答知道当前围绕的是哪个对象；第二，在 task 轨里为 `collect step` 自动补对象 ID 类槽位；第三，让后续追问具备明确上下文，减少歧义。所以它不是单纯的前端选中态，而是后端状态机里很重要的一块运行时上下文。

### [7.23]--[如果再往前优化一版，这个项目最值得继续演进的方向是什么？]：
    [答案]：    如果站在当前版本往后看，我最想继续推进的不是“多加几个功能”，而是两件事：第一，把状态机边界继续收紧，让更多状态流转通过显式 transition 表达，而不是散落在下游逻辑里；第二，把观测和测试体系继续做强，比如长对话回归集、状态快照对比、执行 trace，让每次优化都更可验证。下一阶段的重点应该是：**让系统更稳、更可解释、更适合长期演进**，而不是单纯追求多过几个 badcase。

### [7.24]--[项目里已经有 cancel 了，为什么还要单独设计 exit？]：
    [答案]：    这两个能力解决的不是同一层问题。`cancel` 解决的是**取消某个业务任务**，它本质上还是 task command，适合“当前确实有一个 flow 在跑，用户明确说要取消这个流程”的场景。`exit` 解决的是**退出当前整段运行态上下文**，它面对的不一定只是 `active_task`，还可能包括 `focused_object`、`active_system_task`、`paused_tasks`，或者用户围绕某个对象聊了几轮之后想整体换话题。所以可以把它理解成：`cancel` 是取消流程，`exit` 是退出上下文。如果只有 `cancel`，很多“算了 / 不聊这个了 / 换一个”的表达就会被硬解释成 task cancel，结果要么清不干净，要么语义不准确。

### [7.25]--[为什么把 exit directive 做成显式决策，而不是简单前置 if？]：
    [答案]：    因为这里真正要避免的，不是“多一个概念”，而是“退出逻辑重新散回各层打补丁”。如果只把 exit 当成一个很轻的前置条件，短期当然也能跑，但后面很容易变成 engine 里补一句、object 入口再补一句、task 取消分支再补一句、某个确认阶段再补一句。最后谁都能判断退出，谁也说不清退出到底归谁负责。所以当前项目把它做成显式 `directive`，本质目的是让“退出运行态”成为一个**可被识别、可被校验、可被记录、可被统一执行**的协议动作。但从架构表达上说，它更适合被理解为 **runtime-control 决策**，不应该在心智模型里长期和 `task / knowledge / chitchat` 完全等重。

### [7.26]--[老师原有代码能不能跑？如果能跑，为什么还要改架构？]：
    [答案]：    老师原有代码当然能跑，而且在短链路、标准表达、单对象、单任务的 happy path 上跑得是顺的。真正的问题不是“能不能返回一句话”，而是：**状态一复杂以后，能不能稳定跑对。** 原有代码在这些场景里会稳定出现一类问题：`focused_object` 黏住，用户已经换话题了，系统还在旧对象里答；用户表达的是“退出当前上下文”，系统却只能按“取消当前任务”去解释；对象切换、知识追问、只读 task 追问混在一起时，上下文容易串味；旧状态残留到下一轮，导致继续、切换、退出语义不统一。所以更准确的说法不是“原有代码跑不了”，而是：**原有代码能跑简单路径，但在长对话和混合上下文里会稳定跑偏。** 这也是为什么要提级改架构，而不是继续在几个函数里各补一点判断。

### [7.27]--[什么时候该补丁修复，什么时候该提级到架构改造？]：
    [答案]：    我会用一个很实用的判断标准：看问题是**单点偶发**，还是**同类状态问题反复出现**。适合补丁修复的情况是：问题只出在一个 flow、一个分支、一个明确入口；根因已经很清楚，就是局部实现遗漏；修完不会影响别的状态流转。需要提级到架构改造的情况是：同一类问题在多个对象、多个入口、多个任务里反复出现；修复总是落成“这里补一句，那里再补一句”；问题根因不是单个函数错，而是状态流转抽象本身不统一。这个项目里像“退出上下文”“对象切换后旧状态残留”“只读追问和 task/knowledge 承接不一致”，就都属于后者。所以当时改架构的意义，不是为了把代码写复杂，而是为了把这类重复出现的状态问题从补丁逻辑里提炼出来，变成可解释、可测试、可持续演进的主链设计。

### [7.28]--[既然当前版本里执行函数仍然会改状态，那统一的决策层有什么意义？为什么它会比老师版更稳定？]：
    [答案]：    这个问题的关键，不是“有没有把所有状态写入权都收走”，而是**有没有先把高层裁决权和执行后收口权统一起来**。当前版本没有做到“执行层完全不改状态”，它更像一种渐进接管：先由 `Planner -> Protocol Gate -> State Decision` 统一决定这一轮到底是什么，再让执行层去改它该改的业务状态，比如 step 推进、slot 写入、task start/resume/cancel，最后再回到决策层做一次 outcome 收口，把执行结果统一解释成 `conversation_state`、`last_route`、`last_task_outcome`。所以它比老师版更稳定，不是因为执行层不改状态了，而是因为现在变成了“先定性、再执行、再统一解释结果”，而不是让状态顺着各个函数自然漂移。

### [7.29]--[老师版、当前版、理想终局版，在“决策、执行、写状态、收口”上的分工分别是什么？]：
    [答案]：    可以把三版理解成三个层次。**老师版**更像“执行带状态走”：主链先做基础分流，然后 task / knowledge / chitchat handler 直接推进业务，状态主要顺着执行过程自然发生，高层没有特别明确的统一收口层。**当前版**是“半收口”：`State Decision` 先统一决定这一轮怎么走，执行层继续负责改业务状态，执行完成后再由决策层统一把结果解释成高层会话状态，所以它已经把“解释权”和“收口权”集中起来了。**理想终局版**则更纯：执行层尽量少直接写状态，而是返回标准化的执行结果或事件，再由状态机统一落状态。也就是说，老师版的问题在于状态主要依赖执行自然形成，当前版的提升在于先把决策和收口集中起来，理想终局版才是把状态写入本身也进一步统一起来。

### [7.30]--[如果后面要把 DialogueState 从一整坨 JSON 拆成多字段、多表存储，为什么统一决策层和状态收口反而更重要？]：
    [答案]：    因为当状态还是整份 `state_json` 一起落库时，很多分散的状态变化还能被“整体覆盖写回”暂时包住；但一旦把持久层拆成更细的字段或多表结构，状态不一致就会直接暴露成数据模型冲突。比如 `active_task` 已经切了、`focused_object` 还没清、`conversation_state` 还停在旧值、`last_route` 又被别的分支写掉，这时候问题就不再只是代码读起来乱，而是数据库里的状态本身会互相打架。所以如果真的要把持久层正规化，前提不是先急着拆表，而是先把“谁有权改状态、什么时候改、改完怎么统一收口”讲清楚。也就是说，统一决策层和状态收口不仅是架构优化，同时也是后续持久层细粒度改造的前置条件。
