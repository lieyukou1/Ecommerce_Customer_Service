## 1. 这份 smoke 的目的

这份 smoke 不是为了评估“模型有多聪明”，而是为了抓三类最现实的问题：

1. `Planner` 的 JSON 习惯污染了表达层
2. 模型虽然会答，但语气变硬、变短、变机械
3. 长上下文、对象上下文下，表达层开始失真

所以这里的 prompts 不追求大而全，追求的是：

- 真实
- 不模板化
- 不做排列组合
- 能直接把问题暴露出来

## 2. 使用方式

每条 smoke prompt 都记录四个信息：

- 轨道
- 上下文设置
- 用户输入
- 重点观察项

评估时不要给模型“标准答案”，只看：

- 有没有混入 JSON / 字段名 / 代码块
- 有没有异常短促
- 有没有答偏轨
- 有没有明显不自然

## 3. Clarify 轨 smoke

### C1 模糊办理意图

- 上下文：无 active_task，无 focused_object
- 用户输入：`我这边有个门禁的事想处理一下。`
- 重点观察：
  - 应该自然追问，不要直接编造业务 flow
  - 不要输出任何 JSON 结构
  - 语气不能像命令解析器

### C2 对象后续但方向不清

- 上下文：focused_object=`service_item(远程重置门锁密码)`
- 用户输入：`这个我得怎么弄？`
- 重点观察：
  - 应该围绕对象追问或解释办理方式
  - 不要装作已经理解成唯一业务动作
  - 回复不能发硬

### C3 当前任务补槽前的自然追问

- 上下文：active_task=`work_order_urge_submission`，缺 `urge_reason`
- 用户输入：`嗯，你继续。`
- 重点观察：
  - 应该继续追问原因
  - 不要把“继续”误当闲聊
  - 不能回复成“请提供字段”

### C4 多层意图但本轮应先澄清

- 上下文：无 active_task，focused_object=`work_order`
- 用户输入：`这个单子我也不知道是催一下还是直接投诉。`
- 重点观察：
  - 应该帮用户收束，不要替用户拍板
  - 不要直接输出双动作倾向
  - 回复要像客服，不像状态机日志

### C5 长对话后的轻澄清

- 上下文：前面 5-6 轮都在聊物业规则，当前没有 active_task
- 用户输入：`那这个现在能不能直接办？`
- 重点观察：
  - 需要自然澄清“具体指哪项”
  - 不要生硬重复上文
  - 不要把知识追问误转为固定流程命令

### C6 退出后再开新话题

- 上下文：刚执行过 `exit_runtime`
- 用户输入：`行，那我想问下停车费。`
- 重点观察：
  - 不要残留上一段对话的办理语气
  - 新问题要自然接住
  - 不能出现“当前上下文已退出”这种系统口吻

## 4. Knowledge 轨 smoke

### K1 规则说明，冷启动

- 上下文：空
- 用户输入：`装修报备一般都得注意什么？`
- 重点观察：
  - 回答应自然、概括清楚
  - 不能冒出 JSON
  - 不能只回一句特别短的摘要

### K2 收费说明，口语化

- 上下文：空
- 用户输入：`物业费这边到底怎么收，我先弄明白再说。`
- 重点观察：
  - 语言要像正常客服解释
  - 不要误切为 task
  - 不要过度模板化

### K3 对象聚焦后的知识追问

- 上下文：focused_object=`service_item(地漏疏通)`
- 用户输入：`那这个一般怎么收费，材料谁出？`
- 重点观察：
  - 应围绕当前对象回答
  - 不要串到其他服务项
  - 不要出现字段名式表达

### K4 长上下文后的知识承接

- 上下文：前面连续 4 轮都在问社区公约、宠物管理和装修报备
- 用户输入：`那停车这块你也顺便给我讲讲。`
- 重点观察：
  - 回答要顺畅换 topic，但仍留在 knowledge
  - 不要被长上下文拖乱
  - 不要答得像分类器输出

### K5 明确先咨询不办理

- 上下文：空
- 用户输入：`我先问问，不急着办，宠物这块平时有哪些要求？`
- 重点观察：
  - 应稳稳落到知识回答
  - 不要强行引导进入业务办理
  - 回复不应收缩成一句僵硬结论

### K6 泛物业能力说明

- 上下文：空
- 用户输入：`你们这边平时都能帮住户处理哪些事？`
- 重点观察：
  - 应概括项目能力范围
  - 不要把回复做成条目化字段吐出
  - 不要显出过强“结构化输出”习惯

## 5. Chitchat 轨 smoke

### H1 简单打招呼

- 上下文：空
- 用户输入：`你好，在吗？`
- 重点观察：
  - 回复自然
  - 不能夹带 JSON
  - 不要过度郑重

### H2 身份询问

- 上下文：空
- 用户输入：`你是物业客服吗？`
- 重点观察：
  - 要能自然自我定位
  - 不要答成系统说明书
  - 不要突然切到业务办理口吻

### H3 轻情绪表达

- 上下文：空
- 用户输入：`今天真是被这些事折腾烦了。`
- 重点观察：
  - 应有基本安抚
  - 不要直接追字段
  - 不要因为没明确业务方向就机械澄清

### H4 探测型闲聊

- 上下文：空
- 用户输入：`你平时是不是挺忙的？`
- 重点观察：
  - 闲聊应自然
  - 不要过于官方
  - 不要生硬把话题拉回办理

### H5 闲聊后切业务前的承接

- 上下文：刚做过一轮闲聊
- 用户输入：`行，那我等会再说正事。`
- 重点观察：
  - 应自然承接
  - 不要异常简短到像“确认收到”
  - 不要出现奇怪的状态化话术

## 6. 专门抓污染的高风险 smoke

这些不是常规业务问题，而是专门拿来揪副作用的。

### R1 JSON 污染

- 上下文：空
- 用户输入：`你先跟我说说你能帮什么忙。`
- 重点观察：
  - 回复里不能出现 `task` / `knowledge` / `directive` 这类字段名

### R2 解释意愿降低

- 上下文：空
- 用户输入：`为什么有些服务得先报备？`
- 重点观察：
  - 应给出自然解释
  - 不能只回一句极短结论

### R3 长上下文后的语气发硬

- 上下文：一段 6 轮以上的复杂对话后进入 knowledge
- 用户输入：`那你再给我说细一点。`
- 重点观察：
  - 不要回复成“请补充具体信息”
  - 不要明显像 JSON 生成模型在勉强说话

## 7. 通过标准

Smoke 通过，不要求每条都“回答完美”，但至少要满足：

- 无 JSON 污染
- 无代码块污染
- 无明显字段名污染
- 语气不明显发硬
- 不出现大面积异常短答
- 不出现稳定错轨

如果 TurnPlan 主评测已经提升，但 smoke 里反复出现这些问题，就不能直接上线。

## 8. 为什么这份 smoke 刻意不做排列组合

这次 smoke prompts 的写法刻意遵守两个原则：

1. 不把一句话拆成前缀 + 核心 + 后缀后做机械拼接
2. 不让用户自己在话里写“我是在补投诉原因”“我是在继续当前任务”

原因很简单：

- 这种写法看上去覆盖多，实际信息密度很低
- 还会把模型训成“只会接住显式标注意图的用户”

这份 smoke 的意义，恰恰是反着来：

- 看模型能不能接住真实一点、模糊一点、口语一点的用户表达
