# 连续两轮未压过基座时的诊断规范

- 最后修改时间: 2026-06-07
- 文档定位: 当微调候选连续两轮在关键指标上未能稳定压过未训练基座时，固定采用的停训与诊断流程
- 上级入口: [turnplan_finetune_master_report.md](/D:/Desktop/SGG_Project/Ecommerce_Customer_Service/Material/Finetune/turnplan_qwen3_14b/docs/turnplan_finetune_master_report.md)

## 1. 什么时候触发这份规范

只要出现下面任一情况，就不要直接开下一轮训练:

1. 候选连续两轮没有在关键业务指标上稳定优于未训练基座
2. 候选只在粗粒度指标上略有提升，但在协议相关指标上明显退化
3. 候选已经优于训练前旧口径结果，但一换成统一新口径后仍压不过基座

对当前 TurnPlan 项目而言，下面这些指标视为“关键指标”:

- `task_command_family_accuracy`
- `flow_selection_accuracy`
- `slot_fill_exact_match`
- `protocol_gate_pass_rate`
- `knowledge_false_positive_rate`

`top_level_track_accuracy` 只能作为辅助观察，不足以单独证明微调成功。

## 2. 先承认什么，不承认什么

### 要先承认的

- 当前这版微调方案可能存在系统性问题
- 问题不一定是“参数没调好”
- 数据、任务定义、协议边界、评测链路都可能是主因

### 暂时不要承认的

- “整个微调路线完全失败”
- “模型太小，所以无解”
- “再堆几轮一定会变好”

这个阶段的正确动作不是乐观加码，也不是立刻推倒重来，而是做**诊断性停顿**。

## 3. 固定诊断顺序

### 第一步: 同口径重跑未训练基座

目的:
- 避免不同轮次、不同口径下的比较失真

最低要求:
- 用**当前最新评测脚本**
- 用**当前最新 protocol gate 统计口径**
- 用**与候选相同的 prompt 合同**
- 跑同一份 `val`

产物至少包括:
- `metrics.json`
- `summary.md`
- `predictions.jsonl`

### 第二步: 做“基座对 / 微调错”反例抽取

目的:
- 找出微调真正把模型带偏的地方

固定优先级:
1. 当前最大失败桶
2. 当前最关键协议边界
3. 当前最像“基座常识正确、项目协议要求反常识”的场景

对本项目，第一优先就是:
- `work_order_read_only_task`

### 第三步: 跑训练集回放

目的:
- 判断问题是泛化失败，还是连训练分布本身都没学进去

解读规则:
- 如果微调模型在训练集上也压不过基座，说明这不是 val 太难，而是 SFT 信号本身没有有效塑形
- 如果训练集上显著优于基座、val 上又显著落后，才更像是过拟合或数据切分问题

### 第四步: 把问题强行分型

每个主要失败必须强制落入下列类别之一:

- 任务定义问题
- 标签边界问题
- 数据分布问题
- 超参问题
- 评测口径问题
- 推理接线问题

如果一句话解释里同时混了两三类，说明分析还没收干净。

## 4. 什么结果说明“不要继续堆训练轮次”

只要出现下面任一条，就不建议直接开 `round_n+1`:

1. 微调候选在训练集和验证集上都压不过未训练基座
2. 最大失败桶两轮都不动，且失败模式高度一致
3. 候选提升只体现在 `top_level_track_accuracy`，而协议相关指标持续弱于基座
4. 新增样本补强只能带来轻微回升，无法触及主失败机制

这时应该进入“方案改造”，而不是“继续加轮次”。

## 5. 什么结果说明还可以继续小步试验

满足下面三条时，还可以继续小步实验一轮:

1. 候选至少在部分关键指标上开始稳定优于未训练基座
2. 主失败桶开始收缩，不再稳定贡献双位数失败
3. 失败模式开始从“轨道错”收缩为“flow/slot 细错”

这时可以继续做:

- 小批次定向补样
- 轻量重采样
- 单变量超参试验

但仍然不建议大盘推倒重写。

## 6. 当前项目的具体诊断问题定义

基于 `round_01` 和 `round_02`，当前最需要回答的是三件事:

1. 用 round 2 新口径重跑后，未训练 `Qwen3-14B no-think` 到底是否仍优于候选？
2. `work_order_read_only_task` 里，是否存在大量“基座按自然常识判对、微调被协议标签带偏”的样本？
3. 微调候选在训练集上是否至少应该明显压过未训练基座？

如果这三问的答案分别是:

- 是
- 是
- 否

那么当前阶段更像是**任务定义 / 标签边界 / SFT 塑形方式**的问题，而不是单纯的数据量不足。

## 7. 诊断产物应该落在哪

固定落点:

- 原始结果: `reports/`
- 本轮诊断结论: `rounds/<round_id>/02_problem_analysis.md`
- 决策收口: `rounds/<round_id>/03_decision_log.md`
- 总结性阶段判断: `docs/turnplan_finetune_master_report.md`

不要把这类关键诊断只留在聊天记录里。
