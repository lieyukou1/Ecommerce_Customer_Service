# AutoDL 4090 训练目录与落盘规范

- 最后修改时间：2026-06-07
- 文档定位：面向当前 AutoDL 4090 单机环境的训练目录、缓存落盘、日志与迁移规范
- 上级入口：[11_TurnPlan微调实施方案.md](/D:/Desktop/SGG_Project/Ecommerce_Customer_Service/Material/Finetune/turnplan_qwen3_14b/docs/11_TurnPlan微调实施方案.md)

## 1. 当前机器约束

当前云服务器配置：

- CPU：25 核
- RAM：92GB
- GPU：`NVIDIA GeForce RTX 4090`，显存约 48GB
- 系统盘：`/`，30GB
- 数据盘：`/root/autodl-tmp`，223GB

这台机器的核心特征不是“算力不够”，而是：

- **GPU 够用**
- **内存够用**
- **系统盘很小**

所以训练目录设计的第一原则不是“整齐”，而是：

**所有大体积、会持续增长、可能反复写入的内容，一律优先放数据盘。**

## 2. 总原则

### 2.1 系统盘只放轻资产

系统盘 `/` 只允许放这些内容：

- 极少量启动脚本
- 环境变量文件
- 小型工具脚本
- SSH / screen / shell 配置
- 必要时的轻量 Python 启动入口

不允许把下列内容长期放在系统盘：

- Hugging Face 模型缓存
- `pip` / `uv` / `conda` 大缓存
- `wandb` 运行目录
- `torch` / `triton` 编译缓存
- 训练日志归档
- 训练产物
- merged 模型
- Docker 默认数据目录

### 2.2 数据盘承接重资产

数据盘 `/root/autodl-tmp` 负责承接：

- 项目训练工作区
- 数据集副本
- 基座模型缓存
- LoRA adapter
- merged candidate
- 评测报告
- W&B 本地缓存
- 临时导出文件
- screen / nohup 长日志

## 4. 项目代码放哪里

本轮建议把项目仓库本体也放到数据盘：

```text
/root/autodl-tmp/ecs-llm/repo/Ecommerce_Customer_Service
```

原因：

- 当前系统盘只有 30GB
- 代码仓库后续会继续增加训练配置、评测报告、日志引用和导出说明
- 如果还把环境、缓存、模型都挤在系统盘，会非常容易把系统盘打满

如果本地协作时仍然希望在 `/root/` 下保留一个易访问入口，可以只在系统盘放一个软链接，而不是复制整份仓库。

## 5. 必须迁移到数据盘的缓存

训练前必须显式设置这些环境变量：

```bash
export WORK_ROOT=/root/autodl-tmp/ecs-llm
export HF_HOME=$WORK_ROOT/cache/hf
export HUGGINGFACE_HUB_CACHE=$WORK_ROOT/cache/hf
export TRANSFORMERS_CACHE=$WORK_ROOT/cache/hf
export HF_DATASETS_CACHE=$WORK_ROOT/cache/datasets
export TORCH_HOME=$WORK_ROOT/cache/torch
export TRITON_CACHE_DIR=$WORK_ROOT/cache/triton
export WANDB_DIR=$WORK_ROOT/cache/wandb
export TMPDIR=$WORK_ROOT/cache/tmp
export PIP_CACHE_DIR=$WORK_ROOT/cache/pip
```

如果这一步不做，最常见结果就是：

- 模型刚开始下载时还没问题
- 训练一跑，系统盘突然爆掉
- 接着出现各种看上去像框架 bug、其实只是磁盘满了的错误

## 6. Docker 的特别处理

当前方案里容器化是要做的，但**不能直接默认使用 Docker 的系统盘数据目录**。

如果后续在这台机器上构建或运行服务镜像，必须先把 Docker data-root 改到数据盘，例如：

```json
{
  "data-root": "/root/autodl-tmp/docker-data"
}
```

否则镜像层、容器层、构建缓存很容易把 `/` 撑满。

所以本轮建议：

1. **训练阶段优先用直接 Python 环境**
2. 模型稳定后，再做 `serving image`
3. 只有在确认 Docker data-root 已迁移到数据盘后，才在这台机子上本地构建镜像

## 7. 训练产物怎么落

### 7.1 adapter

LoRA / QLoRA 训练产物统一写入：

```text
/root/autodl-tmp/ecs-llm/artifacts/adapters/<run_id>/
```

### 7.2 merged candidate

供推理使用的 merged 模型统一写入：

```text
/root/autodl-tmp/ecs-llm/artifacts/merged/<run_id>/
```

### 7.3 报告与清单

每次训练和评测都要同时生成 manifest：

```text
/root/autodl-tmp/ecs-llm/artifacts/manifests/<run_id>.json
```

manifest 至少记录：

- base model revision
- dataset version
- train config
- adapter path
- merged path
- eval report path
- 时间戳

## 8. W&B 和对象存储的职责分工

### 8.1 W&B

W&B 负责：

- run 追踪
- metrics
- artifact lineage
- config 快照

### 8.2 对象存储

对象存储负责：

- adapter 备份
- merged 模型备份
- 评测报告备份
- manifest 备份

原因是这台机器虽然关机不丢盘，但本轮设计不能把任何单机状态当成唯一真相。

## 9. screen 与长任务规范

当前机器明确要求长期命令用 `screen` 或类似工具后台运行。  
本轮固定约束如下：

- 一个训练 run 一个 screen session
- 一个评测 run 一个 screen session
- screen 日志落到：

```text
/root/autodl-tmp/ecs-llm/logs/screen/
```

建议命名：

- `tp-pilot-<date>`
- `tp-main-seed42-<date>`
- `tp-main-seed3407-<date>`
- `tp-eval-<run_id>`

## 10. 这台机器上的推荐执行顺序

结合当前磁盘结构，推荐顺序不是先装一堆东西，而是：

1. 在数据盘创建训练工作区
2. 把所有缓存环境变量写好
3. 准备 Python 环境
4. 拉基座模型到数据盘缓存
5. 拷入或同步当前项目仓库
6. 先跑 baseline 和评测脚本
7. 再做 pilot run
8. 模型稳定后，最后再考虑 serving image

## 11. 本轮目录设计和数据集教训的关系

这次训练目录之所以要收得这么严，是因为前面做数据集时已经踩过两类坑：

1. **假增量问题**
   - 大量排列组合式样本会制造“看起来很多、其实信息很少”的幻觉
2. **重做成本问题**
   - 一旦数据、导出、重写结果和评测结果混放，后面就很难分清哪份是可信真相

所以目录和落盘规范里要直接体现两个原则：

- 训练真相和临时产物分开
- 数据版本和模型版本都必须能追溯

这不只是“目录好看”，而是为了防止下一轮又回到“训完了，但不知道自己到底训了什么”的状态。
