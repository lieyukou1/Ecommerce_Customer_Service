## 1. 这份手册解决什么问题

很多微调项目失败，不是失败在“模型太差”，而是失败在一开始就没有把问题定义清楚。

常见误区包括：

- 一上来就想微调整个系统
- 数据一多就以为方向对了
- 只盯离线分数，不看下游链路
- 没有基线，训完不知道到底有没有变好
- 模型坏了还在坏模型上继续训
- 环境全靠机器本地，换机就重来

这份手册的目标就是把这些坑提前写明白。

## 2. 第一步：先定义“要替代什么能力”

做微调前，先回答下面四个问题：

1. 你要替代的是哪一段能力
2. 这段能力现在为什么值得训练
3. 训练后要比什么更好
4. 训练结果会影响到系统哪些其他部分

### 2.1 正确的切法

优先选择这种任务：

- 输入边界清楚
- 输出协议清楚
- 好坏容易评估
- 能接下游回放或业务结果

例如：

- 路由决策
- 意图分类
- 结构化抽取
- 工具调用参数生成
- 短格式审稿判断

### 2.2 不建议一开始就做的任务

- 全链路最终回复
- 多目标混训
- 一边做任务判断一边做长文写作
- 缺乏统一标签定义的“泛聊天提升”

原因不是这些任务永远不能做，而是：

- 它们很难判断到底哪一步出了问题
- 也很难稳定复现收益

## 3. 第二步：先做 baseline，再谈训练

没有 baseline，训练结果没有意义。

至少保留三类 baseline：

1. **现网基线**
   - 当前真实系统表现
2. **本地未训基座**
   - 基座模型本身能力
3. **训练候选**
   - 本次实验的真正增量

这三者的作用完全不同，不能互相替代。

## 4. 第三步：数据集设计

### 4.1 先做 canonical schema

不要先直接产出训练格式。  
先定义一份中间态 schema，作为“人工标注真相源”。

一个合格的 canonical schema 至少要清楚表达：

- 样本 id
- 数据来源
- 场景桶
- split
- 输入字段
- 输出标签
- 审计备注

### 4.2 训练格式与标注格式分离

标注格式的任务是：

- 可读
- 可审计
- 可追加字段

训练格式的任务是：

- 适配训练框架
- 稳定导出

不要让训练框架的输入格式反向绑架标注规范。

### 4.3 先定义白名单与边界

训练前先写清楚：

- 合法输出字段
- 命令白名单
- 意图白名单
- null / reject / fallback 的判定规则

如果连输出边界都没写死，后续所有“审计”和“准确率”都会漂。

## 5. 第四步：数据审计

### 5.1 必做的结构审计

每个数据集至少做：

- 重复检测
- 覆盖检测
- 类别分布检测
- schema 合法性检测
- contract 检测

### 5.2 必做的语言层审计

很多数据集不是结构错，而是语言假。

要重点审：

- 模板残留
- 伪多样性
- 用户自我标注意图
- 系统术语污染
- 机械同义改写

如果一句话读起来不像真实用户，即使 label 是对的，也会把模型训歪。

### 5.3 什么时候要推倒重做

如果出现这些现象，不要补补丁，要重构数据集：

- 高重复率
- 关键意图或 flow 零覆盖
- 大量样本只是在换前缀后缀
- 用户语言里充满系统开发者视角

## 6. 第五步：训练方案选择

### 6.1 什么时候选 QLoRA

以下情况优先选 QLoRA：

- 单机资源有限
- 先做一个清晰子任务
- 更关心可控收益，不追极限指标
- 需要快速迭代多轮实验

### 6.2 什么时候才考虑更重路线

只有在这些前提都满足时，再考虑更重训练：

- 数据规模足够大
- 标签和评测体系已经稳定
- 基础 SFT 已经吃满了主要增益
- 你有明确目标去优化偏好、风格或复杂策略

### 6.3 一个常见误区

不要把“模型大”当成“流程可以粗糙”的理由。  
模型越大，训练和部署成本越高，越应该先把问题切清楚。

## 7. 第六步：训练执行

### 7.1 先 pilot，再 main run

每个项目都先做 pilot run：

- 小数据
- 单 seed
- 短 epoch
- 只验证链路

pilot 过了，才做 main run。

### 7.2 main run 的基本纪律

- 至少两次独立 seed
- 参数变动要有记录
- 不在坏 checkpoint 上续训
- 不覆盖旧模型
- 不把“本次最优”和“历史最优”混在一起

### 7.3 为什么不能在坏模型上接着训

如果你继续在一个效果已经明显跑偏的 checkpoint 上训练：

- 你会失去可解释性
- 也失去“到底是数据问题还是参数问题”的对照

正确做法永远是：

- 冻结基座
- 冻结数据版本
- 冻结配置
- 新开 run

## 8. 第七步：评测设计

评测至少分三层：

1. **静态合法性**
   - 输出是不是合法
   - 格式是不是过关
2. **任务指标**
   - 主任务准确率、召回、命令正确率等
3. **系统回放**
   - 训练结果进系统后到底有没有帮助

### 8.1 不要只看单指标

单一 accuracy 很容易骗人。  
更重要的是：

- 有没有把边界场景打坏
- 有没有把 null / fallback 策略训没
- 有没有把表达层或其他调用场景带偏

### 8.2 必须评估副作用

如果你只训练某个子任务，但部署时会用同一模型覆盖更多场景，那么必须额外评估：

- 表达层污染
- 风格异常
- 输出格式串轨
- 长上下文表现退化

这是很多项目最容易漏掉的一步。

## 9. 第八步：部署与灰度

### 9.1 优先做蓝绿

统一原则：

- 蓝：当前稳定基线
- 绿：本轮候选

切换通过配置，不通过手工改代码常量。

### 9.2 先局部验，再全量替

如果系统里模型承担多个角色：

- 先把主任务评估清楚
- 再做其他角色 smoke
- 再决定是否整模型统一替换

### 9.3 不要把线上变成实验场

线上只做：

- 验证候选是否符合预期
- 快速回滚

线上不要做：

- 临场换参数
- 临场改 prompt 乱试
- 一边服务一边改训练产物

## 10. 第九步：容器化、迁移、灾备

### 10.1 机器不是资产

尤其在云 GPU 常常关机丢卡位的环境里，机器不能当真相源。

真相必须在：

- artifact registry
- 对象存储
- 配置仓库

### 10.2 必须固化的东西

任何微调项目都要固化：

- 基座模型 revision
- 数据集版本
- 训练配置
- checkpoint / adapter
- 部署产物
- 评测报告
- 依赖镜像

### 10.3 恢复动作要短

一个成熟方案的目标是：

- 换一台机器
- 拉镜像
- 拉产物
- 起服务

而不是靠工程师回忆“上次那台机子到底装了什么”。

## 11. 第十步：复盘与提级原则

当结果不好时，排查顺序固定为：

1. 任务定义有没有切对
2. 数据集有没有骗人
3. 评测口径有没有失真
4. 训练参数是否明显不合理
5. 最后才怀疑模型大小或框架本身

### 11.1 什么叫“该提级”

出现以下情况时，说明问题不再是“再补几个样本”能解决的：

- 主任务本身定义就不清楚
- 输出协议与运行时合同经常打架
- 同一模型替换后系统其他轨道持续受污染
- 一轮轮训练都只能换来局部小修

这时应该提级到：

- 重新切任务
- 重做数据协议
- 拆分模型职责
- 调整上线形态

### 11.2 什么叫“不该纠缠”

这些情况不要无限下钻：

- 单个样本争议但不改变主分布
- 小范围话术风格偏好
- 某个 case 要靠补丁 prompt 才能赢回一点点分

如果你为了少数 case 把体系越写越脏，最后损失的是整个项目的可解释性。

## 12. 这份手册的使用方式

建议每次做新微调项目时，先按这个顺序过一遍：

1. 先写任务定义
2. 再写 canonical schema
3. 再做审计标准
4. 再定 baseline
5. 再定训练方案
6. 再写评测门禁
7. 再写部署和回滚口径

如果前四步还没成型，就不要急着开始训练。

真正稳的微调项目，不是训练脚本写得多炫，而是从第一步起就知道：

- 自己在解决什么问题
- 怎么证明变好了
- 变坏了怎么退
- 机器没了怎么重来
