- 最后更新：2026-06-08

- 上级入口：[22_TurnPlan微调全链路复盘报告.md](/D:/Desktop/SGG_Project/Ecommerce_Customer_Service/Material/Finetune/turnplan_qwen3_14b/docs/22_TurnPlan微调全链路复盘报告.md)

## 1. 这次要解决的核心问题

当前项目虽然已经做了：

- 输入最小化
- `read_only` runtime 吸收
- 联合审计

但训练监督目标仍然是旧 `TurnPlan JSON`。也就是说，模型训练时依然直接学习：

- `task / knowledge / chitchat / directive`
- `start_flow / set_slots / cancel_flow / resume_flow`
- flow id
- slots

这会把两类本质不同的判断继续绑在一起：

1. 语义判断  
   例如：继续当前任务、新任务、只读查询、需要澄清、退出、闲聊
2. 协议映射  
   例如：输出哪个 command family、走哪条 flow、是否直接填某个 slot

真正的“语义决策层重构”，就是要把这两层拆开：

- **模型先学语义决策**
- **后端再做协议映射**

## 2. 什么才算“真正做完”

只有同时满足下面四条，才算真正完成，而不是“只改了一半”。

### 2.1 训练监督目标改了

模型训练时的 `assistant` 不再直接输出旧 `TurnPlan JSON`，而是先输出语义决策结果。

建议的语义决策层最小标签空间：

- `continue_current_task`
- `start_or_switch_business_task`
- `read_only_request`
- `need_clarify`
- `exit_runtime`
- `social`

必要时可在二级字段里补少量解释性结构，但不直接让模型输出最终 command / flow / slot。

### 2.2 后端正式存在“语义 -> 协议”映射层

后端必须新增一个明确职责层，负责把语义决策映射成旧 `TurnPlan JSON`。

这层不再是临时兼容补丁，而是正式能力层。

它至少要决定：

- `continue_current_task` -> continuation 型 task command
- `start_or_switch_business_task` -> start / switch / resume 哪种协议动作
- `read_only_request` -> `knowledge` 还是只读 `task flow`
- `need_clarify` -> `all_null` 或 clarify 触发协议
- `exit_runtime` -> `directive.exit_runtime`
- `social` -> `chitchat`

### 2.3 数据 gold 改了

canonical 数据不能再只存旧 `output`，而是要显式拥有两层 gold：

1. `semantic_gold`
2. `protocol_gold`

其中：

- `semantic_gold` 用来训模型
- `protocol_gold` 用来做映射正确性与系统兼容校验

### 2.4 评测也改成两层

评测不再只问“最终 `TurnPlan` 对不对”，而是拆成：

1. 语义决策对不对
2. 语义映射回旧协议后，系统是否对

否则即使训练目标改了，评测仍然会把两层混在一起，结论还是不干净。

## 3. 后端要改什么

这部分只定义改造范围，不代表现在立刻改代码。

### 3.1 新增语义决策模型输出契约

需要新增一个独立的语义决策契约，例如：

```json
{
  "semantic_decision": "continue_current_task",
  "decision_payload": {
    "target_kind": "rule_topic",
    "confidence": "high"
  }
}
```

要求：

- 字段少
- 标签边界清晰
- 不夹带旧协议动作名

### 3.2 新增语义映射层

建议单独建一个 mapper / compiler 层，职责一句话就能说清：

> 把语义决策结果编译成旧 `TurnPlan JSON`

它应该接收：

- semantic decision
- current `DialogueState`
- available flows / intents

然后输出旧协议。

### 3.3 收口现有 `TurnPlanNormalizer`

当前 `TurnPlanNormalizer` 仍是旧协议上的兼容层。重构后应收口为：

- 只处理旧协议残余兼容
- 不再承担主语义分流职责

新的主分流职责应进入“语义映射层”。

### 3.4 明确 runtime 哪些判断继续保留

下列能力仍应留在 runtime，而不是重新塞给 SFT：

- 高损失 `read_only` 执行分流
- 基于当前状态的 command family 收缩
- 对象 ID 补齐
- confirm stage 的退出归一化

原则是：

> 语义由模型决定，协议细化与业务保险由 runtime 决定。

## 4. 数据 gold 要改什么

### 4.1 canonical schema 升级

建议 canonical 增加：

```json
{
  "semantic_gold": {
    "decision": "continue_current_task",
    "subtype": "slot_fill"
  },
  "protocol_gold": {
    "task": {...},
    "knowledge": null,
    "chitchat": null,
    "directive": null
  }
}
```

旧 `output` 可逐步改名为 `protocol_gold`，避免语义混淆。

### 4.2 重新标注的重点不是所有样本，而是边界样本

最优先要重标的是这些 bucket：

- `active_task_slot_fill`
- `task_interrupt_resume_cancel`
- `ambiguous_all_null`
- `work_order_business_complaint`
- 剩余未被 runtime 完整吸收的 `read_only` 边界样本

### 4.3 同句异状态对照要以语义层为主

以后 contrast group 不应再主要对齐旧协议动作，而应优先对齐语义差异，例如：

- 同一句在 active task 下是 `continue_current_task`
- 在 idle 下是 `read_only_request` 或 `need_clarify`

## 5. 评测怎么改

### 5.1 新增语义层主指标

建议新增：

- `semantic_decision_accuracy`
- `continue_current_task_accuracy`
- `read_only_request_accuracy`
- `clarify_trigger_accuracy`
- `exit_runtime_accuracy`

### 5.2 保留协议层指标，但降为第二层

保留现有：

- `task_command_family_accuracy`
- `flow_selection_accuracy`
- `slot_fill_exact_match`
- `adjusted_protocol_gate_pass_rate`

但解释上要明确：

- 这些不再直接用于衡量模型纯语义能力
- 而是用于衡量“语义 + 映射 + runtime”整体系统是否成功

### 5.3 replay 变成双阶段

新的回放顺序应是：

1. 模型输出 `semantic decision`
2. semantic mapper 生成旧 `TurnPlan`
3. `protocol gate`
4. `state decision`
5. runtime replay

这样失败时才能知道：

- 错在模型语义判断
- 还是错在语义到协议的映射

## 6. 第一轮 SFT 应该怎么验

第一轮不要追求“全系统立刻压过当前 base”，而要追求结论干净。

### 6.1 第一轮目标

第一轮只验证三件事：

1. 语义标签空间是否真的更可学
2. hardest buckets 是否不再出现原先那种系统性退化
3. 语义层提升是否能通过 mapper 转化为系统级收益

### 6.2 第一轮成功标准

至少满足：

- `semantic_decision_accuracy` 明显高于旧协议监督基线
- `continue_current_task` 相关 bucket 不再明显弱于 base
- `cancel_flow vs exit_runtime` 不再大面积缠在一起
- `read_only_request` 与 runtime 分流后系统成功率稳定

### 6.3 第一轮失败也要能定位

第一轮如果失败，必须能明确回答是：

- 语义标签本身设计得还不纯
- 映射层设计有问题
- 还是语义层依然样本不足

如果做不到这一点，就说明重构还不够彻底。

## 7. 这条路是大工程，还是还能收成一个像样结果

结论分两层。

### 7.1 作为“完整架构重构”

这是中等偏大的工程，不是小补丁。因为它会同时动到：

- 后端决策链
- 数据 schema
- 导出脚本
- 评测脚本
- 指标解释口径

### 7.2 作为“项目结果收束”

它不是没机会，但前提是**收缩第一版范围**。如果只是为了把这条路验证清楚、做出像样结果，第一版应该：

- 只选 4-6 个最关键语义标签
- 只重标最关键 bucket
- 只做一轮 reduced 语义 SFT
- 只验证最关键的 2-3 类 hardest case

这样它仍然有机会在当前项目里收成一个“方向正确、证据干净”的结果，而不是失控成一个新大坑。

## 8. 正式动手前的前置条件

在真正改代码前，先满足这几个前置条件：

1. 建立 git 仓库并完成第一次可回滚提交
2. 把当前后端可运行版本打一个基线 tag 或至少做一次本地快照提交
3. 把当前训练数据、round 文档、reports 全部纳入版本管理或明确归档
4. 单独写一份“语义标签定义表”，先统一口径再写代码

没有这四步，不建议直接开始重构。

## 9. 当前最重要的判断

如果后续真的要再走一次 SFT，这份清单的意义在于先回答清楚一件事：

> 我们要不要继续让模型直接学旧协议，  
> 还是承认这个目标太混，改成先学语义，再让系统做映射。

当前我的判断是：

- 继续在旧协议监督上缝补，边际收益很低
- 真正值得验证的新路线，是“语义决策监督 + runtime 映射”
- 但这次必须把监督层一起改掉，不能再只改输入层和兼容层
